Character Creator Walkthrough

module/dauligor-pairing · design · 2026-05-30 · foundry-module branch

A guided, in-Foundry character-creation flow that walks a player through six steps and assembles a finished level-1 character. It replaces the Character Creator placeholder that currently opens an "under construction" dialog in the Dauligor Options launcher.

Shape & reuse

Built as an ApplicationV2 + HandlebarsApplicationMixin wizard — same family as the importer, feat browser, and spell manager — so it inherits the unified window chrome, the shared .dauligor-detail panes, and the button/token CSS we just consolidated.

  • scripts/character-creator-app.js — the wizard (new)
  • templates/character-creator-shell.hbs — stepper rail + step body + footer (new)
  • styles/character-creator.css — registered in module.json before responsive.css (new)
  • scripts/ability-roll-pool.js — the shared roll-pool service + socket relay (new)
  • scripts/main.js — launcher entry re-pointed from the stub to the real opener; new world setting registered
No duplicated machinery. The class step does not re-implement advancement resolution. It hands the chosen class to the existing, tested class importer (openDauligorImporter({ actor, sourceTypeIds, selectedEntryIds, targetLevel: 1 })) so Fighting Style / skill picks / option-group exclusions all run through the one code path that already handles them. Backgrounds and races embed exactly the way the feat browser already does.

Step 1 — Ability Scores

Two generation modes the player toggles between, then a common assignment grid (map six values onto STR/DEX/CON/INT/WIS/CHA).

Mode A · Point Buy

Homebrew budget: 32 points, each score 8–16 before racial bonuses.

Extends the standard 5e table arithmetically (+2 per step past 13). A live "points remaining / 32" meter blocks over-spend.

Mode B · Shared Roll Pool

A world-level public pool of roll-sets. Every player may roll one set of six (4d6, drop lowest) into the pool; anyone may choose any set — including another player's. The DM can also enter sets manually (rolled at the table).

Stored as a world setting → propagates to every client automatically. Non-GM rolls are relayed to the GM via socketlib (executeAsGM) since players can't write world settings directly.

Roll-pool mechanics

  • Pool entry: { id, userId, userName, rolls:[6], total, source:"rolled"|"manual", createdAt }.
  • One rolled set per player — re-rolling replaces your own set (with a confirm). The DM can clear any set.
  • Each set shows its six dice and its total, so the "one player rolled 72, another 76, pick from either" comparison is visible at a glance.
  • Picking a set copies its six numbers into the assignment grid; the set stays in the pool for others.
Open question for you: when a player picks a roll-set, should it become claimed (greyed for others) or stay freely shareable? Your description ("both players can choose from either set") reads as freely shareable — that's the default I'll build unless you'd rather sets be claimed once used.

Steps 2 & 3 — Background & Race

Backgrounds and races are served inside /api/module/<source>/feats.json, tagged by featType ("background" / "race") — the same pool the feat browser reads. The creator fetches the selected sources' feat lists once, filters to the relevant featType, and shows a searchable list + a .dauligor-detail preview pane. On selection it lazy-fetches the full item from /api/module/backgrounds/<id>.json / /races/<id>.json and embeds it at apply time (Foundry type:"background" / "race"), reusing the feat browser's embed path.

Data note: background/race system fields (movement, senses, startingEquipment, wealth) currently ship as empty placeholders until the dedicated bg/race tables land app-side. The name/description/flags are real and embed fine today; the structured fields fill in transparently as the app data matures. This is also why step 5 is a stub (below).

Step 4 — Class

Lists the class catalog for the selected sources (/api/module/<source>/classes/catalog.json) with a preview pane. The player picks one class (creation = level 1). At Apply, after abilities/background/race are written, the creator launches the existing class importer pre-scoped to that class against the actor, so the player finishes any level-1 advancement choices (e.g. skills, Fighting Style) in the polished flow that already exists.

Step 5 — Starting Items (stubbed for v1)

There is no item list endpoint yet (only /api/module/items/<id>.json by id), and class/background startingEquipment data isn't populated. So v1 shows an informative placeholder ("available once the item catalog endpoint exists") and the step is skippable. The wizard structure leaves a clean slot to drop the real picker in later.

Step 6 — Review & Finish

A summary card per step with an Edit link that jumps back to that step (the "offer to go back and change anything" requirement). The stepper rail is also freely clickable throughout — every prior step stays editable. Finish runs the apply sequence:

  1. Resolve the target actor (existing, or create a new empty character).
  2. Write the six ability scores.
  3. Embed the background item, then the race item.
  4. Launch the class importer (scoped, level 1) to complete the class.
  5. Open the finished actor's sheet.

Build order

  1. Wizard shell + stepper + footer nav + free back/forward (scaffold).
  2. Step 1 ability scores — point buy + shared roll pool + socket relay + assignment grid (the novel part).
  3. Steps 2–3 background + race (list + detail + deferred embed).
  4. Step 4 class selection + delegated apply.
  5. Step 5 stub; Step 6 review + apply sequence; actor create/target.
  6. CSS pass + module.json registration; owner eyeball in Foundry.
