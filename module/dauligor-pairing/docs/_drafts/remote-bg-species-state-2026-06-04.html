Module API — Backgrounds & Species

Current remote state on www.dauligor.com · checked 2026-06-04

Verdict. The module has no way to list backgrounds or species from the remote. They were migrated out of the feats table into their own tables; detail endpoints shipped, but no list endpoint did, and feats.json no longer carries them. The creator's Species/Background sections — and the import wizard's feat browser — both enumerate via /<source>/feats.json + featType, so both come up empty. Hence "none available."

The data itself is fine — the website's compendium lists backgrounds & species, so the backgrounds/races tables are populated and the detail endpoints read through to them. The gap is purely the enumeration step: nothing tells the module which dbIds exist.

Why this happened

This is a cross-branch contract drift. The module side (this branch) still assumes "backgrounds & races live in the feats table and ride feats.json tagged by featType" (the comment is right there in feat-browser-app.js). The compendium-editors side promoted them to dedicated backgrounds/races tables and shipped per-item detail endpoints — but the feat list stopped including them and no replacement list endpoint was added. So the module's list source silently went empty.

The fix (app-side — compendium-editors)

Either restores both the creator and the importer; the module's detail-routing already knows how to fetch backgrounds/<id> & races/<id> once it has the ids.

  • Option A — re-include in feats.json (smallest): have _sourceFeatList.ts UNION the backgrounds + races rows back into the source feat list, tagged featType:"background" / "race" with a dbId. Zero module changes — the existing featType routing just works again.
  • Option B — dedicated list endpoints: add GET /api/module/<source>/backgrounds.json and …/species.json (mirroring the class catalog). Cleaner long-term, but the module would switch its two loaders (creator _loadFeatFamily + the feat browser) to read them.

Per our cross-branch rule this is a request-handoff to compendium-editors, not something this branch edits. Recommend Option A for the fastest restore.

Probed: feats.json featType counts, eight list-endpoint candidates (all 404), the router's full GET arm list, and _sourceFeatList.ts. D1 direct-query is auth-gated, so item counts come from the public compendium, not a row count.
