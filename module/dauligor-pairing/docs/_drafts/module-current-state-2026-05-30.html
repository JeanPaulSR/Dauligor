TL;DR — what this is

The foundry-module branch is the sole steward of module/dauligor-pairing/ — the FoundryVTT v13 + dnd5e module that consumes the website's /api/module/* export bundles and imports them into Foundry. This document is the result of a read-only verification pass: five parallel surveys read the real script source (~1.5 MB across 20 files) and compared it against the ~40 contract docs. No code or docs were changed — this is the “current facts” snapshot you asked for, so you can tell me which direction each drift should be resolved.

Resolution (applied 2026-05-30): the owner reviewed this audit and set direction. The doc-reconciliation pass is now done — stale class/spell/feat/reference/property/import docs were corrected or given clear “planned vs shipped” status banners, the README was rewritten to current facts, and deferred work was folded into TODO.md. Two findings in this snapshot were corrected on closer inspection: importedAt/lastSyncedAt/moduleVersion are written (on the class import path, as provenance), and virtual folders do exist on the alt character sheet (just not in the prep manager). The next build task is the backgrounds/races importer.
The headline finding: the code is considerably more capable than several docs claim, while the README and a few contract docs describe an earlier generation of the module. The drift runs in both directions, which is exactly why intent matters: for each mismatch, either the doc is stale (code is canonical → fix the doc) or aspirational (doc is the target → code is behind). Only you can set the direction for the ambiguous ones.

The repo is two halves meeting at a JSON contract:

The module reaches a single host, chosen by the world setting apiEndpointMode (production → https://www.dauligor.com, local → http://localhost:3000). Everything below is the module side only.

Boundary note: the Phase 2 live-content viewer (scripts/dauligor-viewer.js, scripts/enrichers/, templates/dauligor-viewer.hbs, styles/dauligor-viewer.css) is owned by system-applications, not this branch. scripts/main.js is shared append-only with that branch.

What the module actually does today

Five subsystems, each verified against source. built = working today; partial = real but incomplete; stub = placeholder only.

1. Class import & advancement largest subsystem

class-import-service.js (7,188 lines) + importer-app.js (7,107 lines) + importer-base-features.js, foundry-id.js, requirements-walker.js, update-character.js, importer-utils.js.

  • Accepts four payload families: dauligor.class-bundle.v1, dauligor.item.v1 (class), raw Foundry class item, and the structurally-detected semantic full export (normalized to the bundle flow).
  • World import and actor import, the latter driven by a sequenced wizard: prepare → levels → hp → skills? → tools? → subclass? → option-groups* → spells? → import.
  • Builds native dnd5e advancements (HitPoints, SavingThrow, Skill, ScaleValue, ItemGrant) and normalizes semantic activities into native system.activities (11 supported activity types) + effects — while also preserving the semantic copy in flags.semanticAutomation.
  • Author-supplied ItemChoice and AbilityScoreImprovement advancements are normalized. ASI is collected via a custom Dauligor app (DauligorAbilityScoreImprovementApp), not the native dnd5e AdvancementManager.
  • Deterministic Foundry _ids derived from SHA-256(moduleId:sourceId) so drag-drop and re-import stay stable; preserves actor-side advancement identity via flags.advancementIdMap.
  • Multiclass-aware proficiency overlay; full requirements-tree evaluation (all/any/one groups) against actor state.

Not built: synthesizing a native ItemChoice from option groups (still wizard-driven + stored in flags.optionGroups); starting-equipment step (a dead stub, deliberately omitted from the sequence); export-side remap of feature requirement leaves (advisory text only).

2. Spell preparation & spell browser

spell-preparation-app.js (4,420 lines) + spell-points-service.js.

  • DauligorSpellPreparationApp — sheet-side prep manager. Two modes: sheet (class-scoped) and importer (source-scoped). Can render standalone or embed into the Feature Manager's Spells tab.
  • DauligorSpellBrowserApp (subclass) — the standalone Spell Browser; merges every selected source's pool, multi-select bulk import into the alt sheet's “Other Spells” bucket.
  • Three sheet modes (prepared/spellbook/free), all using dnd5e 5.x boolean system.prepared + system.method (not the legacy numeric 0/1/2). Cap accounting via the Dauligor-owned flags.sheetMode.
  • Fetches dauligor.class-spell-list.v1, dauligor.source-spell-list.v1, dauligor.spell-item.v1, dauligor.tag-catalog.v1.
  • Favorites stored as an actor-level array of dbIds (flags.spellFavorites), not per-item. Embedded-mode mutations defer into a long-rest queue; known-caster “one swap per long rest” home-rule is enforced.

Not built: virtual folders in the prep manager (folders are built on the alt character sheet). This file registers no hooks; the dnd5e.restCompleted auto-open the docs describe lives in main.js/Feature Manager, not here. Two catalog fetches hardcode the production host, ignoring apiEndpointMode (likely a bug).

spell-points-service.js: offers to attach the third-party dnd5e-spellpoints item to spellcasters; reads/writes only that module's flags + flags.core.sourceId — no Dauligor flags.

3. Feat browser & Feature Manager

feat-browser-app.js (1,377 lines) + feature-manager-app.js (1,887 lines).

  • DauligorFeatBrowserApp — fetches dauligor.source-feat-list.v1 (pool) + dauligor.feat-item.v1 (detail), imports the already-Foundry-shaped feat verbatim (no activity construction on this side). Match by sourceId then name+type. Favorites in flags.featFavorites (actor).
  • DauligorFeatureManagerApp — six tabs. Built: Overview (long-rest card + queued-change preview), Features (real option-group re-pick / “Swap this pick”), Spells (embeds the prep manager). Placeholders: Crafting, Feats, Advancement.
  • Long-rest deferred-mutation queue in flags.featureManagerQueue; the commit engine currently only commits spellChange entries (optionItem is a stub; level-up-scope commit is upcoming).
  • Registers updateActor + dnd5e.advancementManagerComplete hooks for live refresh.

Option-group swap applies immediately (delete+create), not gated on the proposed swappable_on_long_rest source flag (that concept is unbuilt). Dead legacy _renderGroupSection + its queue stub remain in the file.

4. Export / research / reference services

export-service.js (1,626 lines) + import-service.js + reference-service.js.

  • Exporters (Foundry → JSON, for research/corpus building): single-document envelope, research bundle, world snapshot, world-research snapshot, and folder exports for spells / feats / items / actors, plus a live-window dump and an actor class snapshot. Eight carry a dauligor.* kind; two (single-doc, plain world snapshot) are un-kinded.
  • import-service.js is a test harness: consumes dauligor.character.v1, dauligor.item.v1, or a bare Foundry-like object; registers the socketlib handler. Writes sourceId/sourceBookId/entityId/identifier on upsert.
  • reference-service.js normalizes semantic references in whitelisted text/formula fields: scalar refs (@prof, @level, @ability.*, @class.*.level|tier|hit-die*, @subclass.*.level, @scale.*, bare skill/tool/language slugs, class-column shorthand) and entity refs (@class[…], @subclass[…], @feature[…], @option[…] → @UUID[…]).

Notable gaps vs docs: @source[…] is matched but deliberately never resolves; @class.*.spellcasting-ability and the dotted @skill.* form are documented but not parsed; the actor folder export is self-described as research-only with no consumer.

5. Module shell: entry, alt sheet, GM app, UI

main.js (entry) + dauligor-character-sheet.js + gm-app.js + section-filter-panel.js + constants.js + utils.js.

  • main.js registers ~18 hooks: init/ready, sheet header controls, rest-bar button, settings launcher, scene-control GM button, sidebar directory buttons, settings-config form group, spell-tab buttons, dnd5e.preLongRest diversion to the Feature Manager, dnd5e.restCompleted long-rest commit, and a libWrapper patch preserving remote class/subclass image URLs. Four world settings + one GM keybinding.
  • Alt character sheet — opt-in (makeDefault:false), extends the dnd5e CharacterActorSheet, overrides only the Spells tab to host per-class sections/folders and cross-class drag-drop gated by class spell-list membership. Rich flag model (customSections, customFolders, classOrder, collapse state, etc.).
  • GM console (gm-app.js) is a confirmed empty “Coming soon” shell.
  • section-filter-panel.js — shared tri-state filter UI; pure functions, no hooks.

Dead path: constants.js still points SAMPLE_FILE and CLASS_CATALOG_FILE at data/sample-character.json and data/classes/catalog.json — neither file exists on disk (see §Data).

Bundled data & README problems

The README is the single most drifted document. Its “Included files”, “Current controls”, “Directory context menus”, and “Test flow” sections describe an earlier module generation.
  • “Included files” omits 8+ real scripts (feat-browser, foundry-id, importer-base-features, importer-utils, requirements-walker, spell-points-service, update-character, section-filter-panel, constants, utils).
  • Lists data fixtures that do not exist: data/sample-character.json, data/classes/catalog.json, data/classes/sorcerer-bundle.json, the players-handbook/ tree, etc.
  • Actual data/ layout: data/sources/catalog.json + per-source folders (phb, tce, scag, xge, dsotdq, rj, uah, vrgr), plus loose dauligor_artificer_full_export.json and sources.zip. Class JSONs exist only for phb (barbarian/bard/cleric/druid/sorcerer) + tce (artificer).
  • “Directory context menus” section is entirely unimplemented — exports are folder-export toolbar buttons, not per-document context items; no journal export at all.
  • Header-button claims (Export JSON / Export Research / Import URL / Import Sample) don't match the registered controls.
  • docs/corpus/ holds only source-side 5etools files (artificer, wizard, fluff) — no Plutonium or Foundry-native dumps yet, despite the template's three-file pattern.
