Goal: use the components/styling we've defined in a unified manner — same function, same styling; only differ when the function differs. I surveyed the stylesheet and the render scripts. Good news first: there's a real design-token system (21 CSS variables) and inline style="…" is nearly absent (11 total). The inconsistency is at the component and token layers, described below. Nothing changed yet — this is for you to set direction.

The lay of the land

  • 24 component blocks (.dauligor-spell-manager, -feature-manager, -class-browser, -wizard, -section-filter, …) and no shared component primitive — there's no .dauligor-btn / .dauligor-card. Each app block defines its own.
  • The same primitive suffix is re-defined across many blocks: __button in 5 blocks, __empty in 7, __footer in 7, __toolbar in 8, __row in 6, __badge in 3.
  • But re-defined ≠ duplicated. Applying your "same function" test, most of these are genuinely different (a footer status-bar vs a 2-column grid footer). The true duplicates are narrower — see below.

Findings

unify A. Buttons — three near-identical copies of one flat button

There's a clear canonical "Dauligor flat button" (Foundry-token bg/border, border-radius:0, identical 0.12s hover). It's copy-pasted into three places with only a height delta:

RuleSize deltaOtherwise
.dauligor-wizard__button + .dauligor-class-browser__button (already shared)height:34px, pad 0 14pxtokens + radius 0 + hover
.dauligor-spell-tab-tools__buttonmin-height:32px, pad 0 12pxsame tokens + radius 0 + hover (+ box-shadow:none)
.dauligor-spell-manager__buttonmin-height:30px, pad 0 12pxsame tokens + radius 0 + hover (+ box-shadow:none)

These are the same component with cosmetic height drift (34/32/30). Recommend: one canonical button rule, all four selectors grouped, normalized to a single height. CSS-only — class names stay, so no script/HTML changes. (.dauligor-directory-tools__button is intentionally different — it sets no bg/border so it blends with Foundry's sidebar action-buttons. Leave it; add a comment saying so.)

fix B. Phantom tokens — referenced everywhere, never defined

The token block (defined on 5 roots) defines 21 variables. But a parallel vocabulary is referenced and never defined — it only "works" because every reference carries an inline fallback, so the variable name is dead weight and can never be themed:

Referenced tokenUsesDefined?Note
--dauligor-border-soft23noheavily used (spell-picker era); always falls back
--dauligor-surface-muted8noalways falls back
--dauligor-muted2noalmost certainly meant --dauligor-text-muted (which exists)
--dauligor-accent-soft2noalways falls back
--dauligor-surface-hover1noalways falls back

This is the clearest "not using what we defined" issue. Two ways to fix (your call — see questions):

  • Define them — add the 4 real ones to the canonical token block with their de-facto fallback values, and fix --dauligor-muted → --dauligor-text-muted. Makes them real, single-source, themeable.
  • Collapse them — replace each reference with the nearest existing token (e.g. border-soft → border, surface-muted → panel-soft), then delete the phantom names.

fix C. Token scope — defined on only 5 roots

The token block lives on .dauligor-importer, .dauligor-class-browser, .dauligor-class-options, .dauligor-spell-manager, .dauligor-feature-manager. CSS variables inherit through the DOM, so blocks nested inside those windows are fine — but a standalone window whose root isn't in that list (and isn't nested under one) won't inherit the tokens and will silently fall back. Candidates worth verifying: .dauligor-gm-console, .dauligor-subclass-preview, the feat browser. Recommend: hoist the token block to cover every top-level window root (one shared selector list), so tokens are guaranteed everywhere.

leave D. Primitives that are correctly distinct (do NOT merge)

Applying the "same function" test, these re-defined suffixes are genuinely different and should stay separate:

  • __empty — a 11px inline hint (class-browser) vs a centered 30px icon+serif "empty pool" (spell-picker) vs a boxed note (wizard). Different empty-state weights. (Could share the muted-text token only.)
  • __footer — a status+actions flex bar (importer) vs 2-column grids (class-options, sequence). Different layouts.
  • __badge — a label chip (spell-manager) vs an uppercase status tag (feature-manager) vs a tiny V/S/M indicator with ritual/conc variants (spell-picker). Three different jobs. (Naming is loose — "badge" vs "tag" — but visuals differ by function.)
  • __toolbar — mostly flex/grid containers tuned per window. Leave.

fix (low priority) E. Hardcoded colors where tokens exist

196 hardcoded hex values total. Most are legitimate (gradients, one-off shades), but some duplicate a token value and should reference it instead — e.g. #242424 ≈ --dauligor-panel-alt (#252525), and footer background:#141414 is an un-tokenized "deep panel". Low priority; worth a spot-pass once tokens are settled, not a blind find/replace.

Recommended order

  1. Buttons (Finding A) — group into one canonical rule, normalize height. CSS-only, no HTML churn, highest "unified feel" payoff.
  2. Phantom tokens (Finding B) — define-or-collapse per your choice. CSS-only.
  3. Token scope (Finding C) — hoist the token block to all window roots. CSS-only.
  4. Hardcoded-color spot-pass (Finding E) — only the clear token-duplicates. Later.
