Duplicate Components Audit

branch foundry-module · 2026-05-30 · cross-file scan of the 15 split stylesheets

I parsed every rule across the new files, normalized each declaration block, and looked for the same styling under different names. Verdict: there are genuine repeats — three worth acting on, plus a couple of minor ones — and a few things that look duplicated but are legitimately distinct (so the guide still earns its keep). Ranked by payoff:

consolidate 1. The "detail pane" is built twice

subclass-preview (in class-browser.css) and option-picker (in option-picker.css) each define a near-identical detail pane — image/name/meta/section/body. Several rules are byte-identical:

Update on closer inspection (2026-05-30): a rule-by-rule diff shows the two panes are mostly the same but have genuinely diverged in three spots — so they are not a clean single component:
  • __detail-header differs by design: option-picker is a grid (64px minmax(0,1fr)) because it shows a 64px option image; subclass-preview is a simple stacked column (no image). Forcing one layout would break the other.
  • __detail-body color: subclass uses var(--dauligor-text); option-picker hardcodes #d8d4c8 (a near-identical shade — a token-discipline miss).
  • __detail-body p margin differs trivially (0 0 12px vs 0 0 0.5em 0).
Identical (safe to share): __detail, __detail-name, __detail-meta, __detail-section-title, __detail-body strong, __detail-body p:last-child. Option-picker also has unique -img / -pane / -summary / -section / -empty.

Recommendation (revised): the panes share a family but legitimately differ (images). Options:

  • Reconcile into a shared .dauligor-detail with a header modifier — base header = stacked column; add __header--with-image (grid + 64px) that option-picker opts into. Unify body color to the token. Preserves both looks; needs JS markup changes in 2 files + a Foundry eyeball.
  • Share only the identical sub-elements (__detail-name/-meta/-section-title…) — leaves a mixed markup (some .dauligor-detail__*, some block-specific), messier.
  • Leave the panes separate (they diverged for a real reason) and just fix the one inconsistency: option-picker __detail-body color #d8d4c8 → var(--dauligor-text).

Recommendation: extract the shared detail-pane rules into one place. Two ways:


• CSS-only (low risk): group the identical selectors into shared rules in base.css (e.g. .dauligor-subclass-preview__detail-section, .dauligor-option-picker__detail-section { … }) and delete the copies. No HTML change, appearance-identical.
• Shared class (cleaner, more work): introduce a real .dauligor-detail__* component and switch both render sites to it (touches the JS that builds the markup).

consolidate 2. The button family is bigger than __button

The first pass unified __button, but ~13 other button-like selectors exist, and several are the same flat button at a different height:

  • Flat-button twins (token bg/border, square, gold hover) at divergent heights: wizard__toolbar-button (28px), section-filter__toolbar-btn (26px), section-filter__footer-btn (26px). These should fold into the canonical button (normalize height) — same as we did for __button.
  • Deliberately different (leave, but document): spell-manager__footer-button (larger gold CTA with active/ghost/locked states), section-filter__pill / option-picker__pill / class-browser__chip / spell-picker__chip (stateful include/exclude/active tags, not action buttons), feature-manager__change / __scope-toggle (need a look).

Recommendation: add the three flat-button twins to the canonical rule in base.css; record the pill/chip "tag" family and the gold-CTA as their own documented patterns.

tokenize 3. 38 raw accent tints bypass the tokens we just defined

38 occurrences of raw rgba(170, 130, 80, α) / rgba(181, 136, 56, α) still sit in the CSS instead of rgba(var(--dauligor-accent-tint), α) / -accent-gold. The first pass only converted the var(--phantom, …) forms; these plain ones were missed (section-filter active/toolbar/footer states, spell-manager footer-button hovers, etc.). Recommendation: mechanical, appearance-identical conversion to the tint tokens — pure "use what we defined."

minor 4. Shared hover & empty-state patterns

  • Gold-lift hover {border-color: btn-border-hover; background: btn-hover-bg} appears 6× (button, wizard__choice, wizard__toolbar-button, button--primary, spell class-button, spell filter/reset). Accent hover {border-color: accent; color: accent-strong} 4× (feature-manager scope-toggle/change, section-filter pill/chevron). Could each become one grouped hover rule. Optional — saves a few lines, low urgency.
  • Empty states: subclass-preview__empty ≡ section-filter__empty (centered muted italic). The others (wizard__empty boxed, spell-picker__empty rich icon+serif, feature-manager__empty flex-center) genuinely differ. Merge only the two twins if at all.

architectural 5. Layout utilities (not components)

display:flex; flex-direction:column; gap:Npx repeats ~20×, {flex:1; min-height:0; overflow-y:auto} (scroll region) 3×, {display:flex; align-items:center; gap:8px} 3×. These are layout patterns, not components — collapsing them means introducing utility classes (.dauligor-col, .dauligor-scroll) and adding them in the JS markup. Bigger architectural choice; flag for later, not part of "repeated components."

leave Confirmed genuinely distinct

  • Badges — three different jobs: label chip (spell-manager), uppercase status tag (feature-manager — arguably mis-named "badge"), micro V/S/M indicator (spell-picker). Keep separate; maybe rename the tag.
  • Footers — status+action bar (importer) vs 2-col grid (class-options/sequence) vs stretch column (spell-manager). Different layouts.
What I'd do, in order: (3) tokenize the 38 raw tints — zero-risk, do now; (1) consolidate the detail pane — CSS-only grouping into base.css; (2) fold the 3 flat-button twins into the canonical button. (4)/(5) are optional polish. All appearance-preserving and CSS-only (no markup changes) unless you want the cleaner shared-.dauligor-detail class. Then you reload Foundry to confirm.
