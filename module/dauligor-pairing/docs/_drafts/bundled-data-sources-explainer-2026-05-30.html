Bundled data/sources/ — what it is

module/dauligor-pairing/data · 2026-05-30
Resolved (2026-05-30): deleted. Per the owner, the bundled offline data (data/sources/, sources.zip, dauligor_artificer_full_export.json) was removed — it was stale and the module reads everything live from /api/module/* (R2-backed, so repeat requests are cheap). A local cache may be revisited later (low priority) — see TODO.md "Local data cache". This doc is kept as the rationale + a description of what was there.

data/sources/ is an offline, file-based mirror of the app's live /api/module/* source library. It uses the same payload kinds and a folder layout that mirrors the live URL routes — so the importer can read a source library from disk exactly as it would from the server. It is the historical seed data from before the live export pipeline existed.

Layout = the live URL routes, on disk

data/sources/
  catalog.json                         dauligor.source-catalog.v1     ← all sources (the index)
  <slug>/                              one folder per source (phb, tce, scag, xge, dsotdq, rj, uah, vrgr)
    source.json                        dauligor.source.v1             ← source detail (name, cover, rules, tags)
    classes/
      catalog.json                     dauligor.class-catalog.v1      ← class list for this source
      <identifier>.json                dauligor.semantic.class-export ← the FULL class bundle (e.g. bard.json)
    spells/catalog.json                dauligor.spell-catalog.v1
    bestiary/catalog.json              dauligor.bestiary-catalog.v1
    items/catalog.json                 dauligor.item-catalog.v1
    journals/catalog.json              dauligor.journal-catalog.v1

Each file's kind matches the live endpoint it mirrors: data/sources/catalog.json ≙ GET /api/module/sources/catalog.json, phb/classes/bard.json ≙ GET /api/module/phb/classes/bard.json, etc. Same shapes, just static files instead of the D1-backed worker.

What's actually in it (sparse skeleton)

8 sources are listed in the catalog, but almost everything is an empty stub:

SourceClasses (catalog / full JSON)Spells / Bestiary / Items / Journals
phb5 / 5 (barbarian, bard, cleric, druid, sorcerer)all empty
tce1 / 1 (artificer)all empty
scag, xge, dsotdq, rj, uah, vrgr0 / 0all empty

So only 6 PHB+TCE classes carry real semantic.class-export data; every spell/creature/item/journal catalog and the other six sources' classes are present-but-empty placeholders.

Does the module still use it? Mostly no.

In normal operation the module fetches the live /api/module/* endpoints (via the apiEndpointMode setting → localhost:3000 or www.dauligor.com) — that's where current data comes from (the class browser, spell browser, and the Bard fetch we just ran all hit live URLs). The bundled files are largely vestigial:

  • SOURCE_LIBRARY_FILE = data/sources/catalog.json is defined in constants.js but isn't actually fetched anywhere in the scripts.
  • CLASS_CATALOG_FILE points at data/classes/catalog.json — a path that doesn't even exist (the real data moved to data/sources/). It's the defaultClassCatalogUrl fallback, so that fallback is dead.
  • SAMPLE_FILE = data/sample-character.json — also missing; the "Import Sample" button references it.

The two loose files

  • data/dauligor_artificer_full_export.json — a standalone semantic.class-export sample (Artificer), handy as a test fixture for the class normalizer.
  • data/sources.zip — a zipped copy of the sources/ tree (pairs with serve-sample.ps1 for serving the sample library over a local URL).
The catch (why Bard looked wrong earlier): these are static committed snapshots, so they drift from live. The bundled phb/classes/bard.json had no Light Armor and skills.choiceCount: 0; the live bake is correct (Light Armor + one skill + one instrument). Live is authoritative; the bundled files are stale.
Recommendation: they're stale and partly dead-referenced. Options: (1) delete data/sources/ + the dead SAMPLE_FILE/CLASS_CATALOG_FILE constants if offline import isn't a goal; or (2) keep an offline mode but re-export the tree from the live bake and fix the constants to point at data/sources/.
