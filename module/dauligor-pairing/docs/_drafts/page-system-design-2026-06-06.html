Dauligor Page System — Design

Roadmap #1 · the "live-content bridge / Phase-2 viewer" · 2026-06-06 · branch foundry-module

module-sideunblockedgreenfield Load the app's articles, system-page glossary, and campaign content inside Foundry, rendered natively, with clickable cross-references. Auth + CORS are live, so the module reads /api/lore + /api/campaigns with the user's session token (incl. private campaign data).

1 · What the contract investigation settled

  • The viewer = the Bearer token's user. The server filters by the account's role + active_campaign_id (drafts, dm_notes, note blocks, and un-revealed secret blocks are stripped server-side). No impersonation. The viewer requires login.
  • Content is block JSON, NOT HTML. /api/lore + /api/campaigns return raw block rows {id, block_type, order, config} (config a JSON string). The module must render blocks→HTML itself. The app's LayoutBlocks.tsx is React/Tailwind — spec only, not reusable.
  • Cross-refs arrive RAW in body text (@kind[id]{display} entity, &kind[id]{display} rule). The module's enricher must process them. The server's _bbcode.ts deliberately leaves refs untouched (it's for spell-export only).
  • System pages have NO REST endpoint — readable via POST /api/d1/query with a normal player token today, or via a new dedicated endpoint (handoff). See §6.
  • We already have the hard parts. The module's normalizeHtmlBlock/applyBbcodeTags (BBCode→HTML incl. tables, just hardened) and renderDescription/formatFoundryLabel (ref-name resolution) are directly reusable for block bodies + refs.

15 types. Body fields are BBCode → run through normalizeHtmlBlock. note/secret are already server-stripped by role/campaign; the renderer also gates them defensively using the logged-in profile's role.

Per-surface allow-lists differ (articles: no recommended/definition; campaign home: no note/secret/definition; system pages: include definition). The renderer handles all; surfaces just won't contain the excluded ones.

4 · References (the clickable cross-links)

Body text carries @kind[id]#anchor{display} (entity) and &kind[id]{display} (rule). The enricher maps each to an action:

refbehavior
@article[slug]open the viewer on that article (in-Foundry)
&condition[prone] / any & ruleopen the viewer on the system-page entry (needs system-page access — §6)
@spell/class/feat/item[id]decision: open the module's existing compendium browser, OR link to dauligor.com/compendium/…, OR plain badge (v1)
@subclass/option-group/unknownnon-clickable styled badge (app behaves the same)

v1 resolves refs inside the viewer. A later phase can register CONFIG.TextEditor.enrichers so the same refs are clickable in journals + sheets (the original bridge plan — do NOT override enrichHTML; push enrichers).

5 · Auth & gating

  • Every fetch goes through authFetch (Bearer + refresh). Not logged in → the existing login card/dialog; the viewer shows a "log in to view" state.
  • The server already enforces visibility (role + active_campaign_id). The block renderer also gates note/secret client-side (defense-in-depth) using getProfile().role.

6 · System-page access — DECISION

Option A — /api/d1/query now. Works today with a player token (system_pages/system_page_blocks aren't protected-read). Replicates 2–3 SELECTs (page by identifier + kind-map, then blocks, then collectAnchoredBlocks). Downside: couples the module to raw SQL + table names (drift risk); sends SQL through the generic proxy.

Option B — recommended handoff a clean endpoint to dauligor-applications: GET /api/system-pages/<identifier> → {page, blocks, entries} and …/<anchor> → the resolved entry (wrapping the existing fetchSystemPageDetail/resolveSystemEntry), same open CORS as /api/lore. Mirrors the /api/lore/articles/<id>/blocks pattern; stable long-lived contract. Blocked on the app until it lands — but `&` refs are the last phase anyway, so the timing fits.

Articles + campaign pages need no new endpoint (auth + CORS already done) — only the &-rule refs do. So #1 starts fully unblocked; the system-page handoff can be in flight while phases 1–3 build.

7 · Reuse map (don't reinvent)

  • BBCode→HTML: normalizeHtmlBlock + applyBbcodeTags (class-import-service.js) — handles the full tag set incl. tables (just hardened for embedded-in-HTML).
  • Ref-name resolution: renderDescription's ref pass + formatFoundryLabel (importer-base-features.js).
  • Transport: authFetch (auth-service.js) — Bearer + refresh-on-401.
  • Browsing UI: the shared section-filter-panel.js (search/tag filter) for the library browser.
  • Window chrome: ApplicationV2 + the dauligor-importer-app theme (dark panel + tokens).

8 · Cross-branch ownership

The viewer files (dauligor-viewer.js, enrichers/, dauligor-viewer.hbs, dauligor-viewer.css) are reserved for system-applications and jointly owned with foundry-module (registry, owner-granted 2026-05-30). Per the owner's "I build module" call, foundry-module builds them — coordinate with system-applications (append-only on scripts/main.js) so neither clobbers the other. The optional system-page endpoint (§6 Option B) is an app-side handoff to dauligor-applications.

9 · Build phases

Phase 1 The engine: content-service.js + layout-blocks.js renderer + ref enricher. Verify headless on a real article's blocks (render → HTML, no leftover BBCode/refs).

Phase 2 The DauligorViewer window: render an article by slug from its blocks; back/history nav; Contents rail. Launcher "Dauligor Library" entry → opens the viewer.

Phase 3 Article browser: list + search (section-filter) + open. Ref-clicks (@article) navigate within the viewer.

Phase 4 Campaign home view (/api/campaigns/<id>/home-blocks) + system-page entries / &-refs (needs §6 access).

Phase 5 Optional: Foundry-wide CONFIG.TextEditor.enrichers so refs in journals + sheets open the viewer; hover tooltips.

10 · Open decisions

D1 — System-page access: Option A (/api/d1/query now) or Option B (handoff a clean endpoint). Rec: B, with phases 1–3 proceeding in parallel.
D2 — Compendium-entity refs (@spell/class/feat/item): open the module's existing browsers, link out to dauligor.com, or non-clickable badge for v1? Rec: link-out for v1 (cheap; the browsers can come later).
D3 — Foundry-wide enrichers (refs clickable in journals/sheets): in scope for v1, or viewer-only first? Rec: viewer-only v1, enrichers in Phase 5.
D4 — Scope order: articles first (Phases 1–3), then campaign + system pages (Phase 4)? Rec: yes.
