Bonus Ecosystem Investigation

2026-06-12 · foundry-module branch · 9-agent research workflow, all load-bearing claims adversarially verified against primary sources (dnd5e 5.3.1 source, Midi-QOL 13.0.58 source, AC5e source @ 0b5d5a7e, GitHub/Wayback archives). Trigger: Crimson Offering's enchant damage is inert; babonus was the proposed home for a fix and turned out deprecated.

Bottom line: we do not need to build a bonus engine, and we do not need AC5e for the Crimson Rite. Core dnd5e 5.3.1 natively expresses "this weapon deals +(scale die) typed damage while enchanted" via the enchantment-only change key system.damage.parts. The fix is an app-side authoring change, not new module machinery. AC5e is the right future add-on if/when we need conditional bonuses; babonus is dead and legally un-vendorable.

1 · Build-a-Bonus (babonus) — dead, and off-limits

Status: deprecated, deleted, proprietary. Archived by the owner Nov 2, 2024; the GitHub repo has since been deleted (404). Final release v12.5.8 (~Oct 26 2024) supports Foundry 12 / dnd5e 4.x only — it never supported our stack and will not activate on Foundry v13 + dnd5e 5.3.1.
  • Why it died: dnd5e 4.1's roll refactor (per dnd5e issue #4692). Author krbz999 ("Zhell") moved to dnd5e core development (merged PRs #4855, #5514).
  • The future is core: dnd5e issue #4692 — "Integrate parts of Build-a-Bonus into the core system" is open, epic / priority:high, milestoned dnd5e 6.0.0, assigned to a core dev. The babonus niche is being absorbed into the system itself.
  • License: "All rights reserved" — proprietary. We cannot vendor or fork babonus code into dauligor-pairing. (The Frost-Jack/babonus re-upload's MIT file is an invalid relicense of proprietary code — never treat that repo as vendorable.) Reimplementing the concept independently is fine; copying code is not.
  • No viable forks — only a stale 2023 mirror and the v12-only re-upload.

2 · The decisive finding — core dnd5e already does this

The official Elemental Weapon pattern (functionally identical to Crimson Offering) is documented by the dnd5e wiki ("up to date as of 5.3.0") and implemented in EnchantmentData._applyLegacy (dnd5e.mjs 71008–71101): an enchantment-only virtual change key that appends a typed, locked damage part to every attack/damage/save activity on the enchanted weapon. Each part is its own roll, typed by its own types set — natively fire/acid/etc., no flavor-bracket tricks.

The canonical Crimson Rite enchantment changes

// On the type:"enchantment" effect delivered by the enchant activity
// (restrictions.type = "weapon") — ONE effect per rite damage type:
[
  { "key": "name",                "mode": 5, "value": "{} (Crimson Rite: Fire)" },
  { "key": "system.damage.parts", "mode": 2,
    "value": "[[\"@scale.alternate-blood-hunter.rite-die\", \"fire\"]]" }
]
  • @scale resolves end-to-end — damage rolls use activity→item→actor roll data, which spreads actor.system.scale (dnd5e.mjs 12524 → 17813 → 10650 → 26389). Requires the actor's class item identifier alternate-blood-hunter + ScaleValue id rite-die (our import provides both; dice-scale shape fixed in a8ae707).
  • Crit-correct — a native damage part's dice double on crits like all weapon damage (RAW for the Rite). (AC5e's synthetic bonus parts deliberately don't double.)
  • Midi-proof — verified in the installed Midi 13.0.58 source: Midi subclasses dnd5e's activities and delegates the damage roll to dnd5e (super.rollDamage, midi-qol.js 7389) on a re-prepared item clone that re-applies enchantments (dnd5e Item5e.clone → applyActiveEffects; the historical clone bug was fixed by Midi 12.4.21/13.0.52 — we run 13.0.58).
  • Gotcha: the value is JSON parsed inside a bare try/catch — malformed JSON silently no-ops. The app editor must validate.
  • Gotcha: an enchantment effect only applies when type==="enchantment" && origin && origin !== parent item uuid (24562) — fine when applied via the enchant activity (dnd5e sets origin), inert if ever embedded directly with null origin.

Why the current authoring is inert (root cause, confirmed)

system.bonuses.mwak.damage is read from the actor at roll time (dnd5e.mjs 12551); enchantment changes apply only to the item (23304). The change writes nowhere meaningful — with or without Midi. Those system.bonuses.* keys are only correct on actor-applied effects (e.g. a transfer effect on a feature).

Aside: Elemental Weapon itself is not in the free dnd5e packs (it's paid-PHB-module content) — which is why the DB's imported copy lacked an enchant effect. Magic Weapon and Shillelagh are the in-system reference examples.

3 · AC5e (automated-conditions-5e) — the right successor, for a different job

Adopt later, if/when we need conditional bonuses. MIT-licensed, exceptionally maintained (15 releases May–Jun 2026; dedicated v13 line verified dnd5e 5.3.3 — exact stack match), deep first-class Midi-QOL integration. It can express the Crimson Rite, but for enchantments it needs a two-part workaround — native is strictly simpler for this case.
  • What it adds over core: a declarative conditional-bonus flag engine — flags.ac5e.<actionType>.<mode> with a rich value grammar: condition expressions against roller/opponent/aura state, itemLimited, once/oncePerTurn, opt-in dialog checkboxes, uses-consumption, auras, grants.*. This is "only vs undead", "while below half HP", "opt-in smite-likes" territory — what babonus did.
  • Critical caveat: AC5e reads flags from actor.appliedEffects only — dnd5e excludes enchantment-type effects from that list, so an AC5e flag inside an enchantment's changes is inert (same failure class as our bug). The working enchant pattern is two-part: enchant writes a marker flag on the weapon + an actor-side effect carries the AC5e bonus with a condition on the rolled item's marker. Workable, but native needs none of it.
  • Midi coexistence: explicit and deep (defers init to midi-qol.midiReady, dedicated midi hooks file, reads Midi config to avoid double-applying). Rule: one owner per overlapping category (statuses/range/visibility).
  • Risks if adopted: single-author bus factor (the same failure mode that killed babonus — though MIT means we could legally vendor it), plus a new player-facing install.

5 · Recommendation & actions

  1. Withdraw the "build a mini-babonus" idea. Native covers the live case; AC5e covers the conditional tier if needed; dnd5e 6.0 is absorbing the niche. A custom engine would duplicate both.
  2. App-side (handoff filed): the enchantment editor should author system.damage.parts extra-damage parts (structured control: formula + damage-type enum, validated JSON) and stop offering actor-scoped system.bonuses.* keys inside enchantments. Immediate data fix for Crimson Offering's 8 effects included in the handoff.
  3. Live smoke test (the one remaining unknown): everything above is verified by code-reading; one in-Foundry roll with the corrected authoring closes it — covered by the standing verification task.
  4. Defer AC5e until a real conditional-bonus need appears in the enchantments/features catalog; revisit at dnd5e 6.0 (core may obviate it).

Open items inherited from the completeness critic: live smoke test (incl. crit doubling + Midi clone path on damage.parts specifically); audit the enchantments catalog for conditional bonuses (decides if AC5e ever becomes necessary); locate every app-side emitter of the inert system.bonuses.*.damage keys + count affected D1 rows; DAE 13.0.26 interaction unexamined (low risk — DAE doesn't own enchantment application); consumption fix (separate handoff) is a prerequisite for the rite's HP cost.
