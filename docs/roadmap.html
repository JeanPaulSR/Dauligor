How to read this page

The roadmap is split into five sections based on how close each item is to being worked on, not how big it is. An item moves down the page as it gets picked up: from "On the bench" → "Up next" → "Right now" → "Recently shipped." A few one-line conventions:

  • Coloured pills show the item's current state and rough size. Right now Up next On the bench Shipped Small Medium Large
  • "What it is" is a plain-English description for someone who hasn't read the code. "Why it matters" says who benefits and how. "Status" tracks what's done and what's left.
  • Dependencies are listed when an item is waiting on another piece to land first.

Right now — active work Two app-side prerequisites before the larger initiatives below pay off.

Two pieces. The first cleans up known defects in the BBCode rich-text system so the authoring layer is reliable before we lean on it. The second adds the missing authoring UX for cross-references — linking spells, items, feats, articles, and so on — so admins can actually wire content together inside the app. Both must land before any of the larger initiatives below pay off; the live-content bridge in particular renders references that nobody can currently author.

1. BBCode audit — fix the rich-text foundation

What it is
BBCode is the storage format for every piece of rich-text content in The Archive — lore articles, class descriptions, spell text, feat text, item descriptions, DM notes. It's converted to HTML for display by the BBCode renderer and round-tripped through the TipTap visual editor. There are several known defects in how specific tags parse, render, or survive the round-trip; this item is a focused audit-and-fix pass to address them.
Why it matters
BBCode sits in the critical path for everything else on this roadmap. Every authoring action goes through it. The cross-reference picker below will write BBCode. The live-content bridge will read BBCode and rewrite it for Foundry. If BBCode has corner-case defects today, every dependent feature inherits them — and we'll trip over them constantly while testing the work above. Fixing them first removes that risk.
Status
User has flagged the need; specific defects to be enumerated. Once the list is assembled, each becomes a checklist item with a focused fix. The audit should also establish a small regression-test surface so the same defects don't return.

2. Cross-reference authoring — let admins link content together

What it is
Today there is no convenient way to link one piece of Dauligor content to another. The BBCode primitive exists — [ref|kind|id]Display[/ref] renders as a styled cross-reference link — but only three kinds are wired up: spell, class, condition. There is no item, article, feat, race, or background kind, and there is no editor button anywhere that inserts a cross-reference. The only way to author one today is to switch the editor into raw Source mode and type the BBCode by hand, knowing the target's identifier in advance.
Why it matters
A campaign-management app whose admins cannot easily link articles to relevant spells, NPCs to factions, or quests to magic items is missing one of the core utilities of the genre. This piece closes that gap. It also produces the authored content that the live-content bridge later renders — without cross-references in the data, the bridge has nothing to resolve.
Status
Two concrete pieces of work:
  • Add the missing kinds to the BBCode resolver (item, article, feat; plus race and background once those tables exist). Each kind needs a URL route and matching round-trip support so the TipTap editor doesn't drop the reference on edit.
  • Add a Cross-Reference button to the editor toolbar that opens a picker dialog: choose a kind, search by name within that kind, click to insert. Works in both Source mode (inserts the BBCode) and Visual mode (inserts the equivalent link node).
Built on top of: piece #1, because the picker writes BBCode through the editor toolbar and we don't want it landing in a defective tag pipeline.

Up next — clear path, not yet started The live-content bridge (Foundry-side payoff), plus the long-standing races + backgrounds gap.

Once the app-side authoring foundation is solid, the live-content bridge becomes the next big initiative. It's a multi-phase plan that turns the Foundry VTT companion module from a static content importer into a live viewer of canonical Dauligor content. Phase 1 foundation already shipped; the pieces below carry it through to a complete experience. Races + backgrounds is a separate, independent track that closes a long-standing content-type gap.

3. Content hashing — "this content changed" signal

What it is
Every piece of content in the database (spells, classes, feats, items, features, etc.) gets a small fingerprint — a hash — computed from the meaningful parts of its data. When the DM edits a spell, the fingerprint changes. When nothing changes, neither does the fingerprint.
Why it matters
The Foundry module already imports Dauligor content into a player's game world. After this lands, the module will be able to ask "is the copy on this character's sheet still up to date?" and answer the question by comparing the local fingerprint against the latest one from the app. Without fingerprints, the only ways to detect drift are "compare every field" (slow) or "re-fetch everything just in case" (wasteful).
Status
The database columns to hold the fingerprints exist (added 2026-05-27, currently empty). What remains:
  • A small piece of code that computes the fingerprint every time a piece of content is saved.
  • A one-time backfill that walks every existing row and fills in its fingerprint.
Up next Medium Blocks #5

4. Article system unification — make wiki articles addressable

What it is
Today, wiki articles are organised by hierarchy — a region article contains city articles, a city contains neighbourhoods, and so on. They don't carry a stable identifier that's safe to use as a link from anywhere else. Every other content type in the app does carry one (spells, classes, feats — they're addressable by a short slug paired with their source book). Articles need to catch up.
Why it matters
Three concrete problems disappear once articles have stable identifiers:
  • Cross-references stop breaking. Today, an @article[deep-shadow-cult] link is ambiguous — two campaigns can each have an article named "Deep Shadow Cult" and the link has no way to tell them apart. After this lands, every article is addressable as (identifier, source), which is unique.
  • "System pages" become possible. A new article type — a parent article with a set of named glossary children — needs to be able to link to a specific child without using its database row ID. (This mirrors how dnd5e's Conditions and Rules journals work in Foundry.)
  • Deep links from Foundry work. The viewer in piece #5 needs to be able to compose a stable URL pointing back at the app for "open in browser."
Status
Plan finalised; database migrations not yet written. Sequence:
  1. Add an identifier column to the articles table; backfill from existing slugs.
  2. Add a source column so identifiers can be scoped per book.
  3. Add the uniqueness constraint once duplicates are resolved.
  4. Add a content fingerprint column (same shape as piece #3).
  5. Update the wiki routes and the BBCode @article[…] resolver to use the new identifier.
Up next Large Waits on #4

5. Live viewer — Foundry reads canonical content from the app

What it is
The first half of the live-content bridge. After this lands, Foundry text that mentions a Dauligor article (e.g., @article[the-shattered-spire]) will become a clickable link inside any Foundry window — journal entries, item descriptions, chat cards, anywhere text is shown. Clicking the link opens a Foundry window that fetches the article's current HTML from the app, renders it fresh from the source, and caches it for the current world so the next click is instant.
Why it matters
Today, every piece of Dauligor content that reaches Foundry is a copy. When the DM edits an article in the app, the copies in Foundry don't update. Players who saw the old version on their item card continue to see the old version forever. After this lands, reference-style content (articles, conditions, rules) is always live — there's nothing to update because there's no copy.
Status
Specification finalised; no code yet. The shape:
  • New endpoints on the app side that serve article HTML and a short hover-tooltip summary.
  • A new content type (system page) for parent-plus-glossary structures like Conditions and Rules.
  • A new Foundry window class that handles the click-to-open behaviour.
  • Foundry text enrichers that detect the new reference syntax and rewrite it into clickable links.
  • A per-world cache so opening the same article twice is instant, and so the table can read offline.
Built on top of: piece #3 (so the cache can detect when its copy is stale) and piece #4 (so links can address articles unambiguously).
Up next Medium Waits on #5

6. Drag-and-drop construction — Foundry items built on demand

What it is
The other half of the live-content bridge. After the viewer lands, this piece teaches the module to handle drag-and-drop of mechanical content. When a player drags a Fireball reference onto their character sheet, the module fetches the current Fireball data from the app, builds a real Foundry item out of it, and drops it into the inventory — complete with the proper damage rolls, save DC, scaling, and visual effects.
Why it matters
Drag-and-drop is the everyday gesture for adding mechanical content to a character. Today, that gesture requires the content to already exist in a compendium — which means the DM has to have run an import in advance. After this lands, the content doesn't need to be pre-imported. The drag just works.

The content created this way is a real Foundry item, not a live placeholder, because mechanics (damage rolls, ability checks, animations) need a concrete document to attach to. But each created item carries a fingerprint so the next piece can detect when it's gone stale.
Status
Design done; relies on a stable Foundry-ID derivation helper that shipped in early May 2026. Implementation pattern reuses the existing batch-import code, exposed as a single-item path.
Up next Small Waits on #3 + #6

7. Update detection — stale content tells you it's stale

What it is
Once every piece of Dauligor-authored content in Foundry carries a fingerprint, the module can periodically ask the app "any of these out of date?" The app returns a list of items whose fingerprint has changed since the player's copy was made. The module shows a subtle glow on stale item sheets; clicking the glow opens a diff modal where the player can opt in to a refresh or stamp the current fingerprint to silence the glow until the next change.
Why it matters
The "stale Fireball" problem that the live-content bridge was designed to solve only half-disappears with the viewer. The other half — for content that has to be a real Foundry item, like spells and equipment — needs this piece. After this lands, drift becomes a problem players are aware of and can fix in one click, instead of an invisible issue that quietly desyncs the table from the canon.
Status
Design done. Requires the fingerprint backfill from piece #3 to be populated everywhere.
Up next Small Waits on #6 + #7

8. Hardening — offline snapshots, auth audit, documentation

What it is
The polish pass that follows the live-content bridge build-out. Three concrete pieces:
  • A one-click "bake to world" snapshot button that converts the per-world cache into real, persistent Foundry journal entries — useful for tables that play in offline / no-internet venues.
  • An auth audit on the module endpoints. Today they serve public reads, which is fine while the app is open-source and content is shareable. If that changes, a per-world API key model is ready to swap in.
  • A proper architecture document for the whole live-content bridge, replacing the working spec that's been guiding the build-out.
Why it matters
Without the snapshot, tables that lose internet during a session lose access to live content. Without the auth audit, the option to gate content is left informal. Without the document, future-Claude (and future contributors) have to reverse-engineer the design from code.

9. Races and Backgrounds — first-class content types

What it is
Today, races and backgrounds exist in the compendium as placeholder pages. You can browse them; you can open an editor; but there are no proper database tables for them and no proper Foundry export. They piggy-back on the feats system: a "race" is currently a feat with feat_type='race'. That works for the simplest cases and explodes the moment a race needs anything beyond a list of traits (subraces, size categories, speed differences).
Why it matters
Player character creation is incomplete without first-class race and background support. The placeholders today are a known gap that DMs work around by treating races as "just feats." Closing this gap means players can author a custom Tabaxi Sailor (with subraces and starting equipment) end-to-end inside the app, export it to Foundry, and play with it.
Status
Plan written; not started. Required pieces:
  • Two new database tables (races, backgrounds) with the standard entity fields plus type-specific extras.
  • Server-side export endpoints so the Foundry module can fetch them.
  • Module-side importers that drop them as Foundry items of the right type.
  • Editor surfaces that show the type-specific fields (size, speed, languages, subraces for races; skill / tool / equipment / starting feature for backgrounds).
  • A documented decision about what happens to the existing "feat as race" pattern — coexistence or retirement.

Tail-end — follow-ups on shipped work Mostly-complete tracks that have one or two small loose ends.

These are the small remaining tasks on initiatives that have already shipped their main value. Low priority — each one is a polish item, not a missing feature.

Tail-end Small

Module-side item importer for scaling columns

What it is
The app side already serves item data with embedded scaling tables (a "scaling table" is something like "this dagger does +1 damage per character tier"). The Foundry module side doesn't yet consume that endpoint — an item importer exists for the base item data but doesn't read the scaling rows. Pattern is identical to the feat importer that already does this.
Why it matters
Without this, DMs can author scaling items in the app and the values show up in the browser, but the module exports them without the scaling intact. Round-trip is incomplete.
Tail-end Small

Stale-language sweep on module-side docs

What it is
The system's "scaling" concept was originally class-only, then extended to feats, races, backgrounds, and items. Three core module-side docs got the corresponding language updates; eight other docs still read as if scaling were a class-only feature.
Why it matters
A contributor or AI agent reading those eight docs would conclude that scaling only applies to classes, which is no longer true.
Tail-end Small

ItemBumpUses follow-ups

What it is
"ItemBumpUses" is a feature that lets one piece of content (a feat or an item) bump the uses-per-day counter on another piece of content. Example: a feat that says "your Sneak Attack feature gets one extra use per short rest." Version 1 shipped end-to-end in May 2026. A few extension cases remain: feats bumping other feats, items bumping things (instead of only feats bumping things), and a module-side audit surface for "where did this bump come from?"
Why it matters
The bumping mechanic is in production and works for the common case; these follow-ups extend it to the long tail of less-common cases.

Bench Medium

In-app "Systems" page

What it is
A first-class page inside the app (likely at /systems) that lists the major systems The Archive exposes — Worlds, Permissions, Content Proposals, Tags, Spell Rules, Sources, Class Spell Lists, Modular Options, Compendium, Wiki, Foundry Export. Each entry is one modular section with a one-paragraph "what it does," a list of the pages that surface it, the permission model, and links to the deeper documentation.
Why it matters
Today, a new DM has to learn The Archive by clicking around. A Systems page would let them scan one screen and see what's available without having to navigate the sidebar speculatively. It's also the natural place for "what's coming next" callouts — turning this roadmap from a separate doc into an in-app feature.
Bench Large

Self-rolled authentication (Firebase exit)

What it is
Replace Firebase Authentication with an in-house implementation built on the Cloudflare stack. Firebase Auth currently issues the JWT tokens that the rest of the app trusts — it's the only Firebase service still in use. A two-stage exit plan exists: first drop the Firebase Admin SDK (replace with direct REST calls), then replace the JWT issuer entirely.
Why it matters
Strictly optional today — Firebase Auth works fine and is free at our scale. The reasons to do it are fewer moving parts (one less vendor) and full control over the auth flow (which matters if we want features like passkeys or magic-link login that Firebase Auth doesn't make easy). No urgency.

Recently shipped — for context Major work that landed in the last few months. Useful for understanding what the system can already do.

Shipped May 2026

Live-content bridge — Phase 1 foundation

The groundwork that the "Right now" section is building on. Three database changes added uniqueness constraints and fingerprint columns to the content tables; one new helper computes a stable Foundry document ID from a Dauligor source ID. None of this is user-visible on its own — it's the foundation that pieces #1 through #6 build on top of.

Shipped May 2026

Content proposals

A workflow where users with the Librarian role (or anyone holding the additive content-creator permission) can propose changes to content — tags, spell rules, spell lists — without write access to the underlying tables. Proposals enter a review queue; staff approve, reject, or revert. Proposals that drift from the current state of the content during review are flagged so reviewers don't silently overwrite intervening changes.

Shipped May 2026

Spellbook manager (four layers)

A comprehensive system for how spells reach a character. Layer 1 is the per-class spell list (which spells does a Bard even know about?). Layer 2 is the rules that expand or restrict those lists (a feat that lets you cast a single Wizard spell, a subclass that adds Cleric spells to your Wizard list). Layer 3 is the actual preparation interface in Foundry — filters, detail pane, prepared / known / in spellbook hierarchy, favourites, caps. Layer 4 is loadouts (preset combinations of prepared spells you can switch between).

Shipped May 2026

ItemBumpUses v1

The "let one piece of content bump another piece of content's uses-per-day" feature described in the tail-end section above. Version 1 covers the common case end-to-end: a feat that bumps a class feature, exported to Foundry with the bump baked into the target's uses-max formula, with an audit trail flag explaining where the bump came from.

Shipped May 2026

Spell list decoupling

Per-class spell lists now live on a dedicated endpoint that's read-through-fresh, not baked into a per-class bundle. The DM can edit a spell's tags and see the spell move in or out of class lists immediately, without a manual rebake. Lightweight summaries + per-spell full payloads + tag-driven recompute on save.

Shipped May 2026

Class slug routes

Class URLs migrated from database-ID form (the unreadable /compendium/classes/view/sjsnEpcMI1KSHbb1GeUO) to human-readable slug form (/compendium/classes/view/cleric_phb). Matches the convention already used by feats, spells, items, and facilities.
