Class & Subclass Editors — Importer Field Catalog survey

Every part of the two editors, rated by how easily it can be added to the Mark & Build importer — now annotated with the build decisions of 2026-06-05. branch manual-uploads · grounded in the live editor code + the production-seeded D1

The class importer already exists as a manual-entry skeleton (steps 1–2). This catalog maps what to add next: every field, tab, and sub-system of ClassEditor and SubclassEditor, what the importer captures today, the import-difficulty of the rest — and, after a review pass, which ones we're taking on vs. leaving to a manual picker or the editor. Everything imported stays fully editable afterward, which lowers the bar: a best-effort parse you can tweak beats no parse at all.

The guiding principle decided

The real line isn't "easy field vs. hard field" — it's "is the data actually in the source text?"

Proficiencies, features, the plain spellcasting fields, subclass→class linking — that information is in a pasted class write-up, so we parse it (and let you fix it after). Spell-slot grids, scaling tables, automation, images, bespoke advancement logic — that data is not in the prose, so it stays a manual picker or an editor job. A 🔴 tier means "id-bound / structurally hard," not "forbidden": several 🔴s are in scope because an existing editor component already solves the hard part.

How to read the tiers

Easy scalar / enum / free-text — text-derivable or a single field.
Medium a small parser, a simple select, or by-name resolution.
Hard bound to internal DB ids, a nested child entity, or a deep authored tree.
Editor images/focal points, danger zone, automation — not text.

Tier = intrinsic difficulty of the data shape. The plan for each is the separate decision, flagged in the Notes:

What the importer captures today steps 1–2

The class descriptor already writes a faithful, byte-identical skeleton through the editor's real upsertDocument('classes',…) + queueRebake('class',…). The green rows below are done:

name · identifier · source · category · hit_die · description · preview · lore · primary_ability · saving_throws (kept in sync with proficiencies.savingThrows) · starting_equipment · wealth · multiclassing · subclass_title. Everything else is written as the same empty-default shape a pristine class saves.

Decisions 2026-06-05

In scope — we parse these

  • Proficiencies. Embed the existing ProficienciesEditor component; pre-fill via the existing resolveLegacyProficiencyIds name→id matcher; you tweak after. Bonus: the base advancement tree is built from the proficiency object, so this also populates the progression base.
  • Class & subclass features. Import the textual skeleton — name · level · description · requirements · subtype · properties · uses formula. Features are granted automatically by their level (no advancement wiring), and automation is 0.4% of features → manual. Biggest work-saver (esp. the 918 subclass features).
  • Subclass importer with a parent-class picker (fills class_id + class_identifier).
  • Spellcasting "lite" — hasSpellcasting, ability, type, level, description, formula, spellbook counts.
  • Small parses — asi_levels, subclass_feature_levels, primary_ability_choice split, tags by name.

Stays manual / picker — by design

  • Spellcasting slot / known scaling tables (progressionId, spellsKnownId) — pick an existing scaling row; we do not parse the 20-level slot grid.
  • Scaling columns (Rage uses, Sneak Attack dice…) — authored in their own editor; a feature keeps its @scale… reference text.
  • Unique option groups / excluded options / mappings — id-bound; linked in the editor like scaling.

Won't auto-import — not in the source text

  • Automation — activities & effects (rolls, DCs, templates, active effects).
  • Images & focal points — a URL is trivial; the drag-to-frame focal needs the editor.
  • Bespoke advancement nodes — ScaleValue, curated "choose one of these" pools, conditional grants. (Base + by-level feature grants are automatic.)
  • ID cross-references — feature prerequisite item links, option-group wiring. (The numeric prereq level is fine.)

Class Features in scope

The "Class Features" tab edits rows in a separate features table (parent_id = the class, parent_type='class') — 185 of them across the 82 classes. A feature importer (its own descriptor, with a parent picker) captures the textual skeleton and is the single biggest work-saver:

  • In scope: name · level · description · requirements · subtype · properties · uses_max (the formula string, verbatim) · prerequisites_level · source/page. Granted to characters by level automatically — no advancement wiring.
  • Editor-only: activities & effects (automation) — but only 4 of 1,103 features use them. prerequisites_items (id links) and scaling_column_id (2 of 1,103) stay manual.

The four id-bound systems — two we take on, two stay manual

Taking on

1 · Proficiency grids. Every grant is an FK id across five catalogs, but the ProficienciesEditor component and the resolveLegacyProficiencyIds matcher already exist — so it's wire + best-effort parse + tweak, not a from-scratch resolver. And it feeds the advancement base.

2 · Child entities (features; a class's subclasses). Their own importer descriptors. Features are granted by level automatically, and automation — the only truly hard slice — is 0.4% of them. The 1,103 features are where the real authoring time goes, so this is the highest-leverage piece.

Staying manual

3 · Spellcasting scaling tables (progressionId, spellsKnownId). The slot progression isn't prose data and must reconcile to an existing scaling row. Pick it in the editor; parse the rest of the spellcasting object (booleans, ability, type, formula, description).

4 · Scaling columns & unique-option wiring. The 46 per-level value tables (Rage uses, Sneak Attack dice…) and the option-group links are authored in their own editors. A feature still carries its @scale… reference text; the scale itself is built separately.

Recommended build order

1 · Proficiencies  Highest structural payoff: embed ProficienciesEditor, pre-fill via resolveLegacyProficiencyIds, tweak after — and it lights up the base advancement tree (HP/saves/profs/subclass/ASI) at the same time. Turns the import skeleton into a functional class.

2 · Feature importer  The big work-saver (1,103 features). Skeleton fields + parent picker; granted by level automatically; automation left to the editor. Serves both class and subclass features.

3 · Subclass importer  Parent-class picker + the same field set as the class (lighter), reusing the feature importer for its features.

4 · Spellcasting "lite" + small parses  The prose-derivable spellcasting fields (scaling tables picked manually), plus asi_levels, subclass_feature_levels, primary_ability_choice, tags-by-name.

Each step is a descriptor (+ a resolver or picker) of its own; none blocks the others, and everything imported stays editable. The only parts that never come from text are automation, images, bespoke advancement logic, and the id-wiring above — exactly the parts the editor exists to author by hand.
