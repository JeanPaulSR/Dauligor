1. What a system page is

A system page is a site-consistent reference page about one system of the game — Conditions, Skills, Magic, a homebrew ruleset. It is a generic page type: you create instances of it the way you create a class or a spell. Each instance is glossary-shaped — a parent page with addressable entries (subsections) — mirroring how dnd5e ships its Conditions / Rules journals.

The two things that make it special, and that the rest of this spec hangs on:

  • Reference-addressable. The cross-reference system's & rule sigil resolves into these pages. &condition[prone] → the Conditions system page, scrolled to the Prone entry. Today those refs hover but can't navigate (their route is deliberately null) — system pages are the navigation target they've been waiting for.
  • Site-consistent. Unlike lore, a system page is canonical and global — the same everywhere, not scoped to a campaign or era, with no secrets / DM-only layers. It's rules reference, not narrative.

2. System page vs. lore article

Both are "articles" loosely, but they are different enough to be a separate table, not a discriminator on lore_articles (consistent with the project's "new table for new functionality" rule — splitting later is harder than starting split).

3. Data model proposed

Two new tables, following the established identifier + order + JSON-column conventions.

system_pages
  id            TEXT PRIMARY KEY
  identifier    TEXT UNIQUE NOT NULL   -- site-wide slug: "condition", "skill", "magic"
  name          TEXT NOT NULL          -- "Conditions"
  description   TEXT                   -- intro prose shown atop the glossary (BBCode)
  icon          TEXT                   -- optional lucide icon / image_url
  "order"       INTEGER
  created_at / updated_at

system_page_entries
  id            TEXT PRIMARY KEY
  page_id       TEXT NOT NULL REFERENCES system_pages(id) ON DELETE CASCADE
  identifier    TEXT NOT NULL          -- slug, unique within page: "prone"
  name          TEXT NOT NULL          -- "Prone"
  summary       TEXT                   -- short blurb for the hover card
  body          TEXT                   -- full entry text (BBCode); nullable if entity-backed
  source_kind   TEXT                   -- OPTIONAL binding to an existing entity ("condition")
  source_id     TEXT                   -- OPTIONAL  e.g. a status_conditions.id
  image_url     TEXT
  "order"       INTEGER
  created_at / updated_at
  UNIQUE(page_id, identifier)

An entry is one of two flavours (see §5 — this is the open fork):

  • Free entry — body authored directly (a homebrew rule, a magic subsection). source_* null.
  • Entity-backed entry — source_kind/source_id point at an existing canonical row (a status_conditions row); the reader pulls its name / description / image so there's no duplication. body may hold an optional override / note.

4. Reference integration

The page's identifier is a reference kind. Creating a page with identifier condition makes &condition[<entry>] resolve to it — no per-kind code.

&condition[prone]              → /system/condition#prone   (hover = entry summary)
&condition[prone]{knocked prone} → custom display
&skill[stealth]                 → /system/skill#stealth
  • Resolve (references.ts): look up system_pages by the kind = page identifier, then the entry by its identifier; return name + summary + route + anchor. Falls back to the entity-backed source row for name/summary when the entry is entity-backed.
  • Route (resolveRefRoute): /system/<page>#<entry> — flips the currently-null condition route on, so &condition becomes clickable.
  • Authoring: the existing & autocomplete already searches a "rule" family; it gains the system-page kinds (each page = a searchable kind, entries = its results).
  • Foundry (later): the module compiles &condition[prone] to dnd5e's native &Reference[condition=prone] where Foundry already has the system, and to a @UUID[JournalEntryPage…] into our exported system page otherwise. Out of scope here.

5. The conditions question locked: hybrid

Conditions already exist as rich data — status_conditions (identifier, name, description, image, changes/implied_ids for Active Effects) grouped by condition_categories. That table is also the mechanical source (character sheet + Foundry export). So the Conditions system page must not fork that text. Three ways:

OptionHow the Conditions page is builtTrade-off
A — Free onlyEntries authored fresh in system_page_entries; &condition re-points here.Simplest table, but duplicates condition rules text — drifts from status_conditions.
B — Entity-backed onlyEntries are pointers to status_conditions rows; reader pulls their text.No duplication, but every system page must map to a backing table — no pure-prose homebrew pages.
C — Hybrid recommendedAn entry is either free prose or entity-backed (the source_kind/source_id columns above).One extra optional column-pair. Conditions page = entity-backed (zero dup, stays canonical); Homebrew/Magic page = free prose. Matches "handled by the reference system."

Recommendation: C. It lets a system page curate existing canonical entities (conditions, skills) without copying them, and author original rules where no table exists — which is exactly the Conditions-vs-Homebrew spread you described.
