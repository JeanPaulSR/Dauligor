Subspecies design

Add child species under a parent — Elf → High Elf / Wood Elf / Drow — reusing the whole species pipeline. 2026-06-03 · branch compendium-editors

A subspecies is a variant of a parent species with its own traits, options, and features. We model it as a complete species that names a parent — so it reuses the species editor, the Foundry race export, and the live module route with no new plumbing on either side.

Confirmed decisions

1 · Export model — stand-alone race item. Each subspecies is a complete species and exports as its own Foundry race item, exactly like a base species (so "High Elf" is a full race). This matches dnd5e v5's atomic-race model (there is no subrace item type; the 2024 PHB folds old subraces into one species + a lineage choice) and the dauligor-pairing module's current contract — which has no parent/child concept. Net module-side work: zero. The parent link exists for grouping & authoring convenience, not for an export-time merge.

2 · Storage — reuse the species table. Add a self-referential parentSpeciesId column. A subspecies is a species row → it reuses the editor, _raceExport.ts, the browser, and /api/module/races/<id>.json unchanged. (A self-FK is a hierarchy, not a discriminator — so the "new table for new functionality" rule doesn't pull against it here: the shape and export are identical to a species.)

3 · Authoring UI — a "Subspecies" sub-tab on the species editor (species-only), listing & creating the children owned by the open parent — mirroring the background Features tab. The browser groups subspecies under their parent.

Data model

One new column on species

No new table. parentSpeciesId is a plain id (not JSON), so it round-trips through the editor's ...row spread automatically — no jsonFields / d1Tables change needed.

ALTER TABLE species ADD COLUMN parentSpeciesId TEXT
    REFERENCES species(id) ON DELETE SET NULL;
CREATE INDEX IF NOT EXISTS idx_species_parent ON species(parentSpeciesId);
-- NULL parentSpeciesId  = base species
-- non-NULL              = subspecies of that parent

Delete semantics — ON DELETE SET NULL (chosen default). Deleting a parent species promotes its subspecies to base species (non-destructive, recoverable by re-parenting). The alternative ON DELETE CASCADE would delete the children too — destructive, so not the default. Flag if you'd prefer CASCADE.

One level deep. A subspecies cannot itself have subspecies (no grandchildren). Enforced in the UI: the Subspecies tab and the parent picker are hidden/disabled for a row that already has a parentSpeciesId.

Shape — identical to a species

A subspecies carries the full species field set (it's the same row type):

species(
  id, name, identifier, sourceId→sources(id), page, description,
  advancements[], movement{}, senses{}, creatureType{}, tags[],   -- JSON
  speciesOptionIds[],                                              -- JSON (20260603-1600)
  imageUrl, contentHash, createdAt, updatedAt,
  parentSpeciesId→species(id)                                      -- NEW (this design)
)

Authoring flow

Subspecies reuse the entire species editor — the Subspecies tab is just a parent-scoped list + a doorway into the same editor:

Edit "Elf" (base species) ├─ Basics ├─ Traits ├─ Options ← (Species Options, phase 3 of that feature) ├─ Subspecies ← NEW tab (base species only) │ ├─ High Elf [Edit] [Delete] │ ├─ Wood Elf [Edit] [Delete] │ └─ [ + New subspecies ] ← creates a child row, pre-filled from Elf └─ Advancement Edit "High Elf" (subspecies) ← same editor, banner: "Subspecies of Elf · ← Back" ├─ Basics (no parent picker shown; it's already a child) ├─ Traits ├─ Options └─ Advancement (no Subspecies tab — one level deep)
  • Create "+ New subspecies" inserts a species row with parentSpeciesId = parent.id, then switches the editor to that child (setEditingId(childId)).
  • Pre-fill from parent. A new subspecies copies the parent's movement / senses / creatureType / advancements / speciesOptionIds / tags / sourceId as a starting point (the author then edits the deltas). Because export is stand-alone, the child needs the full trait set — pre-fill saves re-entry. Pre-fill is a one-time copy at create, not a live link.
  • Editing a child uses the same tabs; a banner shows "Subspecies of <parent>" with a "← Back to <parent>" affordance (sets editingId back to the parent). The Subspecies tab and parent picker are suppressed for children.

Foundry export

Nothing required. Subspecies are species rows, so they already export through buildRaceItemBundle(dbId) at /api/module/races/<id>.json as ordinary race items. They appear in the catalog as normal races, importable independently.

Optional (nice-to-have): an audit flag. Stamp flags.dauligor-pairing.parentRaceId = <parent.identifier> in the race export (_raceExport.ts / _speciesBackgroundShared.ts) so the module could later group a parent + its subspecies in the browser. The module ignores unknown flags today, so this is additive and safe; actual consumption is the foundry-module branch's job.

Browser / view

SpeciesBackgroundBrowser.tsx: the top-level species list shows base species only (filter parentSpeciesId == null); a parent's detail panel lists its subspecies (each linking to its own detail, since each is a full species). Exact rendering nailed during the build after reading the browser's species branch.

Build phases

1
Schema + plumbing — migration (parentSpeciesId column + index), local apply. No d1Tables/d1.ts change (plain id column).
2
Editor parent field — add parentSpeciesId to SBFormData, hydrate on load, write it in handleSave's species branch.
3
Subspecies tab — new SubspeciesTab (parent-scoped list + create-with-parent + pre-fill + jump-to-child); wire into editorSubTabs for base species only; child banner + "← Back"; suppress tab/picker for children.
4
Browser grouping — base list filters out children; parent detail lists its subspecies.
5
(Optional) Export audit flag — flags.dauligor-pairing.parentRaceId for future module-side grouping.

Relationship to Species Options

Independent but complementary. Species Options (the reusable trait library) and Subspecies both attach to a species via existing machinery: a subspecies, being a full species, can carry its own speciesOptionIds (e.g. High Elf includes the "Darkvision 60" option). Build order is flexible — Species Options phases 2–5 and Subspecies phases 1–4 don't block each other.

Why not the alternatives

RejectedWhy
Inherit + merge (base + subrace both apply)Needs export-time merge logic and the Foundry module to understand parent/child — neither exists; dnd5e v5 has no subrace item type. Stand-alone is the native, shippable model.
Dedicated subspecies tableA subspecies has the identical shape and export to a species (unlike subclasses, which are leaner than classes). A separate table would duplicate the editor / export / browser / route for no shape difference.
