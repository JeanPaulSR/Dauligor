Client-Flip Behavior Audit

Everything that changed in feel/behavior when the client moved from the Firebase User object + Firebase auth lifecycle onto the native Identity + Worker-token model — what's already fixed, what's safe, and what to keep in mind.

The bugs we hit (blank avatar, logout not sticking) share one root cause: code that assumed a Firebase User shape or a Firebase auth-lifecycle behavior. This audit sweeps for the rest of that class.

Checked & safe — no action needed

safe Settings page

Receives the user (Identity) prop but only uses it for a truthiness "are you signed in" check — no Firebase-shape field reads. Display/email/password all go through userProfile + endpoints.

safe Persistent cache holds no private data

The sessionStorage cache (PERSISTENT_TABLES) is shared content only — compendium + lore + campaigns. Per-user data (characters, etc.) uses uncached endpoints, and protected tables (lore_secrets, users) are proxy-blocked from the cacheable queryD1 path. So persisting the cache across logout/login (and across different users on one browser) leaks nothing.

safe No other Firebase-User field reads

Swept the codebase: the only auth-object Firebase-shape reads were the Navbar ones (now fixed). Other .displayName hits are on unrelated objects (form state, API user records, image/folder metadata).

safe Role changes / revocation

Every request re-reads the role from D1 server-side, so a demotion or row-deletion takes effect immediately (a deleted user 403s on the next call). The native token's role claim is cosmetic/unused for authorization.

Behavior differences to keep in mind

watch fixable now Multi-tab auth sync is gone

Firebase broadcast login/logout across all open tabs. The native token lives in localStorage and we don't listen for cross-tab storage events — so logging out in one tab leaves other tabs "logged in" until they reload (or the token expires).

Fix is small: add a window.addEventListener('storage', …) in auth.ts that re-emits when dauligor:authToken changes in another tab. Want me to add it?

watch Token is fixed 30-day, no auto-refresh

Firebase silently refreshed the ID token (~hourly). The native token is a fixed 30-day JWT — after 30 days the user must log in again; there's no sliding renewal yet. This is the planned Phase 3b POST /api/auth/refresh. Fine for now; just know long-lived sessions end hard at 30 days.

minor Error reports lose auth context for native users

reportClientError (in firebase.ts) reads auth.currentUser for uid/email — which is null for a native-only session, so those reports omit the identity. Harmless; could fall back to getIdentity().

Will need rework when Firebase is removed (Phase 4–5)

These still work today because every account has a Firebase identity during the migration window — flagging so they're not forgotten at cut-out.

phase 5 Admin "Sign-in link"

RedeemTokenPage still mints + redeems a Firebase custom token, and those sessions never adopt a native hash. Needs a native one-time-token equivalent before Firebase is dropped.

phase 5 Username change → Firebase email sync

Changing a username PATCHes /api/me, and the server syncs the Firebase login email via Identity Toolkit. A native-only account (created after Firebase removal) won't have that to sync — the path needs rework then.

phase 4 Admin user create / delete / temp-password

Still call Firebase Identity Toolkit server-side. Phase 4 moves these to pure D1 writes + the native sign-in token.
