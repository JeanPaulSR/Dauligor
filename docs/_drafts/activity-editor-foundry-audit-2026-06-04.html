Activity Editor — Foundry 5.3.1 fidelity audit

Every dropdown & window in ActivityEditor.tsx + its activity/ siblings, measured against the real dnd5e 5.3.1 — its CONFIG.DND5E lists, its templates/activity/*.hbs windows, and the activity JSON dumps in Foundry-JSON\windows. Cross-checked against the module team's correcting-activities.md contract. 2026-06-04.

Bottom line: the activity data structure is correct on purpose. The app authors a semantic activity shape; the module normalizer (class-import-service.js) converts it to native Foundry system.activities. I confirmed the conversions in code. So we do not chase Foundry's native field shapes. The real gaps are exactly the two you named: (1) several dropdown option lists carry wrong / missing / mislabelled values vs Foundry's canonical config, and (2) a few windows don't mirror Foundry's conditional layout. This doc enumerates every verified item.

The semantic contract — what NOT to touch

From module/dauligor-pairing/notes-for-app-team/correcting-activities.md, and verified in class-import-service.js: the app stays semantic, the module owns the native conversion. These mappings are deliberate — "fixing" the app side to look native would break the importer:

Our semantic shape→ Native FoundryConfirmed in
attack.type + attack.classification (flat strings)attack.type: {value, classification}normalizeSemanticAttack L3088
save.abilities (string[])save.abilitynormalizeSemanticSave L3109
healing.parts[0]healing (single object)normalizeSemanticHealing L3143
enchant.restrictionstop-level restrictionsnormalizeSemanticEnchantRestrictions L3195
summon / transform sub-fieldspartly flattened onto activity rootper the contract note
One thing to verify on the module side (not ours): normalizeSemanticHealing reads healing.types, healing.number, healing.denomination, healing.bonus from the top level of the healing object, while it also computes firstPart = healing.parts.find(Boolean). Our editor writes those values inside healing.parts[0]. If the module doesn't fall back to firstPart, exported heal rolls land empty. Flag for the foundry-module branch — but it means our heal editor should also surface the type/number/bonus at the level the module actually reads.
