Accounts on Cloudflare — Status & Local Testing

Where the "move accounts off Firebase" work stands today, and exactly how to drive it against your local dev stack.

§1 The project in one breath

Dauligor / "The Archive" runs entirely on Cloudflare now — Pages + a Worker + D1 (SQLite) + R2 — except that user passwords still lived in Firebase Authentication. This project moves passwords into D1 as scrypt hashes and has the Worker issue its own session tokens, so the whole stack is one stack. It ships in phases; each is additive and reversible, and Firebase keeps working the entire time until the final cleanup.

Where we are
The foundation, the login backend, and the full client flip are built, verified, and committed (on branch claude/kind-wright-2cb7a2, not pushed). The current state is safe to deploy as-is — it does nothing visible until its secret + DB column exist, then native login simply becomes available.

§3 How login works now

LOGIN (user types username + password)
 1. client → POST /api/auth/login
   • adopted account → 200 + native JWT stored in localStorage. Done, no Firebase.
   • not adopted yet → 401, so the client…
 2. …falls back to Firebase sign-in, then fires POST /api/auth/adopt
   → writes the scrypt hash to D1 so the next login is native.

EVERY REQUEST carries Authorization: Bearer <token> from getSessionToken()
 → the server verifies it (our HS256 secret OR Firebase JWKS) and reads the role from D1.

Net effect: accounts migrate themselves the first time each person signs in. No reset email, no disruption. The native token lives in localStorage under dauligor:authToken.

§4 Testing against your local server

Your local stack is the main checkout's dev launcher — app on :3001, worker on :8788. Three one-time prerequisites, then a five-minute walkthrough.

Prerequisites (one-time)

  1. Get this code into your running checkout

    The commits live on claude/kind-wright-2cb7a2. From the main checkout (E:\DnD\Professional\Dev\Dauligor), fast-forward your active branch onto them:

    git merge --ff-only claude/kind-wright-2cb7a2

    It's a clean fast-forward — nothing to resolve.

  2. Set the signing secret

    Native auth is disabled until AUTH_JWT_SECRET exists (without it, login just falls back to Firebase). Add a line to the repo-root .env:

    # .env  (repo root — read by the dev server)
AUTH_JWT_SECRET=pick-a-long-random-string-at-least-32-chars

    Any long random value is fine locally. (For production this becomes a Cloudflare Pages + Worker secret — not needed for local testing.)

  3. Add the password column to your local D1

    The login query reads users.password_hash; apply the migration to your local database once:

    cd worker
npx wrangler d1 execute dauligor-db --local \
  --file=migrations/20260531-1200_users_password_credentials.sql

  4. Start (or restart) the stack

    From the repo root. A full restart is required — .env and server changes aren't hot-reloaded:

    node scripts/dev-sysapp.mjs   # app → http://localhost:3001 , worker → :8788

The walkthrough (in the browser)

  1. First login = silent migration

    Open http://localhost:3001 and log in with an existing user's username + their current password. Behind the scenes: native login returns 401 (no hash yet) → Firebase fallback succeeds → /adopt writes the scrypt hash. You're in, exactly as before.

  2. Log out, then log in again — now it's native

    This second login uses the freshly-written hash. To prove it went native, open DevTools:

    • Network → POST /api/auth/login returns 200 with a token (and no Firebase request follows).
    • Application → Local Storage → localhost:3001 → key dauligor:authToken now holds a JWT.

  3. Change your password (native)

    Go to Settings → Security, set a new password, save. Then log out and log in with the new password — it works, and the old one no longer does. (This exercises the new POST /api/auth/change-password.)

  4. Click around

    Open the wiki, a character sheet, the compendium, the admin pages. Every authenticated call now pulls its token from getSessionToken() — if those load normally, the 44-file sweep is good.

Prefer the command line?

Once a user is adopted (or seeded with a hash), you can hit the endpoint directly:

# after the user is adopted — returns { token, profile }
curl -X POST http://localhost:3001/api/auth/login \
  -H "content-type: application/json" \
  -d '{"username":"YOURNAME","password":"YOURPASS"}'

# before adoption (no hash yet) this returns 401 by design —
# log in once via the UI to trigger the adopt, then retry.

§5 If something's off

Login says "Native auth is not configured" (503)

AUTH_JWT_SECRET isn't set, or the stack wasn't restarted after editing .env. Re-check step 2 + 4. Until fixed, login silently uses Firebase (the app still works).

A 500 mentioning password_hash

The migration (step 3) hasn't been applied to your local D1. Run it, then retry.

Logged out unexpectedly / 401 on everything

Clear dauligor:authToken from Local Storage and log in again. A token older than 30 days is auto-dropped — re-login issues a fresh one.

Worker 500s (ECONNREFUSED)

The worker (:8788) died but the app (:3001) stayed up. Restart dev-sysapp.mjs; both must be live.

Note
This is all local. Nothing is deployed and nothing is pushed. Production go-live additionally needs the remote D1 migration applied and AUTH_JWT_SECRET set as a Cloudflare secret — both gated on your explicit go-ahead.
