# DRY audit — proposal-editor sweep (2026-05-21)

Scope: 9 editors touched by commit `6945895` (`proposal-mode editor sweep`).
~1.5k net lines, repeated patterns across both single-work (Class/Subclass/
UniqueOptionGroup) and catalog (Spells/Feats/Items/Tags/SpellRules/SpellListManager)
editors. Total: 19,057 lines across in-scope files. Goal of this audit:
identify highest-ROI extractions before the patterns get applied to 3 more
sites (Subclass/UniqueOptionGroup tombstone wiring is the immediate trigger).

---

## Top picks (high ROI / low risk)

### 1. `useProposalEntityDrafts(entityType)` — collapse the `getDraftedEntities` callsite boilerplate
**What:** 7 callsites repeat the same 4-line `useBlock()` + `useMemo(getDraftedEntities(...))` dance:
- `ClassEditor.tsx:429-433` (`class`)
- `SubclassEditor.tsx:153-157` (`subclass`)
- `UniqueOptionGroupEditor.tsx:58-62` (`unique_option_group`)
- `FeatsEditor.tsx:445-448` (`feat`)
- `SpellsEditor.tsx:486-489` (`spell`)
- `TagsExplorer.tsx:216-223` (`tag_group` + `tag`, TWO useMemos)
- `DevelopmentCompendiumManager.tsx:272-278` (parameterized — already half-extracted)

**Extraction:** `hooks/useProposalEntityDrafts.ts`
```ts
export function useProposalEntityDrafts(entityType: ProposalEntityType | null) {
  const ctx = useProposalContextOptional();
  const { drafts, activeBundleId } = useBlock();
  return useMemo(
    () => entityType
      ? getDraftedEntities(entityType, ctx, drafts, activeBundleId)
      : { byId: new Map(), createdIds: new Set(), deletedIds: new Set() },
    [entityType, ctx, drafts, activeBundleId],
  );
}
```
**Savings:** 4 lines × 8 callsites = ~30 lines, but real win is API clarity at every callsite.
**Risk:** Near-zero. The DCM null-guard pattern (line 272-278) already proves the shape works
with optional `entityType`. TagsExplorer needs 2 calls (intentional — two entity types in one
editor), which composes cleanly.

### 2. `useDraftedEntityIds(entityType)` — collapse the manual id-set rollup
**What:** SpellsEditor / FeatsEditor have nearly-identical 20-line `useMemo` blocks that walk
the queue + drafts to build a `Set<string>` of staged ids:
- `SpellsEditor.tsx:461-480` (`draftedSpellIds`)

**Extraction:** trivial selector on top of #1: `useDraftedEntityIds(t)` returning the union

**Savings:** 20 lines × 3 callsites ≈ 60 lines.
**Risk:** Low. The semantics are identical across all three sites (verified by reading the
identical loops).

### 3. `useProposalPreFlushSave(opts)` — collapse the pre-flush registration ceremony
**What:** All 5 wired editors duplicate the same 15-line pattern:
- `ClassEditor.tsx:1298-1319`
- `SubclassEditor.tsx:569-589`
- `UniqueOptionGroupEditor.tsx:326-346`
- `SpellsEditor.tsx:1296-1317`
- `FeatsEditor.tsx:820-840`
- `DevelopmentCompendiumManager.tsx:481-499`

Shape:
```ts
const handleSaveRef = useRef(handleSave);
useEffect(() => { handleSaveRef.current = handleSave; });
useEffect(() => {
  if (!isProposalMode || !proposalContext) return;
  if (!id && !pendingCreateId) return;        // single-work variant only
  return proposalContext.registerPreFlush(async () => {
    // catalog editors also dirty-check: editingIdRef + formDataRef vs lastLoadedFormRef
    try { await handleSaveRef.current(undefined, { silent: true }); }
    catch { /* swallow */ }
  });
}, [isProposalMode, proposalContext, id, pendingCreateId]);
```

**Extraction:** two-variant hook in `hooks/useProposalPreFlush.ts`:
```ts
// Single-work editors:
useProposalPreFlush({ enabled: isProposalMode, gate: !!effectiveId, handler: handleSave });

// Catalog editors (with dirty-check):
useProposalPreFlushOnEdit({ enabled: isProposalMode, editingIdRef,
  formDataRef, lastLoadedFormRef, handler: handleSave });
```

**Savings:** ~15 lines × 6 callsites ≈ 90 lines. Bigger payoff: locks the "registration order
+ silent-save + swallow + ref-mirror" contract in one place. Today there's already drift
(SubclassEditor swallows, SpellsEditor logs `console.error`, FeatsEditor logs `console.error`).
**Risk:** Low for single-work variant. Catalog variant needs to expose the dirty-check escape
hatch, but the three implementations are byte-identical (lines 1308-1315, 832-839, 490-497).

### 4. `useProposalSingleWorkId(routeId)` — collapse `pendingCreateId` / `effectiveId`
**What:** Three single-work editors share an identical 2-line state + derived id:
- `ClassEditor.tsx:443-444`
- `SubclassEditor.tsx:160-161`
- `UniqueOptionGroupEditor.tsx:63-64`

```ts
const [pendingCreateId, setPendingCreateId] = useState<string | null>(null);
const effectiveId = id ?? pendingCreateId;
```
Plus the `handleSave` create-branch each calls `setPendingCreateId(saveId)` after queueing.
Plus the pre-flush dep list (`[isProposalMode, proposalContext, id, pendingCreateId]`).

**Extraction:** `hooks/useProposalSingleWorkId.ts`:
```ts
export function useProposalSingleWorkId(routeId: string | undefined) {
  const [pendingCreateId, setPendingCreateId] = useState<string | null>(null);
  // Reset when route id changes (remount).
  useEffect(() => { setPendingCreateId(null); }, [routeId]);
  return {
    effectiveId: routeId ?? pendingCreateId,
    pendingCreateId,
    recordCreate: setPendingCreateId,
  };
}
```

**Savings:** Modest (~8 lines × 3 = 24), but the real value is unifying the gating logic.
ClassEditor sometimes uses `effectiveId`, sometimes `id || pendingCreateId` (e.g.
header `effectiveId` line 1466 vs save check). Centralizing this prevents future drift.
**Risk:** Very low. The state is dead-simple and the three sites are byte-identical.

## Worth-it but medium-risk

### 6. `useEditBaseUnlocks(proposalContext)` — Edit Base hook for catalog editors
**What:** SpellsEditor / FeatsEditor / DCM share the same 15-line unlock state + handler:
- `SpellsEditor.tsx:435-454`
- `FeatsEditor.tsx:452-468`
- `DevelopmentCompendiumManager.tsx:165-173` (no focusMode flip yet — DCM diverged)

The three differ on **one line**: SpellsEditor + FeatsEditor call
`proposalContext?.setFocusMode('drafts')` after unlocking; DCM does not (per inspection of
line 165-173 it never wires the focus-mode flip). The `isReadOnly` derived state
(`focusModeEnabled && !!editingId && !unlockedBaseIds.has(editingId) && !draftedXIds.has(editingId)`)
is also duplicated three times verbatim.

**Extraction:** `hooks/useEditBaseUnlocks.ts`:
```ts
export function useEditBaseUnlocks(opts: {
  focusModeEnabled: boolean;
  editingId: string | null;
  draftedIds: Set<string>;
  proposalContext: ProposalAccumulatorContextValue | null;
}) {
  const [unlockedBaseIds, setUnlockedBaseIds] = useState<Set<string>>(new Set());
  const unlock = useCallback((id: string) => {
    setUnlockedBaseIds((prev) => prev.has(id) ? prev : new Set(prev).add(id));
    opts.proposalContext?.setFocusMode('drafts');
  }, [opts.proposalContext]);
  const isReadOnly = opts.focusModeEnabled && !!opts.editingId &&
    !unlockedBaseIds.has(opts.editingId) && !opts.draftedIds.has(opts.editingId);
  return { unlockedBaseIds, unlock, isReadOnly };
}
```

**Savings:** ~20 lines × 3 callsites ≈ 60 lines.
**Risk:** Medium. The focus-mode flip is a *behavior* DCM doesn't currently exhibit —
unifying could be either a bug-fix (DCM should flip) or a regression (DCM intentionally
doesn't). Read commit `6945895`'s message: it says "Mirrors SpellsEditor's pattern" for
DCM — suggesting DCM is just behind. Worth confirming with user; if intentional,
expose a `flipFocusOnUnlock?: boolean` option.

### 7. `actionLabel` save toast — micro-helper for the create/update branch
**What:** Five editors share this same pattern in `handleSave`:
```ts
if (isProposalMode) {
  const isCreate = !effectiveId;     // or !entryIdAtStart for catalog editors
  if (isCreate) { await writer.create({...payload, id: saveId}); setPendingCreateId(saveId); }
  else { await writer.update(saveId, payload); }
  if (!opts.silent) toast.success(actionLabel(writer.mode, isCreate ? 'created' : 'updated'));
}
```
- `ClassEditor.tsx:1239-1248`
- `SubclassEditor.tsx:530-539`
- `UniqueOptionGroupEditor.tsx:290-300`
- `SpellsEditor.tsx:1213-1220`
- `FeatsEditor.tsx:769-776`
- `DevelopmentCompendiumManager.tsx:424-431`

**Extraction:** `lib/proposalAccumulator.ts` (next to `useProposalAccumulator`):
```ts
export async function applyProposalWrite(
  writer: WriterApi, payload: Record<string, any>, opts: { id: string; isCreate: boolean; silent?: boolean },
): Promise<void> {
  if (opts.isCreate) await writer.create({ ...payload, id: opts.id });
  else await writer.update(opts.id, payload);
  if (!opts.silent) toast.success(actionLabel(writer.mode, opts.isCreate ? 'created' : 'updated'));
}
```

**Savings:** Per callsite small (~6 lines × 6 ≈ 35 lines), but the **dedup-baseline-sync**
boilerplate (`lastLoadedFormRef.current = JSON.stringify(...)`) also belongs in this helper for
catalog editors — same 1-liner duplicated 3 times (Spells:1228, Feats:784, DCM:436), each with
identical justifying comments. Including it: ~50 lines.
**Risk:** Medium. The single-work editors call `setPendingCreateId(saveId)` between
create-success and toast; catalog editors call `setEditingId(entryId)` and sync the
dirty baseline. The helper has to either accept post-create callbacks or stay narrow
(just queue+toast). Narrow extraction is safer.

## Don't extract

### 8. `displayEntries` merge `useMemo` — shape diverges per editor
**What:** Four editors (Spells/Feats/DCM/TagsExplorer) merge `draftedEntities` into their list.
The SHAPE looks identical (overlay updates, tag deletes, append creates) but the
**row-normalize step diverges meaningfully**:
- `SpellsEditor.tsx:743-762` uses `mapSpellRow({ ...e, ...overlay })` which RECOMPUTES facet
  buckets (`activationBucket`, `rangeBucket`, `durationBucket`, `shapeBucket`, `hasV/S/M`)
  used by the filter — see line 779-796 referencing these.
- `FeatsEditor.tsx:486-509` uses `denormalizeCompendiumData(overlay)` — flat snake→camel.
- `DevelopmentCompendiumManager.tsx:286-305` also uses `denormalizeCompendiumData(overlay)`.
- `TagsExplorer.tsx:258-281` (groups) pins `id` last to defeat partial UPDATE clobber, AND
  `283-321` (tags) has bespoke `normalizeTagRow` carrying the live tag's id through.

**Verdict:** The "shape" similarity is shallow. Each editor's `mapXRow` does meaningfully
different work that the catalog renderer depends on, and TagsExplorer's `id`-pin defense
against partial UPDATEs is editor-specific. A hook-factory parameterized by `mapRow` would
abstract the trivial overlay loop while hiding the bits that actually need to differ. The
~15-20 lines of merge logic per site is the wrong target — savings are marginal and
indirection cost is real (debugging filter-bucket mismatches across a generic helper is
much worse than reading 15 lines of explicit overlay).

If you want to extract *something*, extract the **deletedIds tag** + **append-creates** halves
(those are byte-identical) and leave the per-row normalize at the callsite. ~5 lines × 4 = ~20.
