Block Review — redesign mockups

Three approaches for displaying a submitted block and reviewing its changes. Static mockups — pick a direction (or mix), then I build it.

Why this needs a redo. Today a block is a flat list of revision rows (op badge + a one-line summary + status). The only way to see what actually changed is a "review" link that opens the entity's editor — and that link only works for class/subclass; features, scaling columns, and option groups/items dump you on the catalog list (e.g. "modular option groups") with no focused diff, and some types have no link at all. So in practice there's no real way to review most of a block before approving it.
  • No before→after diff — you can't tell what an UPDATE changed.
  • No sense of the block as a cluster (a class + its features/columns/subclasses that reference each other).
  • Review navigation is broken for everything but class/subclass.
The common thread in all three mockups: review happens inside the block view — no navigating out to editors.

Sample data throughout: a content-creator's block proposing a Druid class with a Wild Shape feature, a CR scaling column, an Animal Forms option group (with a Wolf option), and a Circle of the Land subclass — all referencing each other.

Mockup A — Review feed (inline diff cards)

The block opens straight to its changes. Each change is a card with an inline diff/preview; expand for full detail. Block-level actions pinned at top. Closest to today's structure, lowest build cost.

Mockup B — Cluster outline (tree)

Shows the block as the dependency hierarchy it actually is — the class at the root with its features/columns/subclasses/options nested under it. Best for understanding "what am I approving" as one coherent thing.

Per-change "looks good / flag" lets a reviewer walk the whole block, then the top-level Approve commits the lot atomically (Part D). Build cost: medium-high.

My lean: A (review feed) as the default — it directly kills the "no way to see what changed / broken review link" problem at the lowest cost — with B (cluster outline) as a toggle for class packs, since it mirrors how the block was authored and visually echoes the guard-#1 integrity check. C is the most work and shines mainly when a block is large.

All three replace the broken per-entity "review" navigation (#4) with in-place diffs. Tell me which direction (or combination), and I'll build it into AdminProposals.
