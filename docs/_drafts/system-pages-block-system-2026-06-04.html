Design Investigation · Dauligor · 2026-06-04

System Pages → the Block System

Now that lore articles are authored as block layouts (the generic engine in src/lib/layoutBlocks.ts + LayoutEditor/LayoutBlocks, shipped to main), can system pages use the same blocks? Short answer: yes for the page body, with one bespoke block type for entity-backed entries — but system pages carry machinery lore articles don't (&-references, #-anchors, live entity-backed entries), so the move is staged, not a copy-paste.

✅
Decisions since this was written (2026-06-05).
  • Stage 2 (page body → blocks) shipped to local (commit f7fdec1, not yet pushed): system_page_blocks table + SystemPageDesigner reusing LayoutEditor; the entries glossary stays as-is for now.
  • Full unification chosen for entries (Stage 3, in progress). Entries become blocks: an definition block (free-prose) is the addressable ENTRY — it IS the &kind[anchor] / #anchor target. A reference block is the OPPOSITE — a pointer that embeds an entity defined elsewhere — so it is not an entry and is dropped from system pages. (This corrects the earlier "reference block = entry" framing, which made a reference both the pointer and the target.)
  • Approach: lazy + fallback (like notes/secrets) — resolver reads blocks, falls back to the system_page_entries table; pages convert on edit; the entries table is kept as a safety net (not dropped); no bulk SQL data migration. Keeps imported Foundry &-refs working through the transition.
Sections below are the original investigation; where they say the entry-placement block is a "reference" block, read "definition block" (entry) per the decisions above.

1 · What a system page is today

Source of truth: src/lib/systemPages.ts, tables system_pages / system_page_entries, reader /system/:identifier (SystemPageView), editor SystemPageEditor (master-detail).

Entries (system_page_entries)An ordered glossary. Each entry has an identifier (the #anchor + &kind[id] target), name, summary, and is either free BBCode (body) or entity-backed (source_kind/source_id → a canonical row whose text is pulled live; a stored field still overrides the source).No direct analogue — closest is the entity-row/entity-feature blocks, which already resolve system kinds.

2 · The core distinction that shapes everything

⚠️
Entries are addressable, live, and reference-targeted. A lore article's body is a self-contained blob. A system page's entries are not: each is a stable anchor (/system/condition#prone), a &-reference target, and may be a live mirror of a canonical row (so editing the condition updates the glossary with zero duplication). Any block design must preserve entry identifier stability + entity-backing, or it breaks every &Reference[…] in imported Foundry content and every hover card.

3 · Two surfaces, two answers

A system page has two distinct authoring surfaces. They don't have to move to blocks together.

3a · The page description → blocks ✅ (clean, low-risk)

The top-of-page description is free BBCode prose — exactly what a lore article body is. It drops straight onto the existing generic engine: a system_page_blocks table mirroring lore_article_blocks, the same LayoutEditor/LayoutBlocks, the same replace-all API. The article block set already includes entity-row/entity-feature, which resolveReference can already resolve to system pages/entries — so a page intro could embed cards of its own entries or related content immediately. This is the obvious first step and reuses 100% of the shipped engine.

3b · The entries → blocks ⚠️ (needs a bespoke block + ref/anchor care)

Two ways to model entries as blocks:

ModelHowTrade-off
B1 — keep entries; add an entry embed blockEntries stay in system_page_entries (master-detail editor unchanged → anchors, &-refs, entity-backing all keep working). The page body becomes blocks, and a new entry block references an entry id to place it inline (like the campaign recommended block references an article). Reorder the glossary by reordering blocks; entries the author hasn't placed fall to an auto-list at the end.Lowest risk. Preserves the entire &-ref/anchor/entity-backing backbone. The "entry" remains the unit of identity; the block is just placement. Two stores (entries + blocks), but each owns a clear concern.
B2 — entries become entry blocks (full unification)Collapse system_page_entries into entry blocks in system_page_blocks: the block config holds identifier, name, summary, and either body or {sourceKind, sourceId}. One ordered block list IS the page. Cleanest mental model, but rewires the &-ref backbone: resolveSystemEntry, searchSystemEntries, getSystemPageKindMap, and the #anchor targets must all read entry identifiers out of block config instead of a table. Higher blast radius (touches imported-content references). Defer until B1 proves the flow.

4 · Recommended path

✅
Stage it: description-as-blocks first, entry-embed block second, full unification only if wanted. Each stage is independently shippable and behavior-preserving, mirroring how the lore designer rolled out (generalize → store/API/migrate → designer → viewer).
Phase S1 · Page body as blocks

Reuse the engine for the description

system_page_blocks table (mirror lore_article_blocks); blocks sub-route on the system-pages API + packet inclusion; one-time idempotent migration wrapping each page's description into a text block (lossless, like the lore content→block seed). A SystemPageDesigner reusing LayoutEditor (embedded + controlled, the modes already built) for the body; entries stay in the existing master-detail panel for now. Reader renders LayoutBlocks for the body, then the entries glossary below. foundation · low risk
Phase S2 · entry embed block

Place entries inline

Add an entry block type (references a system_page_entries id, like recommended references an article). The reader resolves it via the existing resolveSystemEntry and renders the entry card at that position; unplaced entries auto-list at the end. Anchors + &-refs unchanged (still keyed on entry rows). The block set for system pages = article set − note/secret (no per-campaign mechanics here) + entry. UI · medium
Phase S3 · (optional) full unification

Entries become blocks

Only if S1–S2 prove the model and the dual-store bookkeeping chafes. Migrate system_page_entries → entry blocks and repoint &-ref/anchor resolution at block config. Highest blast radius (imported-content references) — gate behind a thorough ref audit. DB + ref-backbone change

5 · What's reusable as-is vs new

Reuse unchangedNew for system pages
The whole generic engine: layoutBlocks.ts model/parse/serialize, generic fetch/saveLayoutBlocks(url), LayoutEditor (incl. controlled/ embedded/renderInspectorExtras), LayoutBlocks renderer, BBCodeRenderer. system_page_blocks table + migration; GET/PUT /api/system-pages/:id/blocks (mirror the lore handlers); a thin systemPageBlocks.ts adapter; a SystemPageDesigner host; an entry block type + its inspector-extras (entry picker / source binding) supplied via renderInspectorExtras (the hook secrets already use).

viewContext: system pages are site-wide — no era/campaign conditionals — so the body's viewContext is just { isStaff } (drives note visibility if that block is ever allowed here; by default it isn't). No per-campaign secret filtering needed.

6 · Open questions for the owner

  • Scope of the ask: just the page intro as blocks (S1), or inline entry placement too (S2)?
  • Entry identity: keep entries as their own rows (recommended — protects &-refs), or pursue full unification (S3) down the line?
  • Do system pages want note/secret blocks? Likely no (site-wide, not campaign-scoped) — default: exclude both from the system-page allowedTypes.
  • Entity-backed blocks: should an entry block that points at a canonical row live-mirror it (zero duplication, current behavior) — yes, reuse resolveEntries's merge.
