Module Rebake SystemAudit & Proposal · 2026-06-12

Question: "a system was meant to auto-bake when enough time passed since the last edit — did it ever get built?" Answer below, against the code.

Your memory is right. The debounce queue exists and works — but only opportunistically, the cron driver was never built, and (the part that actually bites us) a code deploy enqueues nothing, so logic fixes always need a manual rebake. There is also no "rebake a whole system" trigger.

2 · The gaps (why it feels like it "never got in place")

Gap 1 — no cron drains the queue The worker does have a cron (crons = ["30 3 * * *"]), but its scheduled() handler only prunes resolved proposals and prewarms the spell cache — it never calls popDueEntries. The queue's own comment calls the sweep "a future Cloudflare Cron." So due entries only bake when a Foundry client happens to hit the module API (1 per request). No traffic → the backlog just sits.
worker/index.js:9 · worker/wrangler.toml:8–13, 31–35
Gap 2 — a code deploy enqueues nothing (the real pain) queueRebake only fires on an entity save. When we change export logic — the cleanText BBCode fix, the feature-automation fix, the items catalog — the output changes but no entity is touched, so nothing is queued and the opportunistic/cron paths never re-bake. The per-class bundle is a stored R2 cache, so it keeps serving the old shape. That is why every cross-branch fix ends with "rebake the affected classes."
Gap 3 — no bulk / "rebake a whole kind" Every rebake is per-entity (rebakeBundle(kind, id) / one Bake-Now click). There is no rebakeAll('class') and no flag to re-bake a system. After a logic change affecting all classes, today you'd click Bake-Now on each — exactly the manual toil you're describing.

3 · Proposal — two small pieces that close all three gaps

A · Wire the cron to drain the queue ~20 lines

Add a queue sweep to the worker's existing scheduled() (or a more frequent cron): call popDueEntries(budget≈25) and rebakeBundle each. The 1-hour debounce then actually fires on a timer, independent of Foundry traffic. Closes Gap 1. Low risk — reuses the existing, idempotent rebake path.

Decision: reuse the daily 03:30 cron (edits bake overnight) or add a frequent cron like */15 * * * * (drains within ~15 min of becoming due). I'd lean frequent — it's cheap and makes the system feel "live."

B · A "rebake this system" trigger — your flag new

An admin trigger that operates on a whole kind at once. Two flavors (could ship both):

  • Bulk-enqueue (gentle): insert every entity of kind X into the queue with last_edit_at = now − DEBOUNCE so they're immediately due, then let the cron (A) drain them. Cheap write; bakes spread over the next sweep.
  • Bulk-rebake-now (immediate): loop rebakeBundle over all of kind X right away. Instant, but heavier per call.

Surface it as a small admin control (an endpoint POST /api/module/rebake-kind + a button next to Bake-Now), or — matching your "turn on a flag after a change" framing — a tiny config flag the deploy/admin flips to enqueue the affected kinds. Closes Gaps 2 & 3.

Net result — your original vision Make a code change → flip the flag for the affected kind(s) → every entity of that kind is enqueued → the cron sweep (A) bakes them within a sweep. No per-entity clicking, no "remember to rebake." The opportunistic path stays as a fast-track for live edits.

4 · Decisions needed before building

  1. Cron cadence: reuse daily 03:30, or add a frequent (*/15) sweep? (I recommend frequent.)
  2. Flag flavor: bulk-enqueue (drains via cron), bulk-rebake-now (immediate), or both?
  3. Trigger surface: admin button + endpoint, a config flag, or both?
  4. Scope: these are app-side (functions/, worker/, api/_lib/) — all in this branch's territory. No DB migration (the queue table already exists). The worker cron change ships with the worker deploy.
