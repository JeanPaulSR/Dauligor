Backgrounds, Races & Creatures in Foundry

How dnd5e (v13 / system 5.x) models these three object types, what our codebase already has, and the plan: clone the spell/item/feat export buttons for them so we can study their real Foundry shapes before building editors. Branch compendium-editors, 2026-05-30. Schema facts verified against the live dnd5e master source.

The one-sentence version. Backgrounds and races are Items (same family as feats/spells — they live in an actor's items array and carry advancement[]), but creatures are Actors — a fundamentally different document with abilities, attributes, CR, and their own embedded items. So bg/race export can clone the feat builder almost verbatim; creature export is a new shape that wraps an Actor, not an Item.

1 · The core split: Item vs Actor

Foundry has two top-level document classes that matter here. dnd5e subtypes each:

Our typeFoundry documentdnd5e subtypeData model (source)
BackgroundItemItem type:"background"module/data/item/background.mjs → BackgroundData
Race / SpeciesItemItem type:"race"module/data/item/race.mjs → RaceData
Creature / MonsterActorActor type:"npc"module/data/actor/npc.mjs → NPCData

This is the architectural fact that shapes the work. Our existing exporters (_spellExport, _itemExport, _featExport) all build Item bundles — a { name, type, img, system, effects, flags } object that drops into an actor's items. Backgrounds and races fit that mold exactly. A creature does not: it's an actor with its own embedded items.

2 · Background — BackgroundData (Item)

Composed from three mixins + base ItemDataModel. singleton: true (an actor holds at most one background).

system.* fieldSourceNotes
description.{value,chat}ItemDescriptionTemplateHTML blurb (+ chat card text)
source.{book,page,license,revision,rules}ItemDescriptionTemplatepublication metadata (our source_id maps to this)
identifierbase itemslug
advancement[]AdvancementTemplatethe grants — skill/tool/language proficiencies, an ItemGrant for a background feature, ASI/feat (2024 rules), etc.
startingEquipment[]StartingEquipmentTemplatearray of EquipmentEntryData (see below)
wealthStartingEquipmentTemplateFormulaField — starting gold alternative

EquipmentEntryData entries: {_id, group, sort, type, count, key, requiresProficiency} where type ∈ { "AND","OR" (grouping), "armor","tool","weapon","focus" (category choice), "currency", "linked" (specific item by UUID) }. This is the "choose a martial weapon OR (a shield AND a mace)" tree.

Takeaway: a background is essentially a feat with advancements + a starting-equipment tree. The feat exporter already emits system.advancement; background export = feat export with type:"background" + the equipment/wealth fields.

3 · Race / Species — RaceData (Item)

Same mixins as background (Advancement + ItemDescription) minus starting-equipment, plus three creature-shaped fields:

system.* fieldTypeNotes
description.* / source.* / identifier(shared)as background
advancement[]AdvancementTemplateASI, the racial traits (ItemGrant of feature items), Size, etc.
movementMovementField{walk,fly,swim,climb,burrow,hover,units} (bonus/special disabled for races)
sensesSensesField{darkvision,blindsight,tremorsense,truesight,units,special}
typeCreatureTypeField{value:"humanoid", subtype, swarm:false, custom} — the creature type the species confers

Takeaway: race export = feat export with type:"race" + movement/senses/type. The advancement machinery is identical.

4 · Creature / Monster — NPCData (Actor) — the different one

Extends CreatureTemplate (shared with the PC actor) which mixes in AttributesFields / DetailsFields / TraitsFields. This is a whole stat block, not an item:

system.* fieldHolds
abilities.{str,dex,con,int,wis,cha}.{value,proficient,bonuses,…}the six ability scores
attributes.hp.{value,max,formula,temp,…}hit points (+ hd.spent hit dice)
attributes.ac.{flat,calc,formula}armor class
attributes.movement / attributes.sensessame field types as race, on the actor
attributes.{init, spell.level, death, …}initiative, spellcasting level, death saves
details.typeCreatureTypeField — creature type (dragon, fiend, …)
details.crNumberField — Challenge Rating
details.{alignment, biography.value, habitat, treasure, source}flavor + publication
traits.{di,dr,dv,ci}damage immunities / resistances / vulnerabilities, condition immunities
traits.{languages, size, …}languages, size
resources.{legact, legres, lair}legendary actions / resistances / lair actions

Features and actions are NOT system fields. A monster's attacks, traits, and spells are embedded Item documents in the actor's items[] array (each a weapon/feat/spell Item with its own Activities). So a creature export bundle has to carry an actor envelope plus an array of item documents — structurally a superset of everything the item exporters already do.

Design consequence. Backgrounds + races slot into the existing Item-bundle pipeline with a near-verbatim clone of _featExport. Creatures need a new actor-bundle shape (kind:"dauligor.creature-actor.v1") whose system is the NPC schema and whose nested items[] reuse the item exporters. Plan the buttons for all three now, but expect creature export to reveal the most about a shape we don't model yet — which is exactly why we're adding the button first.

7 · The plan (this is the order)

  1. Clone the export buttons for backgrounds, races, creatures (this step). Wire the 5 seams in §6. For bg/race this is a feat-export clone; for creature, a first-pass actor bundle. Purpose: produce real Foundry JSON for each so we can see exactly how the dnd5e module ingests them — empirical, not inferred.
  2. Study the round-trip. Export → drop into a Foundry world → observe how backgrounds/races apply their advancements and how an NPC stat block + embedded items render. Capture the real shapes the system expects.
  3. Then build the editors — the same treatment we gave feats and spells (tagging-aware editor, proposal-mode, draft overlays). bg/race graduate out of the feats-table placeholder into their own tables (per the "new table for new functionality" preference); creature gets a genuine actor-shaped home.

Caveat captured for step 3: per project memory, prefer a new D1 table over extending feats with more discriminators — the feat_type='background'|'race'|'monster' placeholder is explicitly temporary. The export-button step does not require the migration; it reads whatever the editors store today.
