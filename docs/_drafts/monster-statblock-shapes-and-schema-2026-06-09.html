Monster stat blocks — shape study & proposed monsters schema

Understanding the two data shapes the Monster Browser must bridge — the 5etools rendered stat block (the visual target) and the Foundry creature export (the structured source) — and the monsters table they imply. Branch monster-browser, 2026-06-09. Grounded in the full 1,001-creature Foundry export (E:\DnD\Professional\Foundry Export\creatures, dnd5e 5.3.1 / Foundry v13) and two canonical 5etools stat blocks (Aarakocra, Adult Black Dragon). Schema is adversarially reviewed — three structural blockers were found and folded in below.

Bottom line. Foundry's npc actor is a faithful superset of a 5e stat block — every header field and every action/trait/legendary/lair entry is recoverable. The reconstruction has exactly one hard problem (Armor Class) and three shape decisions the naive model gets wrong (spellcasting and actions must be arrays of sub-blocks, not single objects). The single highest-leverage fix is module-side: have the creature exporter emit Foundry's derived values (resolved AC, proficiency bonus, save/skill totals, passive perception) instead of the raw source document — that erases most of the app-side precompute risk at a stroke. No migration is proposed yet — this is the survey that precedes it.

1 · The two shapes

Target — 5etools rendered stat block

What the detail panel must reproduce.

A linear sequence of labelled rows + named entries: Name / Source → Size Type (subtype), Alignment → AC → HP → Speed → ability grid → Saving Throws → Skills → Damage Vuln/Res/Imm → Condition Immunities → Senses → Languages → Challenge / Proficiency Bonus → Traits (unnamed) → Actions → Bonus Actions → Reactions → Legendary Actions → Lair Actions → Regional Effects → variant/inset boxes → Environment → source footer.

It is a render: the "+11 to hit", "17 (2d10 + 6)", "(Recharge 5–6)", "(3/Day)", "Perception +11" are all synthesized from underlying numbers + prose.

Source — Foundry creature-folder-export.v1

109 MB · 1,001 npcs · the structured truth.

Envelope {kind, schemaVersion, exportedAt, game, folder, summary, creatures[]}. Each creature = {id, uuid, name, type:"npc", folderId, folderPath, relativeFolderPath, creatureSummary, sourceDocument}.

creatureSummary is a slim preview projection; sourceDocument is the full Foundry npc actor — system (abilities, attributes, details, traits, skills, resources, spells, source) + embedded items[] (the traits/actions/spells) + effects[] + prototypeToken.

⚠ Read sourceDocument.system.*, never creatureSummary. The summary is lossy for everything load-bearing: source.{book,page} arrive empty, proficiencyBonus is always 0, spellcasting.level is always 0, the damage-trait bypasses/custom and language telepathy/custom are absent, and biography is truncated to ~200 chars. Use the summary only to cross-check. (This is itself a symptom of the AC problem in §6 — the export serialises the raw source document, where these are runtime-derived.)

3 · How the action economy reconstructs

The stat-block body (Traits / Actions / Bonus / Reactions / Legendary / Lair / Regional / Spellcasting) is rebuilt from sourceDocument.items[]. Two facts drive it:

  • Weapons are always Actions (attacks). feat items (always system.type.value:"monster") carry everything else. spell items are the spell list. equipment = worn gear (informs AC). consumable/loot = magic-item actions (rare — §6.4).
  • The section a feature lands in is tagged by flags.plutonium.page — monsterTrait · monsterAction · monsterBonus · monsterReaction · monsterLegendary · monsterLairActions · monsterRegionalEffects · monsterSpellcasting. This is present on 97.5% of feats and is cleaner than bucketing by activation.type (which scatters a monsterTrait that happens to carry an action activity).
Per-activity refinement. 174 items span buckets (e.g. a lair feat that also deals action damage; Dancing Lights = action + bonus). Bucket the page, then refine per activity by activation.type: special (64) → Trait (e.g. Legendary Resistance), hour (48) → a long-cast note. The only plutonium.page:null feats are the 78 "Legendary Actions" wrapper headers (§6.4).

Each rendered entry needs both the prose body (the item's system.description.value, HTML → BBCode via our htmlToBbcode) and a structured tuple (to-hit, reach/range, damage parts, save DC/ability, recharge/uses) so the synthesized suffixes — (3/Day), (Recharge 5–6), (Costs 2 Actions), +11 to hit — render exactly.

5 · The section JSON shapes

Each action/trait stores prose and a structured copy. Note the two array-shape corrections from §6 are already baked in (an Action holds activities[]; spellcasting is an array).

type Action = {
  name: string;                 // base name, NO usage suffix
  description: string;          // BBCode (item system.description.value → htmlToBbcode)
  pageBucket: string;           // plutonium.page (audit / re-bucket)
  order: number;                // item `sort`
  source_book?: string;         // for FTD "Additional X" sub-block grouping
  uses?: { max?: string; recovery?: Array<{period, formula?, type?}> };
                                // ITEM-level system.uses → "(Recharge 5–6)" | "(3/Day)" | "(Recharges at Dawn)"
  costs?: number;               // legendary: activation.value > 1 → "(Costs N Actions)"
  activities: Array<{          // ← ARRAY (332+ weapons have ≥2; 174 items span buckets)
    kind: 'attack'|'save'|'utility'|'heal'|'damage';
    activation: string;         // action|bonus|reaction|legendary|lair|special|''
    attack?: { bonus: number;   // PRECOMPUTED mod(attack.ability)+PB+attack.bonus
               type: 'melee'|'ranged'; classification?: string;
               reach?: number; range?: number; long?: number; units: 'ft' };
    save?:   { abilities: string[]; dc: number; onSave?: 'half'|'none' };
    damageParts?: Array<{ average: number; formula: string; types: string[] }>;
                                // average resolves @mod against THIS activity's attack.ability
  }>;
};

type Trait = { name; description; uses?; source_book?; order };  // passive; Legendary Resistance
                                // injects resources.legres.max as uses (item uses is empty)

type Spellcasting = {           // ← one PER monsterSpellcasting feat (38 creatures have 2)
  ability: string; level: number;          // attributes.spell.level (summary is 0)
  saveDc?: number; attackBonus?: number;   // compute 8+PB+mod / PB+mod; PROSE is authoritative fallback
  method: 'spell'|'innate'|'pact'|'atwill';// header label hint only
  slots?: Record<string,number>; pactSlots?: { count; level };
  prose: string;                           // the feat description (pre-rendered, exact text)
  spells: Array<{ name; level; method; uses? }>;  // resolved from EMBEDDED spell items (not the DB)
};

6 · Adversarial fixes must-read before any migration

An independent critic tried to break the naive schema against all 1,001 creatures. The model held ~80%, but found three structural blockers (column-shape changes that are painful to migrate later) and several majors. All are folded into §4–§5 above; here's the why.

6.1 · Armor Class is not reliably reconstructable for ~306 creatures BLOCKER

AC calc splits {natural:523, default:462, custom:16}. natural reads ac.flat cleanly. But of the 462 default, 306 have no armor item and flat:null — the formula collapses to 10 + dexMod, which is a guess: some monsters print a higher AC with no armor item, and there is no AC number anywhere in the static export for them (biographies contain zero "Armor Class" text). Armored cases are also unvalidated (the canonical Guard prints AC 16; the export's shield math yields 17).

Fix. Don't freeze a 10+dex guess and present it as authoritative. Best: have the exporter emit Foundry's resolved actor.system.attributes.ac.value (§7) — the only ground truth. Interim: store ac nullable with a best-effort compute (apply the 11 "Other AC Bonus" ActiveEffects; their value is a signed string "+4") and set ac_unverified=true on the 306 so QA/UI knows.

6.2 · spellcasting must be an ARRAY BLOCKER

38 creatures carry two monsterSpellcasting feats (innate + prepared, or a coven block + own block — Alhoon, Drow Priestess, every *Hag (Coven), Arcanaloth). A single spellcasting object silently drops one whole block. → Spellcasting[].

6.3 · Action must hold an ARRAY of activity tuples BLOCKER

332 weapons have 2 activities, 9 have 3, 1 has 4. Swarm of Bats "Bites" = full-HP 2d4 + bloodied-HP 1d4; Aboleth "Tentacle" = attack + a CON-save disease rider. A single attack/save field keeps only the first; the rest survive in prose but lose their structured suffix. → activities: ActivityTuple[] (this also subsumes the 174 multi-bucket items as a first-class case).

6.4 · Don't blind-drop the Legendary preamble; don't drop magic-item actions MAJOR

The 78 plutonium.page:null "Legendary Actions" wrapper feats are activity-less — but their description is the 2024 preamble prose ("Legendary Action Uses: 3. Immediately after another creature's turn…"), which you can't reconstruct from legact.max alone. Extract it into legendary_actions_preamble before dropping the shell. Separately, 3 creatures have consumable/loot item-actions (Necklace of Fireballs, Rope of Entanglement, Net) that the naive "weapon/equipment/feat-only" bucketing drops — route them to Actions when they carry an activity.

6.5 · Reference / recharge / damage resolution targets MAJOR

  • @UUID[.X] resolves to the EMBEDDED spell item (Actor.<id>.Item.<X>), not the global spells DB. 694/824 sampled refs are label-less and REQUIRE the embedded item to get a name. Resolve embedded → store {name,level,method} in spellcasting.spells[] so render never needs a live join.
  • Recharge/uses live on the ITEM system.uses, not the activity: {formula:"5", period:"recharge", type:"recoverAll"} + uses.max:"1" → "(Recharge 5–6)". Periods seen: recharge 207, day 1903, sr 65, dawn 2 — keep type, add dawn→"(Recharges at Dawn)" to the render map.
  • Damage @mod resolves against the activity's attack.ability, per-part (Bite = [{2d10, @mod}, {1d8 acid, ""}] → "17 (2d10 + 6) piercing plus 4 (1d8) acid"). Precompute per-part average + resolved formula; capture save.onSave (half/none) on save-with-damage activities (Acid Breath 12d8, half).

6.6 · Minor truths that change the plan

  • Mojibake is NOT repairable — the 3 bad names contain U+FFFD, the original bytes are already destroyed. Hand-patch a 2-entry rename map; fix the export to stop corrupting them.
  • Drop dead modeling: dm (damage modification), di.custom, ci.custom, senses.special, movement.special, habitat.custom are empty across all 1,001. Keep the optional keys; build no columns/UI. (Contrast: languages.custom 296, dr.bypasses 169, telepathy 141 — these stay.)
  • swarm is null (not "") on non-swarms; cr is int/float/null — ensure the fetcher doesn't coerce 0.0 to falsy and drop it.
  • Default the list thumbnail to token_image_url (local) to avoid 967 cross-origin cdn.5e.tools loads until the R2 mirror lands.

7 · The one module-side ask highest leverage

Most of the app-side precompute risk — AC (§6.1), proficiency bonus, save/skill totals, passive Perception, spell save DC — exists only because the export serialises the raw actor.toObject(), where these are runtime-derived (hence creatureSummary.ac.value=0, proficiencyBonus=0). Foundry already computes all of them on the prepared actor. If the creature exporter emitted the prepared/derived values — system.attributes.ac.value, the real pb, each abilities.<a>.save total, each skills.<s>.total, attributes.senses.passivePerception (or attributes.perception), and each spellcasting block's save/attack — the importer becomes a near-trivial copy and the AC blocker disappears.

This is a cross-branch change owned by the foundry-module branch (per the repo's "cross-branch = handoff, not self-modify" rule). It is the single most valuable follow-up. Recommend a handoff to foundry-module requesting an enriched creature-folder-export (or a creatureSummary that reads the prepared actor). Until then the app computes best-effort + flags ac_unverified.
