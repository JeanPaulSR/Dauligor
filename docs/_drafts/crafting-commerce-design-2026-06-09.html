Three asks — new item sets (Enchantment, Crafting Material), a Magic Items tab, and a Shop — are really one system seen from three angles. Crafting is the spine: enchantments + materials are the inputs, magic items are the outputs, and the shop is the economy that moves them. We design the data model first because everything else consumes it.

The core insight: the two "item sets" are not the same kind of thing

You grouped Enchantment and Crafting Material together as "item sets," but structurally they pull in opposite directions — and getting this right is the whole design:

Crafting Material — is an item Dragon scale, mithral, an essence vial. You carry it, stack it, price it, sell it, and consume it. It behaves exactly like loot. It belongs in the items table — the only question is how loud a home it gets (a loot subtype vs. its own item_type).
Enchantment — is not an item "Flame Tongue (+2d6 fire)" is a reusable effect applied to many bases — a shortsword or a greatsword. You never carry "a Flame Tongue" in a backpack. It's an effect definition + restrictions, so it wants its own enchantments table, not a row in items.

This matches your earlier direction exactly ("their own enchantment table… extract generic enchantments to an enchant activity we apply to items") and the repo's new-table-for-new-functionality rule. So: materials extend items; enchantments get a new table.

How Foundry models this (so we round-trip cleanly)

In dnd5e 5.x an Enchantment is an Active Effect of type: "enchantment" applied to an Item (not an actor), delivered by an Enchant activity whose restrictions gate which items are valid targets. The effect's changes modify the target (a name override, system.magicalBonus, extra damage), and riders bolt on extra activities / effects / items. Our SemanticActivity.enchant already speaks this dialect. So an enchantment row maps to: { restrictions, an enchantment Active Effect, riders } — all shapes our editors already render.

Proposed data model

0 · recipes — the spine (build first) decided Per the 2026-06-09 direction, recipes come first: a universal "inputs → output" rule that creates everything — items, magic items, consumables, materials, and enchantments. Output has three modes: item (make a thing), enchantment (craft an enchantment definition), and enchant-item (apply an enchantment to a base → a magic item; this replaces the one-off "bake/apply" tiers in §3 below — applying is now just a recipe). Inputs: consumed items/materials + gold + time + requirements. Full schema: docs/database/structure/recipes.md (migration 20260609-1400, built).

1 · enchantments table load-bearing

A reusable, browsable library of enchantments. Mirrors the feats/spells pattern (own table, own editor surface, reuses the shared ActivityEditor + Active Effect editor). Per-source identifier uniqueness like items.

-- worker/migrations/YYYYMMDD-HHMM_create_enchantments.sql  (local-first)
CREATE TABLE enchantments (
  id            TEXT PRIMARY KEY,
  name          TEXT NOT NULL,
  identifier    TEXT NOT NULL,          -- slug; UNIQUE(COALESCE(source_id,''),identifier)
  source_id     TEXT,                   -- FK sources.id, ON DELETE SET NULL
  page          TEXT,
  description   TEXT,                   -- BBCode
  image_url     TEXT,
  restrictions  TEXT,                   -- JSON {allowMagical,type,categories[],properties[]} — what it applies to
  effects       TEXT,                   -- JSON Active Effect[] type:"enchantment" (the changes: bonus, rename, damage)
  riders        TEXT,                   -- JSON {activity[],effect[],item[]} granted alongside
  magical_bonus INTEGER,                -- convenience flat +N (mirrors items.magical_bonus)
  rarity        TEXT,                   -- rarity it confers on the finished item
  attunement    TEXT,                   -- '' | 'required' | 'optional'
  price         TEXT,                   -- JSON {value,denomination} — economy delta for shop/crafting cost
  level         TEXT,                   -- JSON {min,max} optional level gate
  prerequisites TEXT,                   -- JSON — forward hook for crafting (proficiency, tool, spell known)
  tags          TEXT,                   -- JSON tag FK[]
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);

Register restrictions/effects/riders/price/level/prerequisites/tags in the d1.ts JSON auto-parse list (+ the server mirror). Editor = a new page mirroring FeatsEditor, with the ActivityEditor in enchant mode + the Active Effect editor doing the heavy lifting.

2 · Crafting materials — own table, backed by a loot item decided

Materials behave like items (carried, stacked, priced, sold, consumed) and carry crafting-domain meaning — so they get both: a new crafting_materials table (the crafting layer) where each material is backed by a loot-type items row as its carryable "base item sheet." The backing item is what saves to character_inventory like any other item — sidestepping "how does a non-item land on a character sheet" — while crafting_materials holds the crafting metadata (tier, yields, recipe links) on top.

On the Foundry side they surface as their own "crafting material" type, which maps to dnd5e's existing loot subtype material — the lcat-material loot category already seeded by migration 20260605-1200 (system.type.value = "material"). The module-side labeling is a later handoff.

Open implementation detail (settle when building materials, Phase A step 2): the FK direction (crafting_materials.itemId → items.id) and whether the materials editor auto-creates / pairs the backing loot row. The carryable unit on a sheet is always the items row.

3 · Applying an enchantment to a base — three tiers

"Apply Flame Tongue to a greatsword → a magic greatsword" can land at three depths. We start shallow and deepen as the crafting game grows:

TierWhat it doesWhere it writes
1 · Bake start hereEditor flow: pick base item + enchantment → mint a new catalog items row with the enchantment's AE baked into effects, name/bonus/rarity/price merged. Produces the pre-made magic items the compendium & shop sell.items (new row)
2 · Per-character craftA character spends materials → an enchanted instance in their bag, the applied enchantment riding in custom_data.character_inventory
3 · Recipe / blueprintA recipes table: material inputs + base + enchantment + time/cost/proficiency gates → output. The actual crafting "game."new recipes table

Tier 1 is mostly authoring + a merge function, reuses the item editor + Foundry export (items-with-effects already round-trip), and immediately stocks the Magic Items tab and Shop. Tiers 2–3 are the long game.

4 · Magic Items tab

Lowest-risk as a filtered view, not a new store: the existing browser/editor filtered to magical = 1 (or rarity != 'none'). It's a lens on items — no schema change. Editor tab + browser tab can share the predicate. The Bake flow (Tier 1) is what fills it.

5 · Shop tab

Multiple shops, not one global list (decided 2026-06-10). The shop supports many distinct shops — a global one and per-campaign shops (e.g. "Waterdeep Smithy", "Black Market") — so a DM can stock specific shops rather than being limited to a single price list. Implies a shops table (name, scope: global|campaign, campaignId?) + a shop_inventory join (shopId, itemId, priceOverride?, stock?), with items.price as the fallback.

Two layers, sequenced:

  • 5a — Price view (read-only): a browser over items.price grouped/sorted by cost & rarity. Buildable now; pure read.
  • 5b — Buy / sell (transactions): writes character_inventory and debits/credits a wallet — blocked on the currency gap.
Prerequisite — character currency does not exist yet There is no character wallet today. Only containers carry a {cp,sp,ep,gp,pp} grid; the character builder references currency only in copy. Before buy/sell, we add a character wallet — most naturally a currency JSON column on characters (matching Foundry's system.currency) so it round-trips. Resolved (2026-06-10): support BOTH via named shops (see above), not a single price list.

Suggested sequencing

Each phase is independently useful, so order can flex — but this keeps every step shippable:

A
Author the catalogs (the "first step") In order: (1) the recipes catalog (the universal inputs→output spine), (2) the enchantments library (hooks into recipes), (3) crafting materials (own table backed by a loot item). No live crafting execution yet — the rules/inputs exist and are browsable/editable.
B
Bake + Magic Items tab Tier-1 apply flow (base + enchantment → catalog magic item) + the magical filter view. First end-to-end payoff: authored enchantments produce real magic items.
C
Shop Read-only price view (5a) → add the character wallet → buy/sell (5b).
D
The crafting game (long-term) recipes + per-character crafting (Tiers 2–3): materials + time + proficiency gates that turn inputs into outputs.

Open decisions — settle before the first migration

  1. Materials home — ✓ resolved: own crafting_materials table, each backed by a loot-type items row (Foundry material loot subtype). The items row is the carryable sheet; the table is the crafting layer.
  2. First-step scope — ✓ resolved: Phase A only — author both catalogs, enchantments first. No Bake/apply flow this pass.
  3. Enchantment ↔ Foundry export (later): bake the AE directly onto produced items first (simplest, already round-trips); "emit a source feature carrying the enchant activity for live in-Foundry application" is a later option.
  4. Shop shape (later): one global price list vs. per-campaign inventories; wallet as a characters.currency JSON column (recommended) vs. its own table.
