2 What was built

Server

  • api/_lib/password.ts — scrypt hash + constant-time verify (pure-JS, runs in Node + Workers).
  • api/_lib/sessionToken.ts — issues/verifies HS256 session JWTs signed with AUTH_JWT_SECRET; 30-day sliding.
  • functions/api/auth/[[path]].ts — POST /login, /adopt (hash-on-next-login), /change-password.
  • firebase-admin.ts → verifyEitherToken() — every require* gate now accepts a native token OR a Firebase token during the migration window.
  • migration 20260531-1200 — adds password_hash + password_updated_at (local only; remote pending your go-ahead).

Client

  • src/lib/auth.ts (new) — the single source of truth: getSessionToken(), getIdentity(), isAuthenticated(), onAuthChange(), login(), logout(), redeemToken().
  • ~44 files swept — every auth.currentUser.getIdToken() → getSessionToken(), every auth.currentUser → getIdentity().
  • Entry points — App.tsx (Identity state + onAuthChange), Navbar (login/logout), RedeemTokenPage, Settings (native change-password).

3 How login works now

user types username + password
 1. → POST /api/auth/login  ·  adopted account → native JWT, done.
 2. not adopted → 401 → fall back to Firebase → on success, POST /api/auth/adopt writes the hash.
   → the next login is fully native. Accounts migrate themselves, no reset email.

every request → Authorization: Bearer <token> from getSessionToken() → gate verifies + reads role from D1.

4 Bugs found & fixed in testing

All three came from the same root: code that assumed the Firebase User shape or its auth lifecycle.

fixed Default-user avatar was blank

The avatar fell back to user.displayName/photoURL — Firebase-only fields the new Identity lacks. Now sources from the D1 profile. (2fa466e)

fixed Logout didn't fully sign out

Logout emitted an auth-change while Firebase was still signed in → the app re-loaded the profile → appearance/role/theme persisted. Now: clear native session silently → sign out Firebase → emit once. (7804c2c)

fixed Cache: keep it across logout/login

Confirmed the D1 cache is shared content only (no private data), so it correctly persists across logout/login — no needless re-fetch. (961e3b1)

5 Behavior audit — keep in mind

Checked & safe

safe No private data in the persistent cache

The sessionStorage cache holds only shared content (compendium/lore/campaigns); per-user data uses uncached endpoints, and lore_secrets/users are proxy-blocked from the cacheable path. Safe across users on a shared browser.

safe No other Firebase-User field reads; Settings is fine; roles are live from D1

The only auth-object Firebase-shape reads were the (now-fixed) Navbar ones. Demotion/deletion takes effect immediately because every request re-reads the role from D1.

Differences to watch

watch fixable now Multi-tab auth sync is gone

Firebase synced login/logout across tabs; the native token doesn't. Log out in one tab → others stay "logged in" until reload. A small storage-event listener in auth.ts restores it.

watch Fixed 30-day token, no auto-refresh

Firebase refreshed hourly; the native token is fixed 30-day. Long sessions end hard until the Phase 3b refresh endpoint exists.

minor Error reports omit identity for native users

reportClientError reads auth.currentUser (null for native). Harmless; could fall back to getIdentity().

Will need rework when Firebase is removed

phase 5 Sign-in links (Firebase custom token)

RedeemTokenPage sessions never adopt — they stay on Firebase. Need a native one-time token before cut-out.

phase 5 Username → Firebase-email sync

Changing a username syncs the Firebase login email server-side; native-only accounts won't have that to sync.

phase 4 Admin user create / delete / temp-password

Still call Firebase Identity Toolkit; Phase 4 moves them to D1 writes.

6 How to test locally

The stack is running now — app on :3001, worker on :8788, seeded with a copy of your production data + the password column.

  1. Instant native admin: log in as tester / test1234 (no Firebase needed).
  2. Your real accounts: log in with the real username + Firebase password → native 401 → Firebase fallback → /adopt writes the hash → next login is native. Real prod roles apply.
  3. Prove it's native: DevTools → Application → Local Storage shows dauligor:authToken; Network shows POST /api/auth/login → 200 with no Firebase call.
  4. Change password: Settings → Security, then re-login with the new one.
  5. Logout: theme/appearance revert to default, fully signed out; the content cache stays warm for the next login.

7 What's left

  • Decision points still open: add the multi-tab sync listener now? Build the Phase 3b refresh endpoint?
  • To go live (production): apply the password_hash migration to remote D1 (read-once your go-ahead) + set AUTH_JWT_SECRET as a Cloudflare secret. Until then native auth stays dormant in prod and Firebase carries on.
  • Phase 4–5: move admin CRUD to D1, then drop Firebase entirely (incl. sign-in links + username-email sync rework).
  • Branch: nothing pushed — when you're happy, fast-forward dauligor-applications onto these commits.
