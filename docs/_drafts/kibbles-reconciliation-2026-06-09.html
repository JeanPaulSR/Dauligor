Kibbles' Crafting → Our Schema Reconciliation & Phase-A decisions

2026-06-09 · source: Kibbles' Compendium of Craft & Creation v1.2.1, Chapter 6 (printed p.179+). Text extracted (no poppler) and analyzed by a 9-reader + reconcile + 2-critic workflow. The two critics deliberately disagreed on scope; this doc is the synthesis.

Good news first: our core model held up. Kibbles' "inputs → output" is exactly our recipes table; its reusable +1/+2/+3 enhancements are our enchant-item mode + enchantments; its named magic items are outputType:'item'. The refinements below are additive, not a rethink.

How Kibbles models crafting (the relevant shape)

  • The loop: accumulate progress via crafting rolls (a success = hours of progress) until you reach the item's crafting time. Every recipe table shares 7 columns: Name · Materials · Crafting Time · Checks · Difficulty (DC) · Rarity · Value.
  • Materials are a rich taxonomy: a category (reagent / essence / ingot / hide / part / wood / gem), a rarity tier (Trivial → Legendary), a property (reagents: curative/poisonous/reactive · essences: arcane/primal/divine/psionic), a "used-for" discipline, and a price. Lower materials refine into higher ones (reagents → essences; ore → ingots).
  • Disciplines (Alchemy, Blacksmithing, Enchanting, Poisoncraft, Cooking, Leatherworking, Tinkering, Runecarving, Engineering…) are the organizing axis — each fixes a tool + ability + special rules.
  • Enchanting = combine a base item + spell scrolls + essences → a magic item (rarity + value), plus generic +N for any base. Uses Intelligence.

Decisions — what we change for Phase A (authoring the catalogs)

The guiding line: add what is intrinsic to a recipe; don't denormalize the output's own fields, and don't build execution-time machinery.

✅ Keep — applying now

ChangeWhy
done recipes.craftChecks INTEGER + recipes.craftDifficultyDC INTEGERThe two canonical Kibbles columns; intrinsic to the recipe (not on any output item); stored, not computed (real checks≠time/2 exceptions). Applied to the migration.

Slim, authoring-focused: id, name, identifier, sourceId, page, description, imageUrl, itemId (the backing loot row), category, rarity, subtype (freeform flavor, discriminated by category), usedFor (discipline list), price, weight, tags, contentHash, timestamps.
next Add trivial rarity tierKibbles' bulk mundane materials are "Trivial" (below Common). Zero migration — rarity columns are bare TEXT with no CHECK; it's an app-side vocabulary/ordering edit.
editor Richer inputs shapeKibbles inputs include spell scrolls, typed/tiered essences, and OR-alternatives. inputs is already free JSON — no migration; the RecipesEditor grows the shape (kinds: item / spellScroll / material, with optional OR-groups) as real recipes need it.

↦ Defer — capture later, not now

DeferredWhy later
Denormalized recipes.rarity / recipes.outputValueThese live on the output item/enchantment (items.price, enchantments.rarity/price). Read them lazily like the items list already does. Add only if the recipe browser needs the filter.
Base-item constraints ("sword ≥500gp", "a +1 weapon") + recursive upgrade chainCraft-time validation = execution (Phase D). outputBaseItemId (concrete or NULL=any) is enough to author.

Materials: equivalentGoldValue, salvageable, sourceMetadataShop-era (Phase C) and harvesting/execution (Phase D). price already carries value.
Parametric "Custom Weapon" generator; in-place gear modification/repairAdvanced authoring / execution. For now, author each result as a plain item+recipe; property-adds can ride an enchant-item recipe.

The one open fork — disciplines Kibbles organizes all crafting by discipline (Alchemy / Blacksmithing / Enchanting…), and materials are "used for" disciplines. The completeness critic calls this the biggest missing concept; the scope critic calls a full discipline table premature execution machinery. The honest middle: we need the organizing axis now, but not the execution fields (ability / tool / forge facility) until Phase D. Three ways to seat it — your call drives whether recipes gets a discipline slug or a disciplineId FK, and whether crafting_materials.usedFor holds slugs or ids.
