Species Options design

A reusable, modular library of racial traits — authored once, attached to many species. 2026-06-03 · branch compendium-editors

Today every racial trait (Darkvision, Powerful Build, Fey Ancestry…) is re-created per species. Species Options make each trait a reusable building block: author "Darkvision 60" once, then attach it to every species that has it.

Confirmed decisions

Model — reusable trait → granted feature. Each option is a named, feat-shaped trait (description + optional advancements / activities / effects). Attaching it to a species grants it as a feature: an ItemGrant advancement on the species, with the full feature item embedded in the species export bundle's features[]. One uniform model covers Darkvision, Powerful Build, Fey Ancestry, etc. — and it reuses the proven background-features grant path.

Storage — the species_options type (now the single mechanism). A dedicated reusable-options table + manager. Update (2026-06-03): the earlier species_features table was consolidated into this — its grant-to-species path was never finished and the column set is identical, so species options are now the one place a species's granted features live (innate stat-block traits stay on the Traits tab). The old species-features manager was repointed here; data migration 20260603-1900 copies any rows over.

Data model

New table — species_options

Feat-shaped + camelCase (the same shape the retired species_features used). JSON columns are already in queryD1's auto-parse list.

species_options(
  id, name, identifier, sourceId→sources(id), page, description,
  advancements[], activities[], effects[], uses, tags[],   -- JSON
  imageUrl, contentHash, createdAt, updatedAt
)
-- source-scoped unique identifier (same pattern as feats/species)

Attaching options to a species

A new JSON column on species — a portable id array, consistent with the camelCase compendium tables (no junction table):

ALTER TABLE species ADD COLUMN speciesOptionIds TEXT DEFAULT '[]';
-- ["opt-darkvision-60", "opt-powerful-build", …]

Deleting an option just leaves a dangling id that the export skips — no cascade needed. (Added to d1.ts jsonFields so it round-trips parsed.)

Foundry export

When building the species (race) bundle, expand each attached option:

  • Fetch each species_options row in species.speciesOptionIds.
  • Build a feat-type item per option (system.type.value = "race") — a new _speciesOptionExport.ts, parallel to _backgroundFeatureExport.ts.
  • Emit one ItemGrant advancement per option on system.advancement (pool keyed by the option's flags.dauligor-pairing.sourceId).
  • Embed the full items in the bundle's new features[] so the module imports + grants them without a second fetch (exactly like backgrounds).

Also adds a standalone live endpoint /api/module/species-options/<id>.json. The Foundry-module side consuming these is its own follow-up (the export is self-describing — same shape it already handles for background features).

Notes / deferred

  • Overlap: Darkvision can also be set on the species' Senses field (Traits tab). With this model a "Darkvision" option is a granted feature instead — both can coexist; authors pick whichever fits. (A future "typed" option kind could set senses directly if desired.)
  • Authoring rich mechanics (advancements/effects) on an option reuses the existing AdvancementManager / effects editors — wired in phase 2+ as needed; v1 can ship name + description + image.
  • Module-side import of the species features[] / ItemGrants is the foundry-module branch's job (mirrors background features).
