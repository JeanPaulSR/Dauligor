Items Editor — Foundry 5.3.1 fidelity audit

Every per-type sub-form & dropdown in ItemsEditor.tsx, measured against the real dnd5e 5.3.1 — its templates/items/details/details-*.hbs sheets and the embedded CONFIG.DND5E lists. Both sides read directly (no example exports exist for items). 2026-06-07.

Bottom line: the editor was built Foundry-aligned and the bones are right — types, base-item FKs, weight/price shapes, armor/tool fields all match. But it predates or simplified several item-sheet features. The gaps cluster in three places: (1) a handful of option lists are short a few canonical entries, (2) the weapon and consumable sub-forms are missing whole fieldsets Foundry shows, and (3) properties aren't filtered per item type. ~11 concrete fixes, none structural.

3 · Consumable sub-form gaps

Foundry's details-consumable.hbs shows Type → Sub-Type → Properties → Magical, plus a Damage fieldset for ammo. Ours shows only a Magical Bonus.

  • Sub-type (the 2-axis gap) — Foundry's type.subtype (poison: contact/inhaled/injury/ingested; ammo: arrow/bolt/…) is the documented drop on import (itemImport.ts:479). Add a second subtype dropdown driven by the chosen consumable type, and a column/packed-slug to persist it.
  • Ammo damage — when type = ammo, Foundry shows a Replace Damage toggle + a base-damage editor. Missing.

4 · Properties: per-type filtering cross-cutting

(Foundry also renders these as a checkbox grid vs our add-chip; cosmetic.)

5 · Container capacity model mismatch

Foundry's details-container.hbs capacity is three independent fields: Item Count + Volume (value+units, volumeUnits) + Weight (value+units, weightUnits). Ours is a single either/or toggle (capacity.type = items | weight, one value). To round-trip 5.3.1 containers faithfully we'd move to count + volume + weight. Lower priority — our model captures the common case.

Already correct ✓ no change

  • Armor block (AC value / max-dex / strength / magical bonus), tool block (ability / bonus / chat flavor), weight & price shapes, base-item resolution — all faithful.

Suggested order — small → large

  1. Option-list fixes (§1) — rarity empty, weight units, consumable wondrous, equipment natural, weapon mastery dropdown. An afternoon; all low-risk.
  2. Per-type property filtering (§4) — reuse item_properties.valid_types. One section, clean.
  3. Weapon sub-form (§2) — mastery (done in #1) + versatile + ammunition type.
  4. Consumable sub-type + ammo damage (§3) — needs the persistence decision (new column vs packed slug); closes the documented import gap.
  5. Container capacity model (§5) — only if faithful 5.3.1 container round-trip matters.
Note on damage shape: the weapon damage uses a single die + single type. Foundry's field-damage is the same editor we just rebuilt for activities — if we want full parity (multi-type, versatile), we can reuse DamagePartEditor here rather than the bespoke inline fields.
