Current branch scope: system-applications

Three pieces in strict order. Phase 2 is gated on the article revamp; the article revamp is independent of Phase 1.5 but both should land before Phase 2 begins.

1. Phase 1.5 — hash-on-upsert wiring

Active no remote-D1 dependency

Phase 1 added a content_hash TEXT column to spells / feats / items / classes / subclasses / features / unique_option_groups via migration 20260527-1420_content_hash_columns.sql. Every existing row's content_hash is currently NULL.

What needs to happen:

  • Add a SHA-256 compute hook to src/lib/d1.ts upsertDocument (or to per-entity upsert helpers in src/lib/compendium.ts). Recipe is in the handoff: canonical-key-ordered JSON of an explicit content-field allowlist (name, description, activities, effects, advancements, ...), hashed and persisted.
  • Mirror the hook in api/_lib/d1-internal.ts for server-side upsert paths.
  • One-shot backfill script (new: scripts/backfill-content-hashes.mjs) that walks every existing row, computes the hash, writes it back.

Phase 4 update-detection consumes this column — it's how the module's batch-status endpoint detects "this Fireball on the actor was authored against a stale source version."

2. Article system unification — blocks Phase 2

Active migrations required

lore_articles is the odd one out across content tables. Every other content table uses (identifier, source_id) as identity; lore_articles uses slug + parent_id (hierarchy-based). That works for the existing wiki but breaks three things needed by Phase 2:

  • @article[slug] references have no namespacing — two campaigns shipping articles with the same name collide.
  • System pages (parent + glossary children) need addressable child identifiers that aren't just the parent's row id.
  • Foundry-side DauligorViewer needs stable deep-link URLs.

Recommended path:

  1. Migration: ALTER TABLE lore_articles ADD COLUMN identifier TEXT NOT NULL DEFAULT '', backfill from slug.
  2. Migration: ALTER TABLE lore_articles ADD COLUMN source_id TEXT REFERENCES sources(id).
  3. After de-dup, migration: UNIQUE(COALESCE(source_id, ''), identifier).
  4. Add content_hash TEXT (matches the Phase 1 pattern).
  5. Update src/pages/wiki/Wiki.tsx + LoreArticle.tsx + LoreEditor.tsx to route by identifier instead of slug.
  6. Update src/lib/bbcode.ts [article=…] / @article[…] resolver to compile to identifier-based hrefs.

3. Phase 2 — read-only live viewer

Gated on #2 module + app

The first time Foundry sees Dauligor reference-link content live.

App side:

  • New endpoints in functions/api/module/[[path]].ts:
    • GET /api/module/<source>/articles/<slug> — full HTML (server-side BBCode → HTML)
    • GET /api/module/<source>/articles/<slug>/summary — short HTML for hover tooltips
    • GET /api/module/<source>/system-pages/<system>/<key>
  • New api/_lib/_articleExport.ts + _systemPageExport.ts.
  • New "system page" article type on the app side — parent article + child glossary entries, addressable by slug. Mirrors dnd5e's Conditions / Rules journal structure. New files: SystemPagesList.tsx, SystemPageEditor.tsx, SystemPageGlossary.tsx.

Module side:

  • New DauligorViewer ApplicationV2 class (module/dauligor-pairing/scripts/dauligor-viewer.js) — opens when a <a class="dauligor-link"> is clicked, fetches HTML, renders in a Foundry window.
  • Custom enricher registration via CONFIG.TextEditor.enrichers.push(…) for @article[slug], @condition[key], @rule[key], etc. (New directory: scripts/enrichers/.) Do not override TextEditor.enrichHTML — that breaks every other module.
  • Hover tooltips populated via data-tooltip-html from lazy /summary fetches.
  • World-local cache via Foundry world setFlag, keyed by kind/source/key/hash. Offline reads fall through to cache.

Non-negotiable constraints

From AGENTS.md — these gate every piece of work above.

  • No Firestore imports. The database is gone. Use the src/lib/d1.ts helpers exclusively.
  • No INSERT OR REPLACE. D1 enforces PRAGMA foreign_keys = ON by default; INSERT OR REPLACE resolves PK conflicts by deleting + reinserting, which fires ON DELETE CASCADE on FK children. Documented data-loss incident. Use ON CONFLICT(id) DO UPDATE.
  • No backwards-compat dual reads. When schema changes, update sources to fit the new shape — don't leave row.snake_case ?? row.camelCase readers.
  • queryD1 JSON column passthrough. Inline JSON-column remaps must do typeof X === 'string' ? JSON.parse(X) : (X ?? <default>) so already-parsed values pass through.
  • Drift-managed sibling pairs. Changes to src/lib/X.ts for classExport, classProgression, referenceSyntax, requirements must also land in api/_lib/_X.ts.
  • D1 migrations: local first; remote only with explicit per-migration permission in the current conversation. Earlier "go aheads" don't transfer.
  • Drive the local dev stack via tool background-execution — don't ask the user to flip to a terminal.
Branch-switching gotcha: the harness has been observed to insert git switch calls mid-session. Chain branch-relevant ops in a single Bash call (git switch X && git rebase Y && git push …) and verify with git branch --show-current before committing.
