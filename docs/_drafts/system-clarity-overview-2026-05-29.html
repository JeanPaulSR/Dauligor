Themes across the systems

  • Stub tabs that lie. Campaign & Timelines both show “coming soon / no X yet” placeholders that read as empty data rather than unbuilt features. A single honest pattern (hide, or a clear “not built” banner) would clarify both.
  • The lore identity model is the one true blocker. slug + parent_id (vs the identifier + source_id every other table uses) is what gates live-content bridge Phase 2 — not the system-page work, which shipped around it.
  • Config duplicated by hand. Category lists, metadata mappings, and snake→camel remaps are copy-pasted across pages (lore ×3–4, campaign per-page). Consolidation removes a whole class of silent drift bugs.
  • Authoring half-loops. Maps can place pins but not create maps; highlights render but can’t be drawn. Several features are “readable, not authorable.”
  • Verify before fixing. The Maps investigation was ~40% stale against real source. Treat the other four reports as leads to confirm, not facts — consistent with the project’s “Survey First” & “Empirical Verification” rules.

Suggested order of attack

  • 1Home — quick wins polish
    Lowest risk, highest visibility. Dynamic DM name, fix/remove the dead CTA, fetch-error state, responsive skeleton, import cleanup. An hour of clean, shippable polish.
  • 2Maps — close the loop build
    Add the missing map-CRUD UI (create map · set era · upload background). Turns Maps from “demo” into a usable feature; pins already work on top.
  • 3Campaign — verify & fix verify
    Confirm the image_display schema gap, then either migrate or strip it; enable co-DM assignment; resolve the stub tabs; refresh the schema doc.
  • 4Timelines — decide decision
    Build-vs-remove call needed before any code. At minimum, kill the dead sidebar links so nothing points at nothing.
  • 5Lore — consolidate refactor
    Bigger arc: the identifier unification (unblocks Phase 2) plus category/metadata de-duplication. Worth its own focused session.
