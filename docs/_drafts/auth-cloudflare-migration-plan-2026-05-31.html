Bringing Accounts Home to Cloudflare

A plan to retire Firebase Authentication and self-roll account credentials on the existing Cloudflare / D1 stack — so the Archive runs on one stack, end to end, with no managed-auth vendor in the loop.

Nothing has changed yet. This is a survey + plan for review. No code edited, no migration written, no database touched. The decisions in §9 gate the build.

§1 Where we actually are today

The story you may remember — "we moved off Firebase" — is half true, and the half that's done is the half people usually mean. Worth being precise, because it changes how much work is left.

Key fact
Firebase the database is gone. Firebase Authentication is not. The Firestore→D1 migration completed in May 2026, and the server-side firebase-admin SDK was already removed — the proxy now verifies Firebase ID tokens with jose against Firebase's public JWKS, no credential file needed for the read path. What remains is that passwords still live in Firebase Auth, and admin user-management still calls Firebase's Identity Toolkit REST API.

Concretely, here's what's already Cloudflare-native versus still tied to Firebase:

The synthetic-email layer is the other piece of color: users log in with a handle, not a real email. usernameToEmail('cleric_dan') → cleric_dan@archive.internal — a fake domain that never receives mail. Real addresses, when present, live in users.recovery_email and are not wired to anything. This matters for password-reset design (no inbox to send a link to).

§2 The goal, stated plainly

  • Passwords become ours: an Argon2/scrypt hash on each users row in D1.
  • The Worker issues the token: on login we verify the hash and sign our own JWT with a Worker secret.
  • Firebase leaves entirely: the firebase client SDK is removed, FIREBASE_SERVICE_ACCOUNT_JSON is deleted, and the Identity Toolkit calls become plain D1 writes.
  • One stack: browser → Cloudflare Pages Functions → D1 / R2. No third-party auth vendor anywhere in the request path.

§3 Target request flow

LOGIN   browser → POST /api/auth/login {username, password}
   ↳ Worker: usernameToEmail? no — look up users by username
   ↳ verify password against users.password_hash (Web Crypto / scrypt)
   ↳ sign JWT { sub:user.id, role, exp } with AUTH_JWT_SECRET
   ↳ return { token }  (+ optional refresh cookie)

REQUEST   browser → any endpoint, Authorization: Bearer <our-jwt>
   ↳ require* helper: jwtVerify(token, AUTH_JWT_SECRET)  (was: Firebase JWKS)
   ↳ read role from D1 (unchanged) → allow / deny

SIGN-OUT   client drops the token + clears cache (no Firebase session to revoke)

The shape barely changes — every require* gate already reads the role from D1; only the token-verification line swaps from Firebase's public keys to our own secret. That's the leverage point that makes this tractable.

§5 The one genuine technical constraint: hashing on Workers

Cloudflare Workers have no Node native addons, so the usual bcrypt/argon2 native libraries don't run. Three viable, Workers-safe paths:

PBKDF2 via Web Crypto

crypto.subtle is built into the Workers runtime. Zero dependencies, runs natively, FIPS-grade. ~310k iterations.

Pro: nothing to add. Con: weaker against GPU cracking than memory-hard KDFs — irrelevant at ~10 trusted accounts.

scrypt via @noble/hashes

Pure-JS, audited, memory-hard. One small dependency. The stronger choice if we want best-practice hashing regardless of scale.

Pro: memory-hard. Con: one dep; slightly slower per login (fine at this volume).

A library: Lucia / Better Auth

Full session frameworks that pair with D1. They bring hashing, sessions, and reset flows — but also opinions and surface area we'd have to bend to the existing users schema + RBAC.

Pro: batteries included. Con: heavier; may fight our bespoke role model. Likely over-built for ~10 users.

§6 The hard part: migrating existing users

We cannot read anyone's current password out of Firebase — that's the whole point of a hash. So every existing account needs a credential transition. Three strategies, in order of elegance:

Hash-on-next-login (dual window)

Keep Firebase login alive for a short window. When a user logs in successfully via Firebase, the client has the plaintext for that instant — POST it to /api/auth/adopt, which writes the D1 hash and flips the account to native auth. Zero user disruption; accounts migrate themselves as people sign in. After the window, force-reset the stragglers.

Best UX. Cost: a temporary dual-auth code path that we delete after.

Admin-set temp passwords

Admin sets a temp password for each of the ~10 accounts in one sitting; distribute out-of-band (the owner knows the players). Each user changes it on first native login.

Simplest to build. Cost: a manual afternoon + a "change me" nudge.

One-time reset links

Mint a reset token per user, hand out /auth/redeem?token=… URLs ("we upgraded our login — set a new password"). Reuses the sign-in-link machinery we already have.

Clean story. Cost: everyone must act once before they can log in.

§7 Suggested rollout (each phase shippable)

1

Schema + hashing primitives

Migration adds password_hash / password_updated_at (local first, then remote with the code). Add hash + verify helpers. No behavior change yet.

2

Native login endpoint, dark

POST /api/auth/login + our JWT signing/verifying, behind the scenes. require* helpers learn to accept either a Firebase token or ours during transition.

3

Migrate the accounts

Run the chosen §6 strategy. Verify every account has a D1 hash.

4

Flip the client

Navbar + App use our auth; admin CRUD writes D1 instead of Identity Toolkit; RedeemTokenPage redeems our tokens.

5

Remove Firebase

Drop the dual-token acceptance, the firebase dep, firebase-applet-config.json, and FIREBASE_SERVICE_ACCOUNT_JSON. Update docs (auth-firebase.md → auth.md).

§8 Risks & landmines (some already paid for)

  • Remote migration must ship with the code. The reverted password-reset attempt applied must_change_password only locally; remote endpoints would 500. Same class of bug bit callout/column on the home builder. Apply password_hash to remote D1 (with per-migration go-ahead) before/with the code.
  • The "requires-recent-login" pain disappears. Firebase's updatePassword needed a fresh login (the reverted attempt tripped on this). Once we own the password, a self-service change is just "verify old hash, write new hash" — no re-auth dance. A genuine simplification.
  • No real inboxes. @archive.internal can't receive a reset email. Self-service forgot-password needs either a real address (recovery_email + Resend, later) or admin-mediated reset. For ~10 users, admin-mediated is enough; flag email as a v2.
  • Token longevity / refresh. Firebase auto-refreshed a 1-hour token. We choose: a single sliding ~30-day JWT (simplest, fine at this scale) or access+refresh (correct, more moving parts). Decision in §9.
  • Keep the hardcoded staff-email bypass in sync. Three files still special-case the owner/admin/gm principals; that logic survives the swap and must move with us.
  • Don't push to main. All of this lands on dauligor-applications and ships only on explicit request — per standing rule.

§9 Decisions I need from you before building

1 · Hashing approach

PBKDF2 (Web Crypto, zero deps) vs scrypt (@noble/hashes, memory-hard) vs a full library (Lucia/Better Auth).

◆ Recommend: scrypt via @noble/hashes — best-practice hashing, one tiny pure-JS dep, no framework lock-in.

2 · Existing-user cutover

Hash-on-next-login (zero disruption, dual window) vs admin-set temp passwords vs one-time reset links.

◆ Recommend: hash-on-next-login if you want no disruption; admin-set temp passwords if you'd rather just do it in one sitting and skip the dual-auth code.

3 · Token model

Single sliding ~30-day JWT vs short access token + refresh token.

◆ Recommend: single sliding JWT — at ~10 trusted users the refresh-token machinery isn't worth its complexity.

4 · Scope of this pass

Full exit (Phases 1–5, Firebase gone) vs foundation only (Phases 1–2: schema + native login dark, leave the flip + Firebase removal for a later pass).

◆ Recommend: do it as a deliberate sequence — I build through Phase 2, we verify login works against a test account, then proceed to the account migration and the flip. Each phase is shippable and reversible.
