Manual Uploads — the "Mark & Build" Window design

An in-app window: paste source text, mark spans as compendium fields, and save every entity through the editors' exact write calls. 2026-06-04 · branch manual-uploads · rev 2 — pivoted to in-app after verifying the write paths

Take a block of text from a PDF (or anywhere), highlight a span and mark it — "this is the class name," "this paragraph is a feature's description," "this section is a new spell." The window assembles real compendium entities and writes them with the identical calls and payloads the hand editors use. Reading the PDF is still your job (or the chat's); the window's job is faithful field-filling and saving.

Confirmed decisions

1 · An in-app window with section marking. A new admin page where you paste or load source text, select spans, and tag each as a field of an entity (or as a whole new entity). Manual typing is always available too — marking is a convenience, not the only path.

2 · Byte-identical writes. Every commit goes through the same function the matching editor calls — upsertSpell/upsertFeat/upsertItem/upsertFeature or a direct upsertDocument(collection, id, payload) — building the same payload, minting ids the same way, and firing the same rebake side-effects. No reimplementation of the SQL layer.

Preview, then direct write. The staged-entities panel is the preview — you see each resolved entity and every field before you click Create. On confirm it writes directly and goes live (no proposal queue). Skeleton depth — name, description, required fields, sensible defaults; rich mechanics finished in the editors.

Why a window and not a chat/CLI script (the pivot). Decision 2 settles it. The faithful write functions are browser-side React — upsertSpell/upsertFeat/upsertItem/upsertFeature plus the per-type upsertDocument payloads. A Node CLI could reach D1 (auth is now the app's own native session token — a Worker-issued HS256 JWT via /api/auth/login, with Firebase only a migration-window fallback — so a script could log in for one, or hit the Worker's /query with API_SECRET). The real blocker is fidelity: a CLI would have to reimplement the field-shaping (normalizeCompendiumData, the spell filter-buckets, every per-type payload), which is exactly the drift Decision 2 guards against. The window reuses those functions verbatim in their native context — and already holds a live authed session — so it is the faithful home. The chat's role is to help you draft the text you paste.

The proven precedent

We are not inventing the shape. SpeciesBackgroundImportWorkbench.tsx already is a working in-app import window: load content → master list + detail/preview pane → per-row source assignment → commit via upsertDocument/upsertDocumentBatch with natural-key dedup and createdAt preservation. The four Foundry workbenches (Spell / Feat / Item / Species-Background) are the reference implementations. Mark & Build is the same skeleton with a different front-end: instead of parsing an uploaded Foundry JSON, the candidate entities are built by you marking spans of pasted text.

Required fields (what a candidate must carry)

The only universally hard-required field is name. The window auto-fills id and identifier; everything else defaults. Parents that must exist first: subclass→class (hard FK), and soft links (feature→parent, option-item→group, subspecies→species, background-feature→background) which the window resolves to a row already in the DB or another staged candidate.

The marking interaction

  1. Load source. Paste text into the source pane (or drop a .txt). PDF→text extraction stays out of scope — paste the text. (We can layer extraction on later without touching the window.)
  2. Start an entity. Select the title span → "Mark as → New class" (pick the type). A staged entity appears with its name filled from the span.
  3. Mark fields. Select more spans → "Mark as → description of Warden," "→ new feature under Warden," etc. Each mark routes the span text into the chosen entity's field, or spawns a child entity with the right parent link pre-set.
  4. Re-tag freely. "Mark a section as something else" — re-assign a span you tagged wrong; the old assignment is cleared and the field updates. Marks are non-destructive to the source text.
  5. Fill the rest by hand. The field inspector is a live form — type anything the source didn't give you, pick a source book, attach an image. Required-field gaps are flagged.
  6. Create. Per-entity (or "Create All," parent-first) → the registry's commit() runs the verified write call. Idempotent (ON CONFLICT(id)), so re-running is safe.

Composites (a whole class)

A class is a parent row plus features/subclasses/scaling — each its own write. In the window you stage the class, then mark child features/subclasses whose parent is the staged class; on "Create All" the registry writes parent-first so soft FK links resolve, firing queueRebake('class'|'subclass'|'feature', id) exactly as the editors do. (For a complete, well-formed class-export JSON specifically, the existing importClassSemantic() remains a one-click shortcut — but the marking path is what makes partial/skeleton class-building from raw text work.)

Build phases

1
Registry + commit fidelity — src/lib/import/registry.ts: per-type descriptor whose commit() calls the verified write function with the verified payload. Start with spell (self-contained, exercises the helper path) + class (exercises direct upsertDocument + rebake). Read each editor's handler one more time as I wire it, to match verbatim.
2
The window shell — new admin route, paste pane + staged-entities list + field inspector, cloned from the workbench layout. Manual entry first (no marking yet): pick a type, fill fields, Create. Proves the registry end-to-end against local D1.
3
Span marking — selection → "Mark as" popover → route span text into a field / spawn a child entity. The actual "tag a section as something else" mechanic.
4
Breadth — fill in the remaining types' descriptors (feat, item, feature, subclass, background, species + options, unique options, facility, source, system pages, spell rules) + composites/parent-first ordering.
5
Polish + docs — image attach, source bulk-assign (workbench idiom), validation surfacing; promote this draft into docs/features/.

Deferred / out of scope

  • Out of scope: PDF/text auto-extraction and any AI parsing. You (or the chat) supply the text; the window does faithful field-filling + saving.
  • Proposal-queue path — not used (direct write chosen). Swappable later by pointing commit() at the proposal writer.
  • Rich mechanics (advancements/activities/effects) — accepted if marked, but authored properly in the editors; v1 targets skeletons.
  • Auto-segmentation / "detect the features for me" — explicitly manual in v1 (matches "fill in fields," no automation).
