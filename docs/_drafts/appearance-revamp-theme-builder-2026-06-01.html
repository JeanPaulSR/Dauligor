1 · The core insight

The theme engine is already exactly the right shape — it just isn't exposed. Today a "theme" is a named set of semantic CSS variables on <html> (--ink, --gold, --background, --card, --border, --radius, fonts…), and accent_color already proves the full pattern: a user value is injected live into a CSS variable at the document root.

Therefore "Full customization" is not a rewrite — it is generalizing accent_color into the whole token set: let the user author values for the meaningful variables, store that set, and inject it the same way. The live preview is just those same variables rendered onto sample components in a scoped container instead of the whole app.

3 · The styling sections

You've been insistent that the styling sections read clearly — so the builder is organised as named sections, each mapping to a small group of real tokens. Every control is a guard-railed editor over CSS variables already in src/index.css. The v1 starter set is deliberately just four colours over a preset:

Section v1What the user setsUnderlying tokens
Base presetStarting point: Parchment · Light · Darkthe .parchment/.light/.dark class — seeds every value below
BackgroundPage background colour--background (→ derived --secondary, --muted)
Card surfaceCard / panel background colour--card (→ derived --popover)
TextPrimary text — and, via opacity, its whole faded hierarchy (`text-ink/NN`, 2000+ uses, auto-tracks this)--ink/--foreground (contrast-guarded)
Secondary textThe distinct muted tier — descriptions, hints, timestamps--muted-foreground
HighlightThe app's accent colour (buttons, labels, active states)--gold, --primary, --ring
On text hierarchy The app builds depth with opacity on the text colour (`text-ink/10`…`/95`, ~2000 uses). Those faded tiers are translucent shades of the chosen Text colour — they track it automatically, so "Text" governs the whole hierarchy. "Secondary text" is the separate hue used by the muted tier (`--muted-foreground`); it's the only secondary colour worth a distinct control.
Deliberately out of v1 Density is excluded entirely — layouts are already predesigned, so spacing is not a user knob. Borders & shape (--border, --radius) and typography (font pair, size) are held for a possible later pass — colour first.
Stays locked on purpose --blood / --destructive (danger) is never user-editable — danger must always read as danger in every theme. This is already a documented invariant ("blood and gold are theme-stable").

4 · Guard-rails — the "never-ugly" promise

The difference between "full customization" and "a support nightmare" is guard-rails. The builder enforces readability so a user literally cannot save an unusable theme:

  • Live contrast check (WCAG AA). Text-vs-surface and accent-vs-surface ratios are computed on every change. Below threshold → an inline warning with a one-click "nudge to readable" that snaps the colour to the nearest passing value.
  • Derived, not hand-set, where it's risky. Secondary surfaces (--muted, --secondary, hover tints) are derived from the surface + accent rather than exposed raw — fewer ways to break it, less to fiddle with.
  • Per-section reset + "Reset to preset." Any section reverts to its base-preset value; one button restores the whole theme.
  • Danger colour locked (see above).

5 · Live mock-up architecture

The critical technical choice. Today's preview mutates the global <html>, so edits leak into the whole app immediately. The revamp scopes the preview:

Scoped CSS-variable container The in-progress token values are written as inline custom properties on a single wrapper <div> around the mock-ups. Because every component already styles via var(--token), the cascade overrides only inside that div — the surrounding real app is untouched. State → preview is instant; the global app changes only on Save.

Mock-ups are lightweight, static-data replicas of real pages built from the existing semantic classes (.lore-card, .compendium-card, .h1-title, .btn-gold-solid, .data-table…) — no data fetching, so the preview is instant and self-contained. A segmented control swaps which surface is shown.

Max-isolation alternative (deferred): render mock-ups in an <iframe> sandbox. Cleaner isolation, but needs the stylesheet injected — overkill for v1 since the app is already fully var-driven.

7 · Data model recommended: new table

Per our standing preference (new table for new functionality — extracting later is harder than starting separated), store custom themes in their own table rather than as a blob on users. This also unlocks multiple named themes per user and future sharing.

CREATE TABLE user_themes (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name TEXT NOT NULL, -- "My Dusk", "High Contrast"
  base_preset TEXT NOT NULL, -- parchment | light | dark
  tokens TEXT NOT NULL, -- JSON: only the overridden vars
  created_at TEXT, updated_at TEXT
);
-- pointer on users; NULL = use built-in preset (back-compat):
ALTER TABLE users ADD COLUMN active_theme_id TEXT REFERENCES user_themes(id);
  • Backward compatible: active_theme_id IS NULL → today's behaviour (users.theme + accent_color). Nobody is disrupted.
  • Tokens = overrides only (sparse JSON), so a custom theme is a small diff against its base preset, and preset improvements flow through.
  • Simpler alternative if you'd rather: a single users.custom_theme TEXT blob (1 theme/user, no table). Less future-proof; flagged for your call in §11.

8 · How it applies app-wide

One generalisation of the existing effect in App.tsx (123–138): instead of toggling a class + 3 vars, resolve the active theme (preset class as the base, then the override tokens) and inject the override set onto the document root. Null active theme → unchanged. The whole rest of the app keeps reading the same var(--token)s it always has.

10 · Build plan (phased)

1
Engine + data (dark — no visible UI). Add user_themes + active_theme_id; generalise the App.tsx apply step to inject an override token set; read/write endpoint. Backward compatible — ships invisibly, verify nothing changes for existing users.
2
Scoped preview + mock-ups. Build the scoped-container preview and the 4 static mock-up surfaces from existing classes. Standalone, testable in isolation.
3
The guided builder. The four colour controls over a preset, contrast guard-rails, per-section + global reset, Save → persist + apply. This is the headline feature.
4
Multiple named themes + share/export. future / optional Switch between saved themes; export/import a theme as JSON; optionally share a theme with the group.
