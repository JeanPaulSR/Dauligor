Items system — deep dive & state-of-play

What the items / equipment compendium actually is today — schema, editor, import & export — surveyed across ItemsEditor.tsx, itemImport.ts, _itemExport.ts, and the six items-table migrations, with the Foundry side checked against Foundry-JSON\items. 2026-06-07.

Bottom line: unlike the activity editor — which we just rebuilt kind-by-kind — the items system is already a mature, end-to-end pipeline. A single unified items table absorbs all six Foundry item types; ItemsEditor is a full seven-tab editor; and items round-trip both ways with the Foundry module (import via itemImport.ts, and live export via _itemExport.ts → /api/module/items/<id>.json). The activity work we just finished already powers items — ItemsEditor mounts ActivityEditor + ActiveEffectEditor directly. So "items" is not a build; it's a polish / gap-closing question.
On the example exports: E:\DnD\Professional\Foundry-JSON\items\ holds exactly one file — item-feature.json (a feat, empty system). The two fvtt-Item-sorcerer-*.json at the JSON root are class items. So there are no rich weapon / armor / consumable / container example exports on hand. A true window-by-window fidelity audit (the activity-editor treatment) would need item screenshots + the dnd5e item sheets — flagged below under "if the goal is editor fidelity."

Architecture — one table, six types, both directions

  • Unified table. Every Foundry item type lands in items (alias items in d1Tables.ts). The legacy weapons / armor / tools tables were demoted to proficiency-category definitions (edited in AdminProficiencies); items link to them by FK.
  • Six migrations of evolution (0004_items → 20260524-1800 foundry_aligned → 20260525-1800 unified_shapes → 20260526-1700 completeness → 20260527-1200 advancements). The shapes were deliberately Foundry-aligned from the start — weight/price are {value, units|denomination}, properties are slug arrays, etc.
  • Base-item FKs. base_weapon_id / base_armor_id / base_tool_id resolve Foundry's system.type.baseItem slug ("greatsword") against weapons/armor/tools.identifier. One FK set per row; the raw slug stays in base_item as a fallback if no proficiency row matches.
  • Bidirectional with Foundry. Import (itemImport.ts + ItemImportWorkbench) and export (buildItemBundle in _itemExport.ts, served live at /api/module/items/<dbId>.json, kind dauligor.item-item.v1). Export even synthesizes ScaleValue advancements from the item's scaling columns so @scale.<id>.<col> resolves in Foundry — no module-side hook needed.
  • Editor. ItemsEditor.tsx (~2.3k lines), proposal-aware, seven sub-tabs: Basics · Mechanics · Activities · Advancement · Scaling · Effects · Tags.

Where it's solid ✓

  • Authoring + round-trip for all six types — import, edit, export verified to exist end-to-end.
  • Activities & Active Effects — mounted straight from our (now-complete) editors. Everything we just did to activities benefits items for free.
  • Base-item ↔ proficiency linking — drives the character sheet's weapon/armor/tool proficiency resolution.
  • Scaling → advancement synthesis on export — magic items with scaling columns (e.g. Amulet of the Devout) export correctly.
  • Proposal-aware — fits the content-proposal workflow like the other compendium editors.

If the goal is editor fidelity — the activity-editor treatment

We could run the same window-by-window pass on ItemsEditor that we ran on activities — compare each type's sub-form to Foundry's item sheet, fix labels / option lists / conditional layout. Two caveats:

  • The fields were built Foundry-aligned already (the migration is literally named foundry_aligned_item_shapes), so the delta is likely smaller than activities' was.
  • There are no item example exports to mine for canonical option lists — we'd lean on dnd5e's item data models + your screenshots, type by type.

Recommendation — pick the lane

  1. Treat items as "done enough" and go to monsters. Monsters are the genuinely-new build, and they unblock the Summon/Transform creature pickers we parked. Items already round-trip; the gaps above are polish.
  2. Close one clean gap first — the two-axis subtype is the tidiest (one column + import/export wiring), if you want poison/ammo specificity to survive round-trip before moving on.
  3. Editor-fidelity audit — only if you want items visually pixel-matched to Foundry like the activity windows; needs screenshots, type by type.
My read: items is in good shape — the high-leverage next move is monsters (option 1), since that's the missing piece blocking the parked activity work and there's no creature/NPC table at all yet. Worth a quick decision before we invest in item polish.
