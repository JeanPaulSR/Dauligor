Design Spec · Dauligor · 2026-06-04

Lore Article Block Designer

Repurpose the campaign homepage block builder into a fullscreen designer for authoring lore articles as block layouts. Lore articles first; system articles later (their reference-backed entries need extra work).

✅
Decisions locked.
  • Standalone fullscreen designer — one designer that folds all article settings into a side panel; the body is the block canvas. (Replaces the current tabbed LoreEditor.)
  • Migrate all bodies to blocks — every existing article's BBCode content is wrapped into a single text block; blocks become the one body model; the viewer always renders blocks.

1 · Core insight — nothing is lost

The block model's text block already stores BBCode. So every authoring capability survives the move, just relocated:

CapabilityWhere it lives after
BBCode formatting, @kind[id] entity refs, &kind[id] rule refs, [era]/[campaign] conditional spansInside text blocks (rendered per-block via BBCodeRenderer) — unchanged
Secrets (reveal-per-campaign), visibility junctions (era/campaign), draft statusArticle-level systems — kept exactly as-is, surfaced in the designer's settings panel
Metadata sub-tables (character/location/org/deity), tags, parent hierarchy, images, dm_notes, excerptArticle-level — moved into the settings panel; same tables, same save queries

The builder only changes how the body is authored & laid out — it does not touch the surrounding article machinery.

2 · Architecture — generalize the block engine

The homepage builder is ~95% campaign-agnostic. Rather than copy it (and per the project's "extract a shared primitive, don't bend a specific one" principle — cf. FocalImageField), we generalize it into a reusable layout engine that both campaigns and lore articles (and later system articles) consume.

Today (campaign-only)Generalized
src/lib/campaignHome.tssrc/lib/layoutBlocks.ts — generic block model + serialize/parse; the campaign-only recommended block is registered by the campaign consumer, not baked in
CampaignHomeEditor.tsx (props: campaignId, campaignName)LayoutEditor — props: blocks, onChange, allowedTypes, and a host-supplied settings panel slot
CampaignHomeBlocks.tsx (prop: recommendedLore)LayoutBlocks renderer — block-type renderers are a registry; recommended stays a campaign-supplied renderer
fetch/saveCampaignHomeBlocks(campaignId)Generic fetch/saveLayout(loadUrl, saveUrl); campaigns & articles pass their own endpoints

Phase 1 refactors campaigns onto the generic engine with zero behavior change (the live homepage keeps working), which de-risks everything after it.

3 · The standalone designer (UX)

Fullscreen, like the campaign homepage builder, but the side panel now owns the whole article.

  • Canvas (center) — the block layout: live preview + structure, drag-to-reorder, add/duplicate/delete, nesting (group/columns). Same interactions as the homepage builder.
  • Settings panel (side) — collapsible sections folding the old LoreEditor tabs:
    • Details — title, slug, category, folder, parent, excerpt, draft/published toggle
    • Visibility — era & campaign scope multi-selects
    • Imagery — header / card / preview images (via the shared FocalImageField)
    • Taxonomy — tags; per-category metadata sub-form (character/location/org/deity)
    • Secrets — existing reveal-per-campaign secret manager
    • Storyteller Notes — dm_notes (staff-only)
  • Block inspector — when a block is selected, the side panel shows that block's controls (same Inspector as today).

4 · Block set for articles

Reuse the existing types; drop the campaign-only one; leave room to add article-flavored blocks.

  • Keep: hero, text (the workhorse — BBCode), image, divider, callout, group, columns/column, entity-feature, entity-row (embed class/spell/item/article cards in lore — genuinely useful).
  • Drop: recommended (campaign-specific).
  • Later candidates: quote, infobox (metadata card), gallery, and a block-level secret/era-scope wrapper (defer — inline [era]/[campaign] still works in text).

5 · Storage

Mirror the proven campaign_home_blocks shape so the generalized save/load idiom (replace-all rows) is reused verbatim:

CREATE TABLE lore_article_blocks (
  id TEXT PRIMARY KEY,
  article_id TEXT NOT NULL REFERENCES lore_articles(id) ON DELETE CASCADE,
  block_type TEXT NOT NULL,
  "order" INTEGER NOT NULL DEFAULT 0,
  config TEXT, -- JSON; containers nest children in config.children
  created_at / updated_at
); -- INDEX (article_id, "order")

lore_articles.content is retained but demoted: no longer the render source. We keep it populated with a plaintext/BBCode mirror (derived from text blocks) so existing search, excerpts, and the lore_links mention-extraction keep working without a rewrite. (Alternative: move search to the blocks — out of scope for v1.)

6 · API

  • GET /api/lore/articles/:id/blocks and PUT …/blocks — mirror the campaign home-blocks handlers (staff-gated writes, replace-all). Include blocks in the existing article packet (GET /api/lore/articles/:id) so the viewer gets them in one round-trip.
  • On article save, derive the content mirror from the text blocks server-side (keeps mention-extraction + search intact).

7 · Migration (one-time, both DBs)

  1. For every lore_articles row with non-empty content: insert one lore_article_blocks row — a text block whose config.body = the existing content.
  2. Idempotent guard: skip articles that already have block rows.
  3. Apply to local (dev worker) and remote (one file via d1 execute --remote --file, per the migration-landmine rule — never migrations apply).

After migration, articles render identically (one text block == the old body), so it's a safe no-visual-change cutover.

🔎
Preservation check (verified 2026-06-04, prod D1). 11 articles, all with content; none use [era]/[campaign] conditional blocks, entity/rule refs, wiki-links, or tables today (largest body ~33 KB). So every existing article wraps losslessly into one text block and renders identically — the migration is a clean, no-visual-change cutover for the current set.
Required for correctness: the generic LayoutBlocks text renderer MUST thread viewContext (era / campaign / isStaff) into BBCodeRenderer. The current homepage renderer (CampaignHomeBlocks.tsx:242) calls <BBCodeRenderer content={block.body}/> with no context — fine for homepages, but for articles that omission would stop [era]/[campaign] spans from being filtered (restricted content could leak). No current article triggers this, but the engine must support it before anyone authors conditional content. Added to Phase 1 + Phase 4.

8 · Viewer changes

LoreArticle.tsx: replace the single <BBCodeRenderer content={article.content}/> body with the generic <LayoutBlocks blocks={article.blocks}/>. Secrets, mentions, metadata card, dm_notes, sidebar — all unchanged. Text blocks render via BBCodeRenderer with the article's viewContext threaded in (see the correctness note in §7), so refs/conditionals/secrets-in-text behave exactly as before.

9 · System articles (deferred — why)

System pages (system_pages/system_page_entries) differ: entries can be entity-backed (source_kind/source_id), they're site-wide (no campaign scope), and they use the &kind[id] rule-reference sigil with a master-detail editor. The block engine will extend to them, but their reference-backed entries need bespoke block types (e.g. an "entry" block bound to a source row) — a later phase once the lore flow is proven.

10 · Phased plan

Phase 1 · Generalize

Extract the layout engine

layoutBlocks.ts + LayoutEditor + LayoutBlocks; refactor the campaign homepage to consume them. Add an optional viewContext prop threaded into text blocks (homepage passes none → unchanged; articles will pass theirs). No behavior change — verify the live homepage still works. foundation
Phase 2 · Storage + API + migrate

Article blocks persistence

lore_article_blocks table; blocks sub-route + packet inclusion; content mirror on save; one-time migration (local + remote). DB change · remote apply
Phase 3 · Designer

Standalone fullscreen article designer

New designer page; fold LoreEditor's tabs into the settings panel; canvas = LayoutEditor (article block set). Route /wiki/edit/:id → designer. Keep the old editor reachable until parity. UI
Phase 4 · Viewer

Render blocks

LoreArticle renders LayoutBlocks; retire the BBCode-body render path. UI
Phase 5 · Polish + retire

Cleanup

Remove the old tabbed editor once parity is confirmed; add article-flavored blocks if wanted; docs. cleanup
Later

System articles

Extend the engine with entity-backed entry blocks + &-reference handling. future

11 · Risks & open questions

  • Generalizing live code (Phase 1) — the campaign homepage is in production; the refactor must be behavior-preserving. Mitigation: extract first, point campaigns at it, verify, then build lore on top.
  • The content mirror — deriving plaintext/BBCode from a block tree for search/mentions. Decision needed: mirror on save (recommended) vs. migrate search to blocks (bigger).
  • Folding 4 tabs of article settings into one side panel is the bulk of Phase 3 UI work — metadata sub-forms per category are non-trivial.
  • Secrets in a block world — kept as the existing separate system for v1; a block-level secret type is a later nicety.
  • Per-block era/campaign scoping — could become block metadata later; for v1, inline [era]/[campaign] in text blocks is the path.
