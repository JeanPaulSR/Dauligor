1. What a reference is

A reference is an inline link from one piece of content to another Dauligor entity, stored by identity (a kind + an identifier) rather than as a hardcoded URL. Because it stores identity, it keeps resolving correctly if the target is renamed or re-slugged, and the same reference can render in three surfaces: the in-app reader, the editor, and (later) Foundry.

It replaces the current limited [ref|kind|id]…[/ref] tag, which only handles three kinds, has no authoring UI, and degrades to a plain link when edited in the visual editor.

3. The two sigils: @ vs &

Foundry uses two different inline triggers, and we'll mirror them because the muscle memory is valuable (and the live-content bridge will eventually map onto them):

SigilFoundry equivalentMeansDauligor use
@@UUID[…]{label}Link to a specific documentEntity references — spell, class, subclass, feat, item, article (things with their own page)
&&Reference[…]Link to a rules / glossary entryRule references — conditions, rules, system-page glossary entries ("look this rule up")

Both sigils open the same autocomplete; the sigil just pre-filters which family you're searching. @ → "what thing do I want to link to," & → "what rule do I want to cite."

decided Keep the split, go fully Foundry-native.

@ = entity/document references; & = rule references that reuse dnd5e's &Reference where it already covers the case — conditions, skills, abilities, damage types, weapon mastery, areas of effect, creature types, spell schools. Dauligor-specific rules (custom conditions, system-page glossary entries) get a custom enricher in the module but keep the same & grammar.

Deep-linking to a section (your &Condition=Prone idea). Foundry supports a heading anchor on a link target — @UUID[…JournalEntryPage.xyz#prone]{…} jumps to the Prone section of a page, not just the page. So &condition[prone] (or a section anchor like &rules[conditions]#prone) resolves to the conditions page scrolled to the Prone entry, and clicking does the same in-app. This is exactly the behaviour you described, and it maps onto Foundry's native anchor syntax.

4. Authoring — type, search, insert

The Foundry-style inline flow, available anywhere the rich-text editor is used:

  1. Type @ (or &) in the editor.
  2. An inline autocomplete opens. Optionally narrow by kind first (e.g. Conditions), then search by name within it (e.g. prone). Typing a name directly should also work.
  3. Pick the entity. A reference is inserted, its display text pre-filled with the entity's name.

This replaces the current "switch to Source mode and hand-type the markup" reality.

5. Editing & the display name

This is the reason we paused to spec, so it's the heart of the design:

  • Re-target / re-open: selecting an existing reference lets you re-open the picker to change its target or display text — not just delete and retype.
  • Custom display name: the visible text defaults to the entity's name but can be overridden — e.g. reference Prone but display {prone}, or reference Fire Bolt but display "a bolt of flame." (Braces {…} mirror Foundry's {label} convention.)
  • Display-name policy: default to the entity's current name at insert time; once overridden, the custom text is preserved verbatim. (We are not auto-rewriting display text on every render — too surprising.)

6. Stored syntax — Foundry grammar, semantic ids decided

Key finding: Dauligor already has a Foundry-aligned reference contract — module/dauligor-pairing/docs/reference-syntax-guide.md. It defines exactly the grammar you want. "Go Foundry-native" means adopt/extend that app-side and drop the home-grown [ref|kind|id]…[/ref] tag — not invent a third format.

Entity references (links to a Dauligor document):

@<kind>[<semantic-id>]{<display>}

@spell[spell-fire-bolt]{Fire Bolt}
@class[class-sorcerer]{Sorcerer}
@feat[feat-great-weapon-master]{Great Weapon Master}
@article[deep-shadow-cult]{the Deep Shadow Cult}

Rule references (reuse dnd5e's &Reference — see §3):

&condition[prone]                → "Prone", links to the prone rule
&condition[prone]{knocked prone} → custom display
&condition[prone]#movement       → deep-link to a section within the rule page
Critical: store SEMANTIC ids, never raw Foundry UUIDs. Foundry UUIDs (@UUID[Actor.x.Item.y]) are generated per-world at import time — the Dauligor site cannot know them. So app-authored content stores a stable kind[semantic-id]; the Dauligor reader resolves it to an in-app page, and the Foundry module translates it to a real @UUID / &Reference at import (that normalizer is already partly built). "Foundry-native" = the same grammar + module-side translation, NOT UUIDs in source text. The reference-syntax-guide already forbids raw UUIDs in authored text, so this stays.

7. In the editor decided: text, not chips

A reference is plain enricher text in the editor — you see and edit @spell[spell-fire-bolt]{Fire Bolt} (or &condition[prone]) directly, the same way Foundry's own editor shows enricher syntax. It only renders into a link + hover card in the display (the reader, §8). Your call, and the right one:

  • Edit on the fly — change the id or the {display} by just typing; no modal needed for a tweak.
  • No corruption — because a reference is text (not a TipTap link mark), nothing strips its kind/id on a Visual↔Source toggle. The earlier degradation bug existed only because the old [ref|…] rendered as a real <a> that TipTap then swallowed; plain text has no such failure mode.

The @/& insert buttons / autocomplete (§4) just drop the correct text in for you — a convenience, not a special object. Once inserted it's ordinary editable text.

Trade-off accepted: in the editor you see the raw reference syntax, not a styled chip. That's the Foundry behaviour and the price of free-text editing — the styled, hover-card rendering happens in the reader.

8. In the reader (hover + click)

Hover → a summary card

Hovering a reference shows a card:

  • Top: the reference (entity) name.
  • Body: a kind-appropriate summary. The card is a fixed size; if content overflows, the user scrolls within the card while hovered.
KindCard body source
Spell, Feat, Item, ConditionTheir description (compact enough to show directly; scroll if long).
Class, SubclassThe "brief description" box already on the class/subclass.
ArticleA brief description authored in the article editor (a dedicated short field, distinct from the full body).
Race / BackgroundBrief description (when those exist).

Implication: kinds without a short summary field today (notably articles) need one added to their editor — a "brief / hover summary" field — so the card has something concise to show instead of dumping the whole body.

Click → navigate

Clicking a reference takes you to that entity's page (in-app route). A reference whose target no longer exists renders as a distinct, non-clickable "unresolved" badge so broken links are visible rather than silently dead.

Pin & nest

The hover card carries a pin: clicking it pops the card out into a draggable, closeable window that persists across navigation (within a session) — so a DM can park several references side by side. Cards can also nest: hovering something inside a card (a truncated prerequisite, or a reference within the description) opens a child card without closing the parent — cards stack. Full visual design + the chosen feat-card layout live in docs/_drafts/feat-hover-card-concepts.html.

9. Dynamic references deferred

Beyond linking a single entity, a reference could resolve to a filtered set — e.g. "all CR 1 creatures" for a Wild Shape rules box, or "all 1st-level Wizard spells." This is a query reference: it stores filter criteria instead of one identifier, and renders as a (possibly expandable) list or a link to a pre-filtered browser.

We're deferring this, but the design above should leave room for it — e.g. a future @query[…] kind, or a saved-filter entity that itself becomes referenceable. Worth keeping in mind so the single-entity design doesn't paint us into a corner.

10. Out of scope (for now)

  • Foundry-side resolution — the enrichers + DauligorViewer that make these references work inside Foundry. That's the separate live-content-bridge work; this spec is the app side. (The @/& grammar is chosen to map cleanly onto it later.)
  • Dynamic / query references — section 9, deferred.
  • Formula references (@scale.*, @prof, etc.) — a separate system (formula resolution in roll fields), not this cross-link feature. Unchanged.

Connecting references to articles new — doesn't exist yet

Your point 4: a condition (or other rule) should be attachable to a block in an article, so a feature that references the condition can take the reader to the relevant article section for the full write-up. The vision — feature → condition reference → the article section that explains it — connects the compendium and the wiki.

Reality check: I looked for an existing "attach an article" control on the proficiency editor (AdminProficiencies.tsx) and the conditions editor (StatusesEditor.tsx) and could not find one. So this attachment capability appears not to exist yet — it's new work, not something already wired. If it lives somewhere I didn't check, point me at it.

Once articles are addressable (the article-identifier revamp) and a condition can store an attached article-id + section anchor, &condition[prone] resolves through that link to the article section. That's the systems-connection you're describing; it depends on the article revamp + an attachment field.
