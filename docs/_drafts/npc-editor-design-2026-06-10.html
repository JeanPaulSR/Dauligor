NPC / Monster Editor — design & phasing

The admin authoring surface for monster stat blocks, at /compendium/monsters/manage. Companion to the read-only browser (already shipped) and the stat-block shape study. Drafted 2026-06-10 from the understand-npc-editor research pass.

Locked decisions (user, 2026-06-10): Write access = admins + co-DMs · Action editor = full structured (embedded ActivityEditor) in v1 · Foundry import UI = manual editor first, import is a fast-follow · Derived values = XP auto-follows CR; proficiency bonus / saves / skills / passive Perception keep authored values with a "recompute?" nudge when inputs change.

1 · Architecture

A new admin page MonstersEditor.tsx mirroring ItemsEditor: left list · middle form (flat sub-tabs) · right live preview. The single biggest reuse: the public MonsterDetailPanel read-only stat block is the live preview — form state feeds it directly, no round-trip.

Sub-tabs (flat — NOT the items polymorphic MechanicsTab switch)

Monsters have no per-type dispatch, so the form is a fixed tab strip (the FeatsEditor pattern): Basics · Defenses · Movement & Senses · Actions & Traits · Spellcasting · Lore · Tags.

Save path camelCase-native

formData (camelCase keys === column names)
  → upsertMonster(id, formData)            // does NOT call normalizeCompendiumData
  → upsertDocument('monsters', id, data)   // d1.ts auto-JSON.stringify(objects)
  → /api/d1/query  → d1-proxy mutation gate (admin + co-DM)
  → INSERT … ON CONFLICT(id) DO UPDATE     // never INSERT OR REPLACE
after save: clearCache('monsters') + bump previewBustKey
The #1 footgun. upsertItem/upsertFeat run normalizeCompendiumData (snake↔camel). Monsters are camelCase with no alias layer — running that would map every key to a non-existent snake_case column and silently drop the data. upsertMonster writes keys verbatim. JSON columns pass as objects (never pre-stringify → double-encode); confirm every monster JSON column is in d1.ts's jsonFields allowlist (the read side already parses them, so this is likely complete — verify in P1).

Write gating — admin + co-DM

monsters is currently staff-writable (not in PROTECTED_WRITE_TABLES). Per the decision, add an admin-or-co-DM gate. Implementation in P1: a dedicated proxy branch for monsters writes calling the admin-or-co-DM role check (the same role set campaign management uses), rather than the generic staff path. No route-level AdminOnly wrapper — the Sidebar link visibility + proxy gate match the items/feats convention; co-DMs reach the editor via the public list's admin affordance.

Derived-value UX (per decision). xp silently tracks CR. proficiencyBonus, each proficient save/skill bonus, and passivePerception are authored values that stay as-is; when a driving input changes (CR, an ability score, a proficiency toggle) the affected field shows a subtle "recompute to N?" affordance the author can accept or ignore. This protects hand-tuned imported stat blocks (the whole 1001-creature corpus).

4 · Risks & guardrails

  • camelCase write path — never reuse upsertItem wholesale; upsertMonster writes camelCase verbatim.
  • JSON columns — pass objects, never pre-stringify; verify all monster JSON cols in jsonFields.
  • foundryData + bypasses must survive edits — treat foundryData as read-only round-trip state (merge, never clobber); losing bypasses breaks "from nonmagical attacks" and Foundry re-export.
  • Shared widgets are compendium-editors' territory — ActivityEditor, MarkdownEditor, EntityPicker, ImageUpload, TagPicker, CompendiumEditorShell are import/read-only on this branch. If one needs a change (e.g. ActivityEditor for monster context), write a cross-branch handoff — don't edit it here. Prefer bespoke for the saves/skills grids rather than bending the items-shaped ProficienciesEditor.
  • Column renames hit the module export contract — if any monster column is renamed, flag schema-crosswalk.md + handoff.
  • tsc baseline = 2 (post the asChild fix). New code must not exceed it; verify headless (tsc + route 200 + module transform; full render needs browser auth).
  • main = production — nothing pushed without explicit permission; migrations local-first.
