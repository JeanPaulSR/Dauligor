Foundry Reference Enrichers — Deep Dive & Verdict

Research notes for the Dauligor reference system. Authored 2026-05-26. Safe to delete after the design decisions land in canonical docs.
Grounded in Foundry 13.351 core + dnd5e 5.3.1 source.

TL;DR

Foundry's text-enricher system is one of its best-designed extension points. Do not overwrite it. Instead, layer Dauligor's semantic references on top:

  • Use @UUID[…] as the rendering substrate for every cross-document link — Foundry already gives us tooltip + click + drag-and-drop for free.
  • Register Dauligor-flavoured enrichers via CONFIG.TextEditor.enrichers.push(…) for any syntax we want to expose (e.g. @article[slug], @condition[prone]).
  • Reuse dnd5e's &Reference[…] wherever it already covers the case (conditions, skills, abilities, damage types, weapon mastery, area-of-effect, creature types, spell schools).
  • App-side: extend src/lib/referenceSyntax.ts and BBCodeRenderer to recognise the same syntax and render hover-chips.

All four of the requested features are extensions of existing systems. None require overwriting Foundry behaviour. The deepest piece of new work is a "system page" article type on the app side.

3. Foundry Core Enrichers

3.1 @UUID[uuid#hash]{Label} — Content Link

The bread-and-butter cross-document reference. Output is an <a class="content-link"> that:

  • Shows a tooltip on hover with the document type label.
  • Opens the document sheet on click.
  • Is draggable onto sheets / hotbar / scenes.
  • Renders broken with a strike-through icon if the UUID doesn't resolve.

The #hash suffix targets a page anchor or table-of-contents heading inside a Journal Entry — the basis for sub-section addressing.

@UUID[Compendium.dnd5e.classes.Item.RnpHCwlNzjmcZdcb]{Sorcerer}
@UUID[Item.ovD7tcXrAKbazG3T#font-of-magic]
@UUID[.Activity.dnd5eactivity000]{relative activity ref}

Relative UUIDs (.foo, ..bar) resolve against options.relativeTo. Dauligor's existing entity refs (@feature[id], @option[id], etc.) all compile down to @UUID[…] at module-import time.

3.2 @Embed[config]{Caption} — Document Embed

Pulls the rendered content of another document into the current text. The most powerful enricher for the "include the full thing" use case — spells embedded inside class features, statblocks inside adventure prose, journal pages inside other journal pages.

Recursion cap: 5 levels deep. Beyond that the inner embed renders as a broken-link warning.

3.3 [[/cmd formula]]{Label} — Inline Rolls

Foundry core only handles the generic-roll family: roll/r, gmroll/gmr, blindroll/blindr, selfroll/selfr, publicroll/publicr. The [[/attack]] / [[/save]] family is dnd5e — see §4.

ShapeBehaviour
[[1d20 + @prof]]Eager — evaluated at enrichment time. Result is baked in.
[[/r 1d20 + @prof]]{Save}Deferred — clickable, rolled at click against the sheet's getRollData().
[[/r 1d6 + 2]]]Triple ] escapes a literal ] inside the formula.
[[/gmr 1d100]]GM-only roll mode.

4. dnd5e System Enrichers

Registered via three patterns at dnd5e.mjs:20126:

(a) /\[\[\/(?<type>attack|award|check|concentration|damage|heal|healing|item|save|skill|tool)
       (?<config> .*?)?]](?!])(?:{(?<label>[^}]+)})?/gi
(b) /\[\[(?<type>language|lookup) (?<config>[^\]]+)]](?:{(?<label>[^}]+)})?/gi
(c) /&(?<type>Reference)\[(?<config>[^\]]+)](?:{(?<label>[^}]+)})?/gi

Config is space-separated key=value with "quoted strings" supported. Bare positional values land in config.values and each enricher applies its own heuristic to interpret them. Numerics and true/false auto-coerce. Every config carries _input (the raw match) for diagnostic warnings.

The killer feature is activity=<id> with implicit fallback. Every action enricher will auto-find the first activity of the matching type on options.relativeTo when no args are passed. So [[/attack]] inside a longsword feature description pulls the longsword's first attack activity. This is the "pull info from within features or spells" mechanic in its purest form.

KeyPurpose

4.3 [[/damage …]], [[/heal …]]

Multiple damage parts in one tag separated by &:

[[/damage 1d6 fire & 1d6 cold]]
[[/damage 2d6 type=bludgeoning average=true]]
[[/heal 2d4 + @mod]]

type=fire/coldMulti-type renders "X or Y".

extended or format=extended"Hit: 7 (2d6) bludgeoning damage." statblock line.

activity=Pull formulas from activity's getDamageConfig().

4.4 [[/item <ref> activity=<ref>]]

Cross-item activity prompt — the literal "pull info from another feature/spell" use case. Resolution order for <ref>:

  1. Parses as a UUID? → resolve directly.
  2. Matches an item ID on options.relativeTo.parent.items? → use that.
  3. Fall back to item name lookup at click time on the speaker's actor.
[[/item "Heavy Crossbow" activity=Poison]]
[[/item Compendium.dnd5e.items.Item.crossbowId]
[[/item .Item.xxx]                                 # relative

4.5 [[/award]], [[lookup]], [[language]]

TagBehaviour
[[/award 100gp 50sp 20XP each]]Currency/XP grant chip with trophy icon and click-to-award action. each multiplies per-token.
[[lookup @path]]{fallback}Reads roll-data path as live text in prose. Style modifiers: capitalize, lowercase, uppercase. activity=<id> switches context to that activity's rollData.
[[language deep]]{Deep Speech}Cosmetic language tag — doesn't fetch a document.

4.6 &Reference[type=key flags]{Label} — the Glossary Enricher

Read this carefully — this is the closest existing analogue to your "system page" idea.

Resolves through CONFIG.DND5E.ruleTypes. Each type has a registry of UUIDs pointing into compendium journal pages (dnd5e's "rules" compendium, conditions glossary, etc.). On hover the link shows a rich tooltip; on click it opens the journal page.

TypeKey vocabularyBacked by

conditionunconscious, prone, grappled, …conditionTypes
abilitystr/dex/…enrichmentLookup.abilities
skillacr/ath/…enrichmentLookup.skills
damagefire/cold/…damageTypes
spellSchoolevo/abj/…enrichmentLookup.spellSchools
spellComponent / spellTagvocal/somatic/material/…itemProperties
areaOfEffectcone/sphere/line/…areaTargetTypes
creatureTypeaberration/beast/…creatureTypes
weaponMasterycleave/flex/graze/…weaponMasteries

Flags

  • apply=false — suppress the apply-status button on conditions.
  • Bare value with no type= → autodiscovers across all types (e.g. &Reference[unconscious] finds it as a condition).

Why this matters for use case #3: the implementation is one parent JournalEntry per system (Conditions, Rules, Weapon Masteries) with one JournalEntryPage per subsection (Prone, Stunned, etc.), and a flat { slug: uuid } lookup table at CONFIG.DND5E.<system>Types. The slug becomes the addressable key.

5. Your Four Use Cases — Verdicts

Legend: Reuse use Foundry/dnd5e as-is · Extend register a new custom enricher / new entity kind · Build net-new app-side construction (no Foundry analogue) · Overwrite replace Foundry behaviour (avoid)

UC #1 — Article cross-references with hover preview + click-to-open

Extend Build (app)
"Being able to reference articles from other articles with good UI pop-ups that have a small description when hovered, and the ability to open the full article when clicked. In both app and Foundry as a window."
What Foundry already gives us
  • @UUID[JournalEntryPage.xxx]{Label} renders a content link with built-in hover tooltip + click-to-open journal sheet.
  • The tooltip is generic by default (just the document type label), but Foundry v13 supports rich HTML tooltips via the data-tooltip-html attribute — RuleJournalPageData.richTooltip() at dnd5e.mjs:77098 is a working example.
  • Drag-and-drop onto sheets and the hotbar works for free.
What we need to build
  • App side: new BBCode tag (or extend @article[slug]{label} in the existing entity-ref grammar). Renderer attaches an onMouseEnter handler that fetches a short summary and shows a hover card. Click navigates to /wiki/<slug>.
  • Module side: add article to the entity kinds in reference-service.js's ENTITY_REFERENCE_PATTERN. Compile to @UUID[JournalEntryPage…]. Optionally enhance the rendered anchor with a data-tooltip-html attribute carrying the summary so the Foundry hover matches the app's.

Recommendation: extend, don't overwrite. Foundry's @UUID rendering pipeline gives us 80% of the work for free. The hover-with-summary part is purely additive — we set a custom data-tooltip-html when we render the link. App side gets a parallel hover-card component.

UC #2 — Reference items from anywhere

Extend
"Being able to reference items from anywhere in our application, or features, classes, spells, etc."
Where we already are
  • Dauligor's entity-reference grammar already covers @class, @subclass, @feature, @option, @source (see src/lib/referenceSyntax.ts + module/dauligor-pairing/scripts/reference-service.js).
  • Each compiles to @UUID[…] at module import time via findEntityDocument().
  • Foundry's @UUID + fromUuid() handles every document type universally on its end.
What's missing
  • New entity kinds to add: @spell, @feat, @item, @weapon, @armor, @tool, @race, @background.
  • matchesEntityKind() needs branches for each new kind. findEntityDocument() already works generically — it keys off flags.dauligor-pairing.sourceId.
  • App side: extend ENTITY_REFERENCE_PATTERN in referenceSyntax.ts with the new kinds.

Recommendation: this is ~90% built. Each new entity kind is one new resolver case on the module side and one new entry in the reference sheet vocabulary on the app side. Same architecture, no Foundry override.

UC #3 — Articles with addressable subsections (system pages)

Reuse Extend Build (app)
"Being able to create an article, and then being able to create small subsections in that article that can be referenced. For example &Reference[condition=prone] could have a condition page with all of the conditions referencable from that page. Probably with a unique System page."
How dnd5e already does this
  • One JournalEntry = one "system" (e.g. Conditions).
  • One JournalEntryPage per subsection (Prone, Stunned, …).
  • Each subsection has a stable slug.
  • CONFIG.DND5E.conditionTypes.prone = "Compendium.dnd5e.rules.JournalEntry.….JournalEntryPage.…".
  • &Reference[condition=prone] looks up that UUID → renders link to that exact page → for conditions, also stamps an "apply status" action button.
What we'd add
  • App side: a "System Page" article type. Pattern: a parent article (e.g. Conditions) with each subsection as a child article. The parent renders a glossary view (all children listed in one scrollable page). Each child has a stable slug addressable from elsewhere.
  • App side: new entity-reference forms keyed by system: @condition[prone], @rule[cover], @area-effect[cone], etc.
  • Module side: for systems Foundry already covers (conditions, skills, abilities, damage types, weapon masteries, area-of-effect, creature types), @condition[prone] compiles to &Reference[condition=prone] — we ride dnd5e's existing glossary entirely.
  • Module side: for Dauligor-specific systems (custom rules, homebrew conditions), compile to @UUID[…] pointing at the user's authored system pages exported as JournalEntry+JournalEntryPages.

Recommendation: Foundry already has the exact architecture you want — one parent doc + many addressable children + a slug-to-UUID lookup table. The work is (a) replicate the structure on the app side as a "system page" wiki article type, and (b) extend the module resolver so our semantic refs compile to &Reference[…] when Foundry has the system, and @UUID[…] otherwise. No overwrite.

UC #4 — Admin proficiency hover + click → system page

Reuse Extend
"Being able to hover an admin proficiency, but when clicked being pushed towards its system page where it can be explained in detail what that skill is used for."
What Foundry already gives us
  • &Reference[skill=acr] already does this for all 18 5e skills — renders a link to the Acrobatics journal page in the dnd5e rules pack.
  • Same for abilities (ability=str), damage types, conditions, area-of-effect, etc. (see §4.6).
  • Hover tooltip is built in.
What's missing
  • App side: a second mode for the bare-name shortcuts. Currently @stealth resolves to the formula path @skills.ste.total. We need an entity form — @skill[stealth]{Stealth} — that resolves to a link to the Stealth system page on the app side and to &Reference[skill=ste] on the Foundry side.
  • App-side: when the cursor sits on an admin-proficiencies row, the click handler navigates to the system page (the Skills system article).
  • For homebrew tools / weapons / armor not in the 5e SRD, we extend CONFIG.DND5E.enrichmentLookup.tools at module init time with the UUIDs of our own system pages.

Recommendation: nothing to overwrite — we add the entity-form variant on the app side and lean on dnd5e's existing &Reference on the Foundry side. For homebrew proficiencies the module just adds entries to enrichmentLookup at init time pointing at our authored system pages.

6. Overall Recommendation

Strategy: layer, don't replace

Across all four use cases:

  • Reuse 4 places — existing Foundry / dnd5e behaviour covers the case.
  • Extend 4 places — one new resolver branch, one new compendium pack entry, or one new entity kind.
  • Build (app) 3 places — net-new app-side work (hover-card component, "system page" article type, entity-mode variant of proficiency shortcuts).
  • Overwrite 0 places. Avoid replacing Foundry's enricher pipeline — it would break interop with dnd5e and any other module.

Architectural shape

App side

Extend referenceSyntax.ts with new entity kinds + the entity-mode variant of bare shortcuts. Add a "system page" wiki article type. Render every reference as a hover-card chip.

Module side

Extend reference-service.js's findEntityDocument() + matchesEntityKind() with the new kinds. For systems Foundry already handles, compile to &Reference[…]. For Dauligor-only systems, compile to @UUID[…] at our authored journal pages.

Foundry side

No core changes. We benefit from the entire enricher pipeline (content links, embeds, inline rolls, tooltips, drag-and-drop) for free. For homebrew proficiencies we extend CONFIG.DND5E.enrichmentLookup.* at init.

Where the "system page" model fits

The user-authored "system pages" on the app side map cleanly to Foundry's JournalEntry with many JournalEntryPages pattern. Each subsection (one Condition, one Skill, one Damage Type) is a separately-addressable child. We export the parent + children together when we sync the world's content, and the slug-to-UUID table becomes our extension of CONFIG.DND5E.enrichmentLookup.

What we don't do

  • Don't override TextEditor.enrichHTML or its pipeline stages. Modules that do this conflict with each other and with dnd5e itself.
  • Don't reinvent @UUID. Every Foundry document already has a stable UUID; the renderer already handles tooltip + click + drag for free.
  • Don't write a parallel reference resolver inside Foundry. Continue using reference-service.js as the bridge — Dauligor semantic refs in, Foundry-native paths/links out.

7. Footnotes & Gotchas

The "@-paths" naming collision — @abilities.cha.mod is not an enricher

Inside a formula field, @abilities.cha.mod is roll data substitution — handled by Roll.replaceFormulaData, not the text enricher. Inside a prose field, the same value renders only if wrapped: [[lookup @abilities.cha.mod]]. Dauligor's existing dual output mode (formula vs text) already handles this distinction.

Module enricher ordering — first match wins per pass

If two custom enrichers register patterns that match the same substring, the order they were pushed to CONFIG.TextEditor.enrichers determines who wins. dnd5e pushes its three patterns at init time. Any module that loads later and registers a colliding pattern will be silently shadowed. Practical implication: namespace any Dauligor enricher with a unique prefix (@dauligor:… or similar) to avoid collisions.

The relativeTo option carries virtually all context

Every action enricher's "auto-find an activity" behaviour, every relative UUID resolution, every getRollData() fallback — all of these go through options.relativeTo. When we render Dauligor content in our own sheets we should always set this. When Foundry renders journal pages and item descriptions natively, it sets it for us.

Drag-and-drop is free — but only on real @UUID links

The draggable="true" + data-link attributes on a content link wire it up to Foundry's drag handlers. If we render our own enriched output as a plain <a>, drag won't work. Easiest fix: always compose Dauligor references down to a real doc.toAnchor() call so we inherit the standard wiring.

Tooltip flavours: simple vs rich HTML vs custom

Foundry v13 supports three flavours:

  • data-tooltip="plain text" — default tooltip behaviour, plain text only.
  • data-tooltip-html="<p>…</p>" — rich HTML tooltip rendered inside the standard tooltip frame.
  • Custom tooltip via the TooltipManager API — full control over content and lifecycle. dnd5e uses this for spell item-tooltips and rule rich-tooltips.

For UC #1 the simplest path is to set data-tooltip-html on the rendered anchor with a short HTML summary. We can author the summary on the app side and ship it with the export.

Inline rolls and the prosemirror editor

The ProseMirror editor (default in v13, replacing TinyMCE in v14) understands content links, embeds, and inline rolls when they're authored in a journal page. Custom enrichers display correctly in read-only rendered output but the editor renders them as plain text while editing. This is fine for Dauligor — authoring happens on the web app, not inside Foundry's editor.

The "what about @Embed for cross-feature inclusion?" question

@Embed is powerful but easy to abuse. When a feature description embeds another feature, the rendered HTML grows quickly and the recursion cap (5 levels) becomes load-bearing. Recommended uses for Dauligor:

  • Embedding a spell's description inside a class feature that grants it ("You learn @Embed[Item.fireball details]").
  • Embedding a system-page subsection inline ("see @Embed[JournalEntryPage.prone inline]").
  • NOT embedding entire characters or large NPCs — use @UUID there.
