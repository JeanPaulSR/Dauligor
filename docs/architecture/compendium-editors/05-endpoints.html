05 — Endpoints

Pages Functions and the api/_lib/_*Export.ts server-side composition libraries.
stub Foundry-side consumption explicitly out of scope
Foundry-module-side consumption is out of scope for this reference. Another agent owns module/dauligor-pairing/. This section stops at the wire — what the server emits, why it's shaped that way, what gates it — and does not document how the Foundry importer / viewer ingests it.

Planned outline

  1. The runtime split. Pages Functions (functions/api/**) run on Cloudflare Pages; the Express server (server.ts) is the local-dev mirror. Some endpoints are Pages-only (/api/module/*); local dev requires npx wrangler pages dev to test them. See docs/platform/runtime.md.
  2. Shared server libraries. Drift-managed sibling pairs (per AGENTS.md rule #5):
    • src/lib/classExport.ts ↔ api/_lib/_classExport.ts
    • src/lib/classProgression.ts ↔ api/_lib/_classProgression.ts
    • src/lib/referenceSyntax.ts ↔ api/_lib/_referenceSyntax.ts
    • src/lib/requirements.ts ↔ api/_lib/_requirements.ts
    Why the drift exists (Vercel serverless bundling history) and the contract for keeping them in sync.
  3. Generic D1 proxy. functions/api/d1/query.ts + api/_lib/d1-internal.ts + d1-proxy.ts. Why every queryD1 call goes through this proxy rather than the editors talking to D1 directly. The proxy gate at api/_lib/d1-proxy.ts that refuses non-bump writes to system_metadata (per the deprecation comment on setSystemMetadata).
  4. R2 image proxy. functions/api/r2/[action].ts — the four actions (upload / list / signedUrl / delete). How ImageUpload / Image Manager / class+feat+item icon uploads route through here. Why R2 isn't accessed directly from the client.
  5. Foundry module bundle — the big one. functions/api/module/[[path]].ts dispatches by path segment. Per-source / per-class / per-spell-list / per-feat / per-item endpoints, plus the queue-backed full-bundle export (api/_lib/module-export-pipeline.ts + module-export-queue.ts + module-export-store.ts). For each route:
    • URL shape
    • Auth gate
    • Composition library / function
    • Shape of the emitted payload (link to the contract doc in module/dauligor-pairing/docs/ — not duplicated here)
    • Caching behavior (the staleness story)
  6. Content proposals. functions/api/proposals/[[path]].ts + functions/api/admin/proposals/[[path]].ts + api/_lib/proposals.ts + cascadeStrategies.ts. From the editor's perspective, not the proposal-system internals.
  7. The auth / profile / favorites / characters / admin endpoints. Briefly, because they don't directly serve the compendium editors — the editors call fetch('/api/me') for profile, the favorites endpoints from the read-only browsers, the characters endpoint from the builder/sheet. Mostly out of scope; documented to a depth that lets the reader understand what the editor's surrounding chrome calls.
  8. Auth + permissions throughout. api/_lib/firebase-admin.ts + api/_lib/permissions.ts — how every endpoint gates by role (admin, co-dm, lore-writer, trusted-player, user). The requireAdminAccess idiom.
