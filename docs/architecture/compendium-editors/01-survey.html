The system in one paragraph

The Archive is, at its heart, a structured database of D&D 5e content — spells, feats, items, classes, subclasses, features, races, backgrounds, facilities, plus the supporting vocabulary (tags, sources, proficiencies, scaling) and the wiki/lore on top. Admins author it through web forms; players read it through public list pages; trusted players propose changes through a review queue. The same data is then exported via /api/module/* for the Foundry pairing module to consume at game time. The compendium editors are the authoring half of that loop — the part where the data goes in.

Three roles, three URL families

Everything I see in the routing falls into one of three buckets:

Read — the public browsers

/compendium/<entity>

Every signed-in user can browse spells, feats, items, classes, races, backgrounds, facilities, etc. These pages share a master-detail browser shell (compact list on the left, detail pane on the right, filter bar on top). They're read-only and don't know anything about editing.

Edit — the admin manager pages

/compendium/<entity>/manage or /edit/:slug

Admins author the same content via manager pages. For the catalog entities (spells, feats, items) the manager looks intentionally like the public browser — same list rhythm, same filter bar — with an editor pane and admin actions on top. For classes and a few special editors (sources, tag manager, weapons/armor admin tools) the layout is bespoke.

Propose — the same editors, wrapped

/proposals/edit/<entity>

Trusted players (and roles below admin) suggest changes through "proposal mode" — the exact same editor components, wrapped by ProposalEditorWrapper. The wrapper queues writes locally, the proposer reviews their bundle, and the bundle goes to admins for accept / reject / cascade decisions. From the editor's point of view, the only difference is that the save handler is swapped — it doesn't know it's in proposal mode.

Four shapes of compendium content

Looking at what gets authored, I see four families of thing — each shaped slightly differently:

Family What it is Examples
Catalog entities The big enumerated content. Each has identity (name, identifier, source), a description, mechanical fields, and Foundry-shaped automation (activities + effects). spells, feats, items, classes, subclasses, features, facilities, races, backgrounds
Taxonomies Small "what's allowed in the game" lists. Typically name / identifier / order / description, sometimes a free-text category. skills, tools, weapons, armor, damage types, languages, attributes, condition categories, tool categories, weapon categories, armor categories, weapon properties
Special structures Things that don't fit the catalog mold — usually because they have a stronger shape or relate other entities. unique option groups (Metamagic, Invocations), scaling columns (cantrip dice, sneak attack), tags + tag groups, sources, spell lists, spell rules
Wiki / lore Hierarchical narrative content — campaign lore with DM-secret blocks, metadata (characters, locations, organizations, deities), and a separate set of junction tables. lore articles, secrets, character/location/organization/deity meta (currently owned by the system-applications branch)

Five editor patterns, in plain language

Instead of one generic CRUD form, the codebase has converged on five recognizable patterns, each tuned to a kind of authoring problem. I've kept the naming compatible with the existing patterns doc (A–E) but the assignments are reconciled against what the editor files actually do today.

# What it is, in human terms When to reach for it Who uses it today
A — Quick form Name + identifier + source + description + a few entity-specific fields. Renders via a generic shell that takes a callback for "the unique fields go here." Flat table, no children to track, no relational complexity. Facilities
B — Bespoke editor Whatever shape the entity needs — full custom page, hand-rolled state, custom layout. Says nothing in common beyond "doesn't fit the shells." Entity has structure that doesn't reduce to a flat form (e.g. classes), unusual UX needs (multi-step, multi-tab with non-uniform contents), or it predates the shells and hasn't been migrated. Classes, subclasses, sources, tag manager, tag group editor, unique option groups, the admin mechanic editors (weapons, armor, statuses, spellcasting types, multiclass chart)
C — Taxonomy Inline-edit list-of-rows. Looks like a styled spreadsheet with an "add row" button and a modal for the long fields. Small reference lists. Editor shape is identical across instances; only the table changes. Skills, tools, all the admin proficiencies tabs (tool categories, weapon categories, weapon properties, armor categories, language categories, damage types, attributes, languages), status condition categories
D — Relational entity Parent row plus multiple child tables (metadata, junctions) written atomically as one batched D1 transaction. Anything where partial writes would leave the database inconsistent — usually parent-with-children that need to be created/updated/deleted in lockstep. Lore articles + the seven supporting tables. Owned by the system-applications branch; referenced here for completeness, not deep-dived.
E — Catalog manager Three-pane master-detail: compact list on the left (hundreds of rows, virtualised), tabbed editor in the middle, read-only detail preview on the right. Filter bar on top, super-tabs for Editor vs Tags. Catalog-shaped entities where the list of "all rows" is itself an authoring tool — you scroll dozens of records to find the one to tweak. Spells, feats, items (Races + Backgrounds wrap the feats editor with a discriminator)

My read on the trajectory: the codebase has gravity toward Pattern E for catalog entities. The legacy Pattern A shell is literally named DevelopmentCompendiumManager on purpose — you confirmed it's the compendium manager used during development, and Facilities (its only remaining consumer) will be replaced with its own thing once everything else is cleaned up. At that point the legacy shell has no consumers and can be removed. The new Pattern E shell (CompendiumEditorShell) is where spells, feats, and items live now — the patterns doc still describes spells/feats/items as Pattern A, but that's drift.

The shared building blocks

The thing I take away from reading the editor files: most editors are thin assemblies of a small set of shared widgets. Once you know the widgets, you know most of what's going on inside any one editor — the bespoke code is mostly wiring (which widgets in which tabs, with which lookup pools) plus a small core of entity-specific fields.

The widgets I see used most often:

Widget What it does, conceptually Mounted by
Advancement editor Authors compose a level-by-level progression timeline — hit points, ability score improvements, item grants, item choices, subclass markers, scale values, size, spell grants, and "bump uses" hooks. Same component, different per-editor behaviour selected via a context prop. Classes, subclasses, feats, items, races, backgrounds (via the feats wrapper)
Activity editor Authors define the automation behavior the Foundry side will execute — per-type forms (attack, save, check, heal, damage, utility, spell, enchant, forward, summon, transform, cast). Vocabulary kept aligned with Foundry's so the export round-trips. Classes (via features), subclasses (via features), feats, items, spells, unique option items, facilities
Active Effect editor Authors define the persistent modifications the entity applies when on the actor sheet — mirrors Foundry's ActiveEffectConfig with the same modes, priorities, and change keys. Same as the activity editor.
Requirements editor Structured prerequisite tree (And / Or / Xor over level / class / ability / proficiency / feature / spell / etc.) with a free-text fallback for narrative gates that don't fit the tree. Feats, unique option items
Tag picker Hierarchical chip selector with a "subtag implies parent" invariant. Used for categorising entities; the same vocabulary drives the public-browser filters. Spells (inline variant), feats, items
Scaling columns panel The per-class / per-feat / per-item table of values that scale with level (cantrip dice, sneak attack, etc.). Both authoring and referencing flow through this. Classes, subclasses, feats, items
Filter bar Search input + multi-axis filter modal + reset + result count + trailing actions slot. Same component on public browsers and admin managers. Every Pattern E manager, every public browser, plus some bespoke editors
Detail panel Read-only preview of the entity (icon + name + mechanical summary + description). The Pattern E managers mount the public-browser's detail panel inside their right pane so admins see what end-users see. Spells, feats, items (one panel per entity, reused by both the browser and the manager)

Data flow at a glance

Every save goes through the same pipeline:

form state
(camelCase) → normalize
camel→snake → upsert helper
per-entity → D1 row
snake_case → cache invalidation → foundation bump

And the reverse on read:

D1 row → queryD1
auto-parses JSON cols → denormalize
snake→camel → form state

Sharp edges worth flagging at the conceptual level:

  • Two-way camel/snake dance. Form state is camelCase by convention; D1 rows are snake_case. A normalise / denormalise pair handles the swap at the boundary. The patterns doc flags this as a future cleanup (pick one and stop dancing), but for now it's the model.
  • Cache is two-tier. In-memory cache for the session, plus sessionStorage for tables tagged "persistent" (sources, skills, weapons, armor, spells, etc.). A global "foundation update" timestamp gets bumped on writes and clients poll it every 30 seconds — when it changes, everyone wipes their caches.
  • INSERT OR REPLACE is forbidden. D1 enforces foreign keys, and INSERT OR REPLACE deletes the row before reinserting — which cascades to children. Editors use ON CONFLICT(id) DO UPDATE instead. There's a documented data-loss incident from before this rule was added.
  • JSON columns auto-parse on read, but editors have to handle "already parsed." The queryD1 helper auto-parses an allowlist of JSON columns; remap code in editors has to use typeof X === 'string' ? JSON.parse(X) : (X ?? default) so an already-parsed value passes through. Forgetting the passthrough silently empties the field.

Server side at a glance

Two surfaces of server code:

Pages Functions — the production runtime

functions/api/**. Each file under there is a Cloudflare Pages Function. Some serve the client app (/api/d1/query, /api/r2/*, /api/lore/*, /api/proposals/*, etc.); one in particular serves the Foundry pairing module (/api/module/*).

Express — the local-dev mirror

server.ts. Runs alongside Vite in dev (npm run dev), proxies to a local Cloudflare worker (wrangler dev in worker/) for D1 + R2 simulation. Mirrors most Pages Functions for local-dev parity, but not all — the /api/module/* family is Pages-only, so to test it locally you need npx wrangler pages dev instead.

A handful of shared libraries live in both src/lib/ (client) and api/_lib/ (server) as drift-managed sibling pairs — classExport, classProgression, referenceSyntax, requirements. The original reason was a Vercel bundler limitation; the pattern is preserved on Pages because the contract is clean and the sibling-pair scoping keeps the server bundle small. Any change to one has to land in both.

The biggest server-side surface for our purposes is /api/module/*. That's the wire to the Foundry pairing module. The composition libraries (_classExport, _featExport, _itemExport, _spellExport, plus the per-source spell/feat lists, the spell-list resolver, the filter helpers, the tag catalog, the queue-backed full-bundle pipeline) turn D1 rows into the JSON shape the Foundry side expects. What happens past the wire — how the module consumes the payload — is someone else's job, and this reference set explicitly does not document it.

Findings — what surprised me

Things that stood out as I read the code against the existing docs:

F1 — The patterns doc is mid-migration

docs/architecture/compendium-editor-patterns.md documents an earlier state where spells, feats, and items used a generic CRUD shell (Pattern A). They've migrated to a dedicated shared shell (Pattern E in this reference). The doc still names them as Pattern A. The migration happened; the doc didn't follow. I've called out six concrete drift items in the reference page — if you sign off on the pattern map here, I'll reconcile the existing patterns doc as part of task #5.

F2 — The advancement editor is the heaviest piece of shared authoring

AdvancementManager is mounted by classes, subclasses, feats, and items. It varies its behaviour by a context prop ("class" / "subclass" / "feat" / "race" / "background" / item-as-bump-author). Changes to it ripple across the editor surface. If there's one component to be careful with when touching the editor side, it's this one.

F3 — Pattern B is "everything that didn't fit elsewhere"

Sources, tag manager, classes, subclasses, the admin mechanic editors — they share nothing structurally beyond "they're bespoke." Calling something Pattern B tells you very little. ClassEditor in particular is its own beast: advancements, features, scaling, proficiencies, spellcasting, multiclass requirements all on one page.

F4 — Races and backgrounds aren't yet first-class

They live in the feats table with a feat_type discriminator. The "race editor" and "background editor" are 20-line wrappers around the feats editor that pass a scopeFeatType prop. The doc inside those wrappers explicitly anticipates a future migration to dedicated tables.

Confirmed: this is intentional placeholder behavior, and Races + Backgrounds will be promoted to their own tables as part of the current cleanup pass (see roadmap step 1). The wrappers are the seam where that swap happens — swap their bodies, leave the routes intact.

F5 — Proposal mode is layered on, not baked in

The proposal-edit URLs wrap the same editor components with ProposalEditorWrapper. The editor mostly doesn't know it's in proposal mode — the save handler is swapped via context. Almost every editor file imports a handful of proposal-aware hooks (useProposalAccumulator, useDraftedEntityIds, ...), but they're conditional pass-throughs. Clean separation; not a special-case spaghetti.

F6 — The browsers and the managers share components on purpose

The Pattern E manager's right-pane preview is literally the same React component the public browser uses for its detail panel. The filter bar is the same. The list-row layout is intentionally aligned. Visual parity isn't a happy accident — the editors are designed so admins navigate them the same way end-users navigate the browsers.

Your clarifications

Captured here for posterity; the relevant findings above are updated to reflect what's now confirmed. Each answer also informs how the rest of the reference set gets written.

A1 — "New table" vs "discriminator": new table for new functionality

Principle: prefer a new table when the functionality is genuinely new. Extracting later is harder than starting separated — one table touched, not one table excavated. The cases that look like discriminator use are deliberate:

  • Races / Backgrounds living in feats — placeholders. They will get their own tables once prioritized (see roadmap step 1).
  • Statuses + Condition Categories — already separated. Categories type the statuses ("what type of condition is this status?"). Two tables; intentional.

A2 — DevelopmentCompendiumManager is on a glide path to retirement

The name is literal — "compendium manager used during development." Facilities (its only remaining consumer) will be replaced with its own thing once everything else is finished (see roadmap step 4). At that point the legacy shell has no consumers and can be removed.

A3 — /api/module/* contract is stable

May gain new endpoints over time, but existing endpoints should rarely change unless there's a real problem or convention issue. Designed for "get every source then get every one of their resources" as the optimized happy path. Treat as a stable wire for the purposes of this reference; the new endpoints landing on system-applications (articles, system pages) are additions, not revisions.

A4 — Class cascade-delete: yes, but bounded

A class delete should remove:

  • Its subclasses
  • Its class features (the rows in features with parent_id ↔ this class)
  • Its advancements (the advancement rows authored on the class, including those on its subclasses)

It should not remove anything those advancements grant. Worked example: a class → subclass → advancement that grants Warlock Invocations. Deleting the class deletes the subclass, the features, and the advancement row itself. It does not delete the Invocation option items — those are independent compendium content and might be referenced by other things.

Low priority for now — no classes are being deleted in practice. The rules above are documented now so they're clear when someone gets to it.

A5 — Proposal mode is not done

The editor surface still needs work for proposal mode to be considered complete. Not super high priority right now — not blocking other work, but worth surfacing as a real backlog item when the editor cleanup pass reaches each editor.

Where we're going next (per your direction)

The current priority order. Documentation in this reference set is written with this priority in mind — the Pattern E editors get the deepest walkthroughs first.

Step What it covers Notes
1 Finish the main D&D character editors using Pattern E:
  • Backgrounds — clean up view + editor; promote to its own table
  • Races — clean up view + editor; promote to its own table
  • Items — clean up view + editor (already in Pattern E)
 Per A1, Backgrounds and Races likely get their own tables in this step. The RaceEditor / BackgroundEditor wrappers are the seam where the swap happens — routes and sidebar entries stay put.
2 Clean up Class and Subclass view + editor. These stay bespoke (Pattern B). They're innately complex (advancements + features + scaling + proficiencies + spellcasting + multiclass on one page) and Pattern E would be the wrong fit. Cascade-delete (A4) is a known follow-up here but low priority.
3 Clean up the shared widgets the editors mount. Advancement editor, activity editor, active effect editor, requirements editor, tag picker, scaling columns panel, filter bar, and the rest of the building blocks. Touched here once, propagates across every consuming editor.
4 UI polish + specific features for edge cases. Includes Facilities' replacement (per A2) and retirement of the legacy DevelopmentCompendiumManager shell. Also where proposal-mode completeness (A5) and class cascade-delete (A4) land if they haven't been picked up sooner.

Not a commitment to ship steps in strict sequence — just the framing for prioritisation. Items out of this list (lore / characters / admin pages / wiki) sit outside this priority chain and evolve on their own cadence.
