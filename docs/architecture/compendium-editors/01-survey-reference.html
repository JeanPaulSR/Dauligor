Drift items found against existing docs

Reading the editor source against the existing docs/architecture/compendium-editor-patterns.md surfaced several stale claims. None affect the user-visible behavior — the editors work as the source code intends — but the documented patterns no longer match where each editor actually sits. Listed here for transparency; will be folded into the patterns doc in a later iteration.

D1 — FeatsEditor is no longer Pattern A

patterns doc says: "Pattern A — Generic CRUD shell, used by FeatsEditor.tsx, ItemsEditor.tsx, and the manual-editor side of SpellsEditor.tsx."
Source check: src/pages/compendium/FeatsEditor.tsx:37 imports upsertFeat, deleteFeat, fetchFeat from src/lib/compendium.ts and renders the editor body inside CompendiumEditorShell (not DevelopmentCompendiumManager). That's Pattern E in this reference's reconciled vocabulary.

D2 — ItemsEditor is no longer Pattern A

patterns doc says: same as D1.
Source check: src/pages/compendium/ItemsEditor.tsx:24 imports upsertItem, deleteItem, fetchItem and mounts CompendiumEditorShell. Same Pattern E.

D3 — The "unused per-entity helpers" claim is partly stale

patterns doc says: "upsertItem/fetchItem/deleteItem and upsertFeat/fetchFeat/deleteFeat exist with the same shape but nothing calls them — DevelopmentCompendiumManager bypasses them and calls upsertDocument directly."
As of the current code, FeatsEditor and ItemsEditor both call the per-entity helpers directly (FeatsEditor.tsx:37, ItemsEditor.tsx:24). The pattern doc's "currently unused" line predates the editors' migration to CompendiumEditorShell.

D4 — A fifth pattern shell exists: CompendiumEditorShell

patterns doc Pattern E talks about a master-detail layout but pins it specifically to SpellsEditor and lists the layout idioms (the spell-list-fullscreen body class, FilterBar with trailingActions, etc.) without naming a shell component.
src/components/compendium/CompendiumEditorShell.tsx exists and is the shared shell that FeatsEditor, ItemsEditor, and SpellsEditor all mount. It owns the top toolbar + FilterBar + 3-pane layout + super-tab strip + per-tab fieldset + preview slot. This is the "Pattern E" of reality.

D5 — DevelopmentCompendiumManager still has a live consumer: FacilitiesEditor

The patterns doc lists Pattern A consumers as FeatsEditor / ItemsEditor / SpellsEditor's manual side. With D1+D2 above, you might assume Pattern A is dead.
src/pages/compendium/FacilitiesEditor.tsx:6 still imports and renders DevelopmentCompendiumManager. Pattern A is alive but its current footprint is one editor.

D6 — RaceEditor and BackgroundEditor are wrappers, not independent editors

Not explicitly addressed in the patterns doc.
src/pages/compendium/RaceEditor.tsx and src/pages/compendium/BackgroundEditor.tsx are 20-line wrappers around FeatsEditor with scopeFeatType="race" / "background". Races + backgrounds live in the feats table with a discriminator. Worth noting explicitly so other agents don't go looking for a dedicated races / backgrounds table that doesn't exist yet.

The five editor patterns — reconciled

Five distinct shapes show up across the editor surface. Naming kept compatible with the patterns doc (A–E) so cross-reference doesn't break, but the assignments are refreshed against the current code.

Pattern Shell / Library Current consumers Use when
A DevelopmentCompendiumManager.tsx shell — declarative collectionName + defaultData + renderSpecificFields callback. Loads via fetchCollection and saves via upsertDocument. FacilitiesEditor Flat table with a small body of entity-specific fields and no relational children.
B Bespoke page-level component — hand-rolled useState + useEffect, direct upsertDocument / fetchDocument / deleteDocument calls. SourceEditor, TagManager, TagGroupEditor, UniqueOptionGroupEditor, ClassEditor, SubclassEditor, and the admin-side bespoke editors (ArmorEditor / WeaponsEditor / StatusesEditor / SpellcastingTypeEditor / StandardMulticlassEditor) Custom UX (multi-step flow, custom field shapes), nontrivial validation, or a layout that doesn't fit the master-detail mold.
C ProficiencyEntityShell.tsx — taxonomy / proficiency editor with prop toggles (includeFoundryAlias, includeSource, includeBasicRules, includeOrder, categoryFreeText). SkillsEditor, ToolsEditor, and every tab in AdminProficiencies (toolCategories, weaponCategories, weaponProperties, armorCategories, languageCategories, damageTypes, attributes, languages), plus the conditionCategories tab in StatusesEditor. Taxonomy-shape entities: id / name / identifier / order / description, possibly with a free-text category.
D Relational entity helper in src/lib/lore.ts — parent + metadata tables + 4 junction tables written atomically via batchQueryD1. LoreEditor / LoreArticle (owned by the system-applications branch; in this reference for completeness, not deep-dive) Parent-with-children entities where atomic FK + junction writes are mandatory. Future candidates: classes (cascade delete), characters, anything with 2+ child tables.
E CompendiumEditorShell.tsx master-detail shell — top toolbar + Foundry/Manual mode tabs + admin actions + FilterBar + 3-pane viewport-locked layout (list / editor / preview) + super-tab strip (Editor / Tags) + per-super-tab fieldset. FeatsEditor, ItemsEditor, SpellsEditor (and indirectly RaceEditor / BackgroundEditor via the FeatsEditor wrapper) Catalog-shaped entities where the editor's list of "all rows" is itself an authoring tool (scan dozens of records to find one to tweak), and where browser/editor visual parity is worth the shell investment.

Route map

Compendium content has three URL spaces. Read-only browsers under /compendium/<entity> (open to all signed-in users for the most part), admin authoring under /compendium/<entity>/manage or /edit/:id, and proposal-mode under /proposals/edit/<entity> (which wraps the same editor component inside ProposalEditorWrapper).

Entity Public browser Admin editor Proposal-mode
Classes /compendium/classes (ClassList) /compendium/classes/view/:slug (ClassView), /compendium/classes/new, /compendium/classes/edit/:slug (ClassEditor) /proposals/edit/classes/new, /proposals/edit/classes/edit/:id
Subclasses (no public browser; nested inside ClassView) /compendium/subclasses/new, /compendium/subclasses/edit/:id /proposals/edit/subclasses/new, /proposals/edit/subclasses/edit/:id
Spells /compendium/spells (SpellList) /compendium/spells/manage (SpellsEditor) /proposals/edit/spells

Spell rules (none) /compendium/spell-rules (SpellRulesEditor, admin only) /proposals/edit/spell-rules

These two files exist in src/components/compendium/ but my survey grep didn't surface them in the editor imports I scanned. Most likely consumers: TagsExplorer / TagClassifications. I'll verify before writing the components page.

Q3 — ClassEditor pattern: pure B, or B-with-shell-fragments?

ClassEditor.tsx is the heaviest editor (advancement timeline, features list, scaling, proficiencies, multiclass) and uses many of the shared components, but the page-level composition is bespoke (no CompendiumEditorShell, no DevelopmentCompendiumManager). Categorizing as Pattern B for now. Worth checking whether to call out a "Pattern B+ composite" variant in a later iteration.

Q4 — Cascade-delete strategy for classes — answered

Original question: documented as open in docs/features/compendium-classes.md; should the data-flow page describe it as a follow-up rather than a current behavior?

Answer (2026-05-27): Yes — treat as a known follow-up. The rules when it lands: a class delete cascades to its subclasses, its class features, and its advancements. It does not cascade to anything those advancements grant (e.g. an advancement that grants Warlock Invocations: the advancement row goes, the Invocation option items stay — they're independent compendium content). Low priority for now since no classes are being deleted in practice. See summary A4 for the human-language version.
