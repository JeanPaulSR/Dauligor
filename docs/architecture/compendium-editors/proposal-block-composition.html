How a Class is Built in a Proposal Block

Deep dive into the proposal-block composition model + the bugs we hit authoring a class — branch compendium-editors, 2026-05-29.

The one-sentence version. A class isn't a single save — it's a cluster of separate entities (the class row, each of its features, each scaling column, option groups + their items, and the advancements that live on the class), each queued as its own proposal revision into one block, all cross-referencing each other by client-minted UUIDs, and (eventually) approved together as a unit. Every bug you hit was a place where that "proposal-by-proposal" composition leaked — a child gated on a parent that isn't live yet, or a create mis-queued as an update against a row that doesn't exist yet.

1 · The model — a class is a cluster, not a row

When an author builds, say, a Druid in a block, they're really creating a graph of entities that all land together:

class:Druid (id C)
 ├─ feature:Wild Shape (parent_id = C)         ← features table, parent_id → class
 │    └─ advancements[] (on the feature)
 ├─ feature:Druidic (parent_id = C)
 ├─ scaling_column:"Beast CR" (parent_id = C)   ← scaling_columns, parent_id → class
 ├─ advancements[] on the class                 ← stored on the class row; reference ↓
 │    ├─ ScaleValue → scaling_column "Beast CR"
 │    ├─ ItemGrant  → option_group / feat
 │    └─ GrantSpells→ spell / spell_rule / class
 ├─ unique_option_group:"Wild Shape Forms" (+ items)   ← group_id → its items
 └─ subclass:"Circle of the Moon" (class_id = C)        ← subclasses, class_id → class

Each node is a separate row in a separate table, so each is a separate proposal revision. They all reference each other by id. The whole point of the block is to let those references resolve even though none of the rows exist live yet.

2 · The six principles that make it work

  1. Client-minted UUIDs, stable through approval. Every new entity gets its id now, on the client. A feature's parent_id can point at the class's id before the class is live, because the same id is used at author-time and at approval-time.
  2. The block accumulates in two stages. Each editor save calls useProposalAccumulator → queueChange into an in-memory queue; "Save Progress / Submit" flushes the queue to the server as pending_revisions rows (the persisted drafts).
  3. The editor shows live + drafts merged. getDraftedEntities overlays the block's drafts onto the live fetch (→ byId/createdIds). Pickers read it via useProposalDraftOptions; lists/panels via useBlockDraftedList. So you see what you're building.
  4. CREATE-then-EDIT folds into one CREATE. You must never end up POSTing an update for an entity that only exists as a same-block create — the server tries to snapshot a live row, finds nothing, and 404s. The flush folds them (dedupe in-queue; patch the existing create draft across flushes).
  5. Children are authored under a not-yet-live parent. A class created in-block has no route id (URL stays on /new). Child sections must gate on effectiveId = routeId ?? pendingCreateId (the client-minted id, set via recordCreate), not the raw route id — otherwise the Features/Columns/Subclasses sections stay locked until a reload.
  6. Approve atomically. On approval the whole block applies in dependency order with reference-integrity (guard #1). proposal-system Part D — not built yet.

4 · The bugs, diagnosed against the model

Fixed  Issue 1 — Class Features locked until reload  ours

Violated principle 5. The Features tab button and handleSaveFeature already used effectiveId, but the Features tab content was still wrapped in {id && (…)} (ClassEditor line 3179). On a class created in-block, id (route param) is null, so the tab was clickable but its body rendered nothing until a reload set the id. Fix: {id && …} → {effectiveId && …} (the same gate the subclasses content + scaling panel already got). Now child sections unlock the moment the class is queued.

Fixed  Issue 3 — 404 "update on missing feature" → can't link advancements, scaling "didn't save"  ours

Violated principle 4 — and it was the big one. The "Add Feature" button pre-mints the new feature's id (id: crypto.randomUUID(), line 3187). My feature-slice handleSaveFeature then decided create-vs-update with isCreate = !editingFeature.id — which is never true for a new feature (the id is already there). So a brand-new feature was queued as an UPDATE. On flush, the endpoint tries to snapshot the live feature row, finds none, and throws 404 "Cannot propose update on missing feature".

Worse: that 404 is thrown mid-flush, so it aborts the entire flush — which is why the scaling column and everything else queued in the same block "didn't save" either. And you "couldn't link advancements" because every feature save bounced.

Why admin-direct never hit it: the old path called upsertFeature (insert-or-update), which happily creates when the row is absent. Routing through create/update for proposals removed that safety net and exposed the bad branch.

Fix: decide create-vs-update by live-row membership, not id-presence:

const isCreate = !features.some(f => String(f.id) === String(editingFeature.id));

So a new feature or a same-block draft feature is a CREATE (the accumulator folds repeat creates / patches the existing draft — principle 4), and only a genuinely live feature is an UPDATE (where the snapshot succeeds). Applied in ClassEditor + SubclassEditor.

Verify  Issue 2 — scaling column shows "(unnamed draft)"  ours (needs a repro)

The picker shows (unnamed draft) when the queued scaling_column payload's name is empty. The matrix editor does include name in its create payload (ScalingMatrixEditor line 163), and getDraftedEntities merges a later rename onto the create — so an empty name means the column was created (and never renamed) without a name typed in the modal. Next time: confirm whether the modal's Name field was filled at save. If it was filled and still shows unnamed, there's a capture bug to chase; if it was blank, that's expected. (Now that Issue 3 no longer aborts the flush, re-test this in isolation.)

6 · How to test it now

Hard-reload (you've applied the local migrations), then in a block create a new class and, without reloading: add a feature (it should save + appear in the Features list), add a scaling column, link a ScaleValue advancement to it, add a subclass. Submit the block — it should flush without the 404. See proposal-mode-test-checklist.html for the full pass. Approve still won't land the cluster (Part D pending) — that's expected.
