Purpose

One shell that owns all the chrome shared by every Pattern E editor page, so the editor files themselves only have to supply data + field bodies. It owns:

  • The top toolbar — back-link + a mode switcher (Foundry-Import / Manual-Editor) + an admin-actions slot.
  • The FilterBar row — search + filter modal + result count + "New {entity}" button.
  • The 3-pane viewport-locked layout — compact virtualised list (280px) | tabbed editor (flex) | read-only preview (420px).
  • The super-tab strip — Editor vs Tags — with consumer-supplied inner sub-tabs.
  • The single <form> wrapper so the submit button fires regardless of which super-tab is mounted (Radix unmounts inactive tab content, but the form reads from the consumer's controlled formData, so unmounting is harmless).
  • Responsive collapse — xl+ shows all three panes; lg–xl drops the preview; below lg it's a single-pane master-detail drilldown.
  • Proposal-aware chrome — cascade banner, read-only banner, Edit-Base swap, chrome-offset bump.

Internal state vs consumer state

Owned by the shellOwned by the consumer (editor)
  • paneHeight (viewport-derived, resize listener)
  • listView ('list' | 'editor') for the <lg drilldown
  • editorView ('editor' | 'tags') super-tab
  • editorSubTab + tagsSubTab inner-tab selection
  • The spell-list-fullscreen body class (added on mount, removed on unmount)
  • formData — the entire field state
  • selectedId + the selection logic (incl. the preserve-selection idiom)
  • Filter state (search, active filters) — the shell just renders it
  • The actual save / delete / reset logic
  • Proposal-mode wiring (the shell receives resolved isReadOnly / banners as props)

Notable: the super-tab + sub-tab state persists across selectedId changes. A proposer editing Tags > Prereqs on Fireball who clicks Frostbite stays on Tags > Prereqs. Only onNew resets to Editor > first-sub-tab (the shell's handleNew resets the view before calling the consumer's onNew).

Data shape

The shell is generic over TRow — it doesn't care what an entity row looks like. It only touches rows through the consumer-supplied getRowId and the listColumns[].render functions. The actual field editing happens in the editorSubTabs[].render() / tagsSubTabs[].render() bodies, which the consumer supplies and which read/write the consumer's own formData. So the shell has no opinion on the entity's data shape at all — that's the point.

UI issues

My initial observations from the source, plus room for yours. Status: open / in progress / fixed / deferred / verify.

I1 — verify List card may be pinned to 280px on narrow viewports

The list Card (line ~477) carries an inline style={{ width: 280px, height: paneHeight }} that applies at every breakpoint. At <lg with listView==='list' the card also gets flex flex-1 (meant to fill the column), but the inline width:280px likely wins — so on a phone the list could render as a 280px column with empty space to its right instead of full-width.

Proposed fix: drop the inline width (and the redundant lg:w-[280px] className) in favor of a responsive class: full-width below lg, 280px at lg+. Same likely applies to the inline height fighting flex-1. Needs visual confirmation on a narrow viewport before changing — this is a "verify" not a confirmed bug.

I2 — cosmetic Redundant width declarations on the list card

The list card sets its 280px width three ways: the lg:w-[280px] className, the LIST_COL_WIDTH_PX inline style, and the layout assumption. Whatever the fix for I1, collapsing these to a single source of truth (one constant, one place) avoids future drift.

Gotchas

  • The body class is named spell-list-fullscreen but applies to all Pattern E editors. Historical name — the CSS rules (in src/index.css) strip the global container padding + hide the footer for any fullscreen page. If we ever rename it, it's a coordinated change across the shell, the browsers, and the CSS.
  • Inactive tab content unmounts (Radix default). The single <form> wrapping both super-tabs is what makes Save work anyway — the submit reads the consumer's controlled formData, not the (unmounted) DOM fields. Don't switch to uncontrolled inputs inside the sub-tabs or Save will silently drop the unmounted ones.
  • The <fieldset disabled={isReadOnly}> wraps each super-tab body, but the action row (Save/Reset/Delete) lives OUTSIDE both fieldsets (in the identity row), so the buttons stay clickable in read-only/browse mode. Intentional — the Edit-Base swap needs the button live even when the form is locked.
  • chromeOffset is 200 normally, 320 in proposal mode. If the proposal wrapper's banner/footer height changes, this magic number needs to track it or the bottom pane gets clipped / leaves a gap.
