Proposal Cross-Reference Audit

Every editor surface the proposal system could touch, and every place one entity references another — so guard #1 (reference-integrity) can be made exhaustive and Part C (picker overlays) complete. For proposal-system. · 2026-05-28
audit for proposal-system compendium-editors branch
Why this exists. The Parts B+C handoff and the cross-referential-cluster design both scope tightly to scaling_column + the class/subclass/option-group cluster. A walkthrough of the actual editor code shows the real cross-reference graph is much larger, spans three different picker-injection layers (only one of which the handoff's Part C addresses), and references several proposable entity types that guard #1's enumerated ref-walk doesn't visit. This doc lays out the complete graph so the scope line can be drawn deliberately instead of by omission. Nothing here is a criticism of the design — the stable-UUID + atomic-block approach is sound; this is about completeness of coverage.

TL;DR — the headline findings

0. The most fundamental gap — a proposed class has no features

A class is its features (Wild Shape, Rage, Channel Divinity), stored as separate rows in the features table (parent_id = the class). But feature is not a proposable type, and ClassEditor saves features via a direct upsertFeature → upsertDocument('features', …) (ClassEditor.tsx:1140/1147) that bypasses the accumulator the class row uses (:1313). So a content-creator proposing a Druid can queue the class shell but 403s the moment they add Wild Shape — the exact Bug #2 pattern, for features. Decision #2 defers features; that deferral removes the ability to propose a usable class. This is arguably the scope call — see Features: the biggest deferred hole.

1. Part C reaches only one of three picker-injection layers

The handoff's Part C plan is "EntityPicker gains a draftEntries prop; merge drafts at the ClassEditor / SubclassEditor / UniqueOptionGroupEditor pickers." But the cross-reference pickers live in three places, and only the direct-EntityPicker sites are covered:

  • AdvancementManager pickers (option groups, option items, feats, features, scaling columns) read from available* props passed by the parent editor — they use SingleSelectSearch + inline selects, not EntityPicker.
  • RequirementsEditor pickers (class, subclass, feature, spell, spell-rule, option-item, proficiency) read from a lookups prop passed by the parent — also SingleSelectSearch, not EntityPicker. The handoff doesn't mention RequirementsEditor at all.
  • SpellAdvancementEditors (GrantSpells / ExtendSpellList) fetch their own spells / rules / classes via fetchCollection — live-only, no overlay path.

A draftEntries prop on EntityPicker alone leaves the advancement and requirement pickers blind to block drafts.

2. Guard #1's reference-walk is missing most of the JSON-buried refs

The design's guard #1 enumerates subclass.class_id, option-item group_id, scaling_column.parent_id, advancement scaling_column_id/quantity_column_id, class.unique_option_mappings, group.class_ids, subclass.unique_option_group_ids. The full graph (below) adds at least: advancement spell-grant refs (spellIds, ruleId, countsAsClassId, classId, scopeClassId), advancement grant/choice pools (pool, optionalPool, excludedOptionIds → option groups / option items / feats), optionScalingColumnId, the entire requirements_tree (class / subclass / spell / spell-rule / option-item leaves), tag refs (tags / required_tags / tag_ids → draftable tags), and item.container_id. Since the schema has no FK on any of these, a dropped draft reference goes live dangling unless guard #1 walks it.

3. Cross-block references are silently un-authorable and would fail guard #1

Draft overlays are scoped to the active bundle (useProposalEntityDrafts keys off activeBundleId). An entity in block A referencing a draft in block B: (a) can't be authored — block B's drafts never appear in block A's pickers; and (b) would fail guard #1, which resolves a ref only to "a live row OR another revision in the same block." This is probably the correct constraint, but it should be a stated one with a clear authoring-time message, not a silent dead-end.

What can be a block draft (reference targets)

Only proposable entity types can exist as block drafts, so only references to these types are draft hazards. From src/lib/proposalAware.ts (ProposalEntityType) plus Part A's addition:

Proposable (can be a block draft)NOT proposable (refs always resolve to live rows)
tag, tag_group, spell_rule, spell_rule_application, spell, class, subclass, feat, item, unique_option_group, unique_option_item, scaling_column (added by Part A) feature (deferred — Decision #2), proficiencies (skills/tools/weapons/armor/languages + their categories), attributes, spellcastingTypes, pact/known scaling tables, sources

Consequence: a reference whose target is in the right-hand column can never point at a block draft, so it's not a guard #1 hazard and needs no picker overlay. This is what keeps the problem bounded — see feature-refs are safe.

The picker-injection layers

A draft overlay has to be applied wherever a picker's option list is assembled. There are four distinct assembly points:

Layer Where options are assembled How a draft overlay must be applied In handoff Part C?
L1 AdvancementManager — reads availableScalingColumns, availableOptionGroups, availableOptionItems, availableFeats, availableFeatures (props) The parent editor (ClassEditor / SubclassEditor / FeatsEditor / ItemsEditor) merges useProposalDraftOptions(type) into each available* array before passing it down. AdvancementManager itself needs no change. no
L2 RequirementsEditor — reads lookups.{classes, subclasses, features, spells, spellRules, optionGroups, proficiencies} (prop) The parent editor (FeatsEditor / UniqueOptionGroupEditor) merges drafts into the lookups arrays. RequirementsEditor uses SingleSelectSearch, so a draftEntries prop on EntityPicker doesn't reach it. no
L3 SpellAdvancementEditors (GrantSpells / ExtendSpellList) — fetches its own spells, rules, classes via useSpellAdvancementFoundation() This component needs its own useProposalDraftOptions calls (or the parent must pass pools down, which it currently doesn't). Self-contained fetch = self-contained overlay. no
L4 Direct EntityPicker sites — UniqueOptionGroupEditor class-restriction picker; SubclassEditor class picker; ClassEditor option-group / column pickers The handoff's draftEntries prop, merged at each call site. yes

Good news: every catalog editor already imports useProposalEntityDrafts(<own-type>) (ClassEditor → 'class', SubclassEditor → 'subclass', FeatsEditor → 'feat', ItemsEditor → 'item', SpellsEditor → 'spell', UniqueOptionGroupEditor → 'unique_option_group'). That hook is currently used only to overlay each editor's own list with its own drafts. The same machinery, called for other types, is exactly what L1/L2/L3 need — the wiring exists; it just isn't pointed at the cross-reference pickers yet.

Roughly two-thirds of the draftable references aren't on the enumerated walk. Each one is a place a dropped/rejected draft could leave a live row pointing at nothing, since the schema has no FK to catch it (exactly the S2 failure mode the guard is meant to close).

Cross-block hazard (the "different block" case)

useProposalEntityDrafts(type) → getDraftedEntities(type, ctx, drafts, activeBundleId) — scoped to the active bundle. Outside a ProposalEditorWrapper it returns empty (admin-direct editors never see block drafts — correct). Inside a wrapper, it returns only the current block's drafts. So:

  • If a content-creator splits a logical cluster across two blocks (class in block A, its subclass in block B), block B's class picker won't show block A's draft class. The reference can't be authored.
  • Even if authored by other means, guard #1 resolves a ref to "a live row OR another revision in the same block" — a block-B ref to a block-A draft fails validation.

This matches Decision #3 ("current active block only"). The audit's only ask: make the dead-end legible — when a picker has nothing to show because the target is in another block, say so, rather than rendering an empty dropdown that looks like a bug. (Same applies to another user's block.)

Features: the biggest deferred hole

A class is its features. Wild Shape, Rage, Channel Divinity are rows in the features table (parent_id = class.id, parent_type = 'class'), authored in ClassEditor's feature modal. handleSaveFeature persists them with upsertFeature → upsertDocument('features', …) (compendium.ts:747, ClassEditor.tsx:1140/1147) directly — there is no proposal branch in that handler (contrast the class row at :1313, which routes through applyProposalWrite(classWriter, …)). And feature is not in ProposalEntityType.

Consequence: a content-creator proposing a class can queue the class row, but adding a feature hits the same staff-only DB gate that blocked scaling columns — a 403. A proposed class is a feature-less shell. The handoff's "done" scenario reflects this: it lists "creates a class, adds a scaling column, … adds a unique option group" but never "adds Wild Shape."

Why it was deferred (and why it's a real lift): a feature isn't a leaf — it's an interior node carrying its own reference graph:

  • activities + effects (Foundry automation JSON)
  • advancements (JSON) — re-dragging in the entire advancement reference set (scaling columns, option groups, spells, …)
  • scalingColumnId / quantityColumnId directly on the feature row
  • unique_option_groups.feature_id points back at features — option groups attach to a feature

So "make features proposable" is Part A/B/C-for-features plus it re-opens the advancement graph and the option-group→feature link. That heft is presumably why Decision #2 punted — but the punt isn't free: it's the difference between "propose a class" working and not.

Correction to an earlier framing in this audit. Feature-targeting references (requirements feature leaf, advancement usesFeatureId, ItemBumpUses target.kind='feature', unique_option_groups.feature_id) are not "safe because features can't dangle." They're blocked — the feature they'd point at can't be created in the block at all. The matrix rows that mark a feature target "(not draftable)" should be read as "(blocked by the feature deferral)," not "(benign)."

Recommendations for proposal-system

  1. Decide the features question first — it gates "propose a class." A class without its features isn't a class. Either bring feature into scope (Part A/B/C for features, accepting the activities / effects / advancements surface and the option-group→feature link), or explicitly decide that a proposed class is a shell an admin completes with features after approval — and make the editor say so instead of 403-ing when a content-creator adds one. Either way it should be a stated decision; right now it's an unannounced 403. For the "build a class in a block" use case this outranks the scaling-column work.
  2. Draw the rest of the scope line explicitly. If this pass is genuinely "scaling_column + class/subclass/option-group cluster only," then advancement spell/feat grants, requirements-tree refs, tag refs, and item.container_id are out — say so in the design, and have the editors not offer those draft overlays (so authors don't build references the approval side silently won't validate).
  3. If the cluster is meant to be authorable end-to-end (your "done" scenario implies it — it includes a spell grant), then:
    • Part C must cover L1, L2, L3, not just L4. Concretely: merge useProposalDraftOptions into the parent-passed available* arrays (AdvancementManager) and lookups arrays (RequirementsEditor), and add overlay calls inside SpellAdvancementEditors.
    • Guard #1's walk must add every no row in the gap table.
  4. Resolve the parent_type question for scaling columns (class/subclass only, or all six owners) and have guard #1's draft-parent resolution + Part B's accumulator wiring agree on the answer.
  5. Make cross-block dead-ends legible (a one-line "this is in another block" affordance) rather than empty dropdowns.
  6. Link the feature deferral to its safety guarantee in the design, and flag the three feature-ref sites as the first things to revisit when features become proposable.

This audit deliberately stops at what references exist and where. Which to bring in scope is proposal-system's call; the editor-side wiring for whatever is chosen (the L1/L2/L3 overlays) is compendium-editors' to implement.
