0 · Setup — the two modes you'll switch between

Admin-direct mode — reach editors via /compendium/… (e.g. /compendium/classes, /compendium/feats, /compendium/items, /compendium/unique-options). As admin, saves write straight to the DB. Toasts say "…created / updated / deleted." This is the no-regression baseline.

Block mode — reach editors via /proposals/edit/… (e.g. /proposals/edit/classes, /proposals/edit/feats, /proposals/edit/option-groups). You'll hit a "pick or create a block" gate first; once a block is active, the editor mounts and saves are queued into the block. Toasts say "…added to block." (You can do this as admin — you don't need a content-creator account for the UX checks. The content-creator "no longer gets a 403" win is verified later at the full end-to-end, which needs the DB migration.)

Check  1 · No regression in admin-direct mode

Most important: confirm routing feature/scaling saves through the new writer didn't break ordinary editing. Do these via /compendium/*, no block.

  • Edit an existing class → Class Columns panel → add a column, rename it (blur the field), open the matrix editor, save values, delete a column. Expect: each action persists immediately; toasts are "created/updated/deleted" (NOT "added to block"); columns survive a page reload.
  • Same class → Features → add a feature, edit it, delete it. Expect: saves immediately, shows in the feature list, survives reload.
  • Open a feat (/compendium/feats) that allows scaling → add a column → reference it in a ScaleValue advancement → save → reopen. Expect: the advancement still points at the column after reopen; no console errors.
  • Open an item (/compendium/items) with an advancement → confirm the advancement editor + any scaling reference still save.
  • Open a class's Subclasses tab → edit a subclass → add/edit/delete a subclass feature. Expect: saves immediately, persists.

If anything here behaves differently than it did last week, that's a real regression — flag it.

Check  2 · proposal-system's new behaviors (now in your worktree)

  • Go to /proposals/edit/classes (or feats / items / option-groups) with no active block. Expect: a "pick or create a block" gate appears instead of the editor body — you can't author until a block exists.
  • Create or pick a block → the editor mounts normally.
  • Make any save inside the block. Expect: the toast reads "…added to block" (the old confusing "submitted for review" wording is gone in-block).

Check  3 · Part C — draft entities selectable in pickers

This is the headline new work. Do it in a block, on an existing class/feat (not a brand-new one — see Skip §2). Remember: a freshly-queued draft will NOT show up in its own list/panel yet (that's F2, Skip §1) — you're checking that it shows up in the picker.

3a · Scaling column (cleanest single-editor check)

  • In a block, edit an existing class → Class Columns → Add a column → fill the matrix → save. Expect: toast "added to block." The column will not appear in the Class Columns panel (F2 gap — expected, not a bug).
  • Same class → Progression & Advancements → add/edit an advancement that has a scaling-column dropdown (e.g. ScaleValue, or a Trait/ItemChoice using scaling) → open the dropdown. Expect: your just-created column appears as "<name> (in this block)" and is selectable.

3b · Feature

  • Same block/class → add a feature (feature modal → save → "added to block").
  • In an advancement that links features (the availableFeatures picker) → confirm the draft feature shows as "(in this block)."

3c · Cross-editor, same block thorough

These prove a reference can cross editor boundaries inside one block. More fiddly; optional if short on time.

  • Option group: in the same block, go to /proposals/edit/option-groups/new → create a group → save. Back in the class editor (same block) → an advancement that references option groups shows the draft group "(in this block)."
  • Feat: create a feat draft in /proposals/edit/feats (same block) → in a class advancement that grants feats (ItemGrant), the draft feat appears.
  • Requirements (L2): in a feat editor (in block) → open the Requirements builder → a class / subclass / spell-rule / option-group leaf picker shows same-block drafts.
  • Spell advancements (L3): in a GrantSpells / ExtendSpellList advancement → the spell / rule / class pickers show same-block drafts.
  • Class restriction (L4): in the option-group editor (in block) → "Class Restrictions" picker → shows same-block draft classes.

Verify  Recently fixed (were "skip" — now wired)

Implemented this session — these are the ones to actually exercise now.

1 · Drafts now appear in their own list / panel (F2)

A queued draft now shows up where you authored it, no reload:

  • Add a scaling column in a block → it appears in the Columns panel (with breakpoints), not just the advancement dropdown.
  • Add a feature in a block → it appears in the class's Features list / the subclass level grid.

Still partial: the option-group catalog list (/proposals/edit/option-groups) won't show a draft group yet — that route isn't wrapped (proposal-system). You can still add options to a new group (see §2).

2 · New entity's child sections unlock without reload (reload-gap)

Create a new class / subclass / option group in a block, then immediately:

  • Class / Subclass → the Columns panel and Features are usable right away — no reload.
  • Option group → the Individual Options section shows and Add Option works straight away (this was your Issue 2).

Child sections gate on the client-minted id now, not the URL id.

3 · Subclass under a draft parent class now displays you flagged this

In a block, create a class, then add a subclass to it (both still drafts). The subclass editor now shows the draft parent class (name / hit die / reference labels), resolved from the block instead of a failed live fetch — and the class_id link was always correct. (Bonus: re-opening a draft scaling column in the matrix editor now loads its draft values too, instead of a blank form.)
