The four storage idioms

Every column in the compendium tables is one of four things:

Idiom What it holds Example
Flat scalar A single string, number, or 0/1 boolean. Direct SQL column with a simple type. name, identifier, level, rarity, magical
FK column Points at another row's primary key. Usually _id suffix. PRAGMA foreign_keys = ON enforces them. source_id → sources.id; parent_id → classes.id or subclasses.id; class_id on subclasses
JSON column A serialized JSON value (object or array). Auto-parsed on read by queryD1's allowlist. Read-only at the SQL level — you can't WHERE on a JSON path efficiently. activities, effects, advancements, proficiencies, spellcasting, tag_ids
Junction table A two-FK row connecting two entities. Mostly used by the lore family; the rest of the compendium uses JSON arrays for tag-style relationships today. lore_article_eras, lore_article_campaigns, lore_article_tags, lore_secret_eras

The choice between JSON column and junction table is the design decision worth understanding; see section below. The other three idioms are uncontroversial.

The entity-relation map

The big picture of how the catalog entities link to each other:

sources ← classes
subclasses
spells
feats
items
facilities
features → scaling_columns
unique_option_groups
tags / tag_groups

Read it left-to-right: sources is the universal FK target — every catalog row points at the sourcebook (PHB, XGE, DMG, homebrew, etc.) it came from. The catalog entities are the content (classes, spells, etc.). They produce relations forward to scaling tables, option groups, tags, and back-link via FK to their owning class / subclass / feature.

The links inside the catalog cluster, expressed as plain sentences:

  • Subclasses belong to one class. subclasses.class_id → classes.id.
  • Features belong to a class OR a subclass. features.parent_id → classes.id OR subclasses.id; features.parent_type says which. (Polymorphic; see below.)
  • Scaling columns belong to a class OR a subclass OR a feat OR an item. Same polymorphic pattern.
  • Items reference a base proficiency definition. A weapon-item points at a weapons row; an armor-item at armor; a tool-item at tools. (Polymorphic FK chain; see below.)
  • Unique option groups (Metamagic, Invocations, Maneuvers, etc.) belong to a feature. unique_option_groups.feature_id → features.id. The option items inside the group sit in unique_option_items with a group FK.
  • Spells carry a lightweight summary alongside. spells + spell_summaries — the summary table is denormalized for picker performance. Spell rules + spell lists are independent tables that resolve "which spells does this class actually know" through the rules engine.
  • Tags live in tags + tag_groups. Today most entities reference tags via JSON arrays (spells.tags, feats.tags, items.tags, classes.tag_ids). The future plan is to migrate these to junction tables; see below.

The lore family sits next to the catalog cluster but on its own: lore_articles with seven supporting tables (four lore_meta_* for characters/locations/organizations/deities, three junction families for eras/campaigns/tags, plus lore_links and lore_secrets). It's the canonical Pattern D consumer; out of scope here in any depth (owned by system-applications).

The three polymorphic patterns

"Polymorphic" appears three times in the schema, each with a different flavor and a different reason:

1. parent_id + parent_type — "which entity owns this child?"

Used by: features, scaling_columns.

Why: a feature can be owned by a class or a subclass; a scaling column can be owned by a class, subclass, feat, or item. Rather than one nullable FK per possible parent type, store the FK in parent_id and the type in parent_type. Queries are WHERE parent_id = ? AND parent_type = 'class'.

Trade-off: the FK can't be enforced by SQL (since parent_id doesn't know which table to point at), so referential integrity is enforced at the application layer. The upside is one polymorphic pattern instead of N FKs per child table.

2. base_*_id — "which proficiency definition does this item reference?"

Used by: items only.

An item row carries three FK columns: base_weapon_id, base_armor_id, base_tool_id. Exactly one is non-null per row, driven by item_type. Each points at a row in weapons / armor / tools — the proficiency definition tables that hold rules-level shape (Greatsword's category, weapon type, properties, etc.).

Why a chain instead of a single polymorphic parent_id: SQL can enforce each individual FK, and proficiency resolution code can just check the three columns in sequence. Sibling field base_item (TEXT) preserves Foundry's slug verbatim so a re-resolve can wire the FK later if the proficiency definition arrives after the item.

3. feat_type discriminator — placeholder, not real polymorphism

Used by: feats.

The feats table currently holds rows of six feat-type variants: feat, class, subclass, race, background, monster. Of those, race and background are placeholder behavior until they get their own tables (per summary A1 and roadmap step 1).

This is a different shape of "polymorphic" — it's a single table doing two jobs. The other two patterns above are real polymorphism (designed-in); this one is mid-migration.

JSON columns vs separate tables

The big design decision the schema makes repeatedly. Reasonable rules of thumb based on what the schema actually does today:

Use JSON column when… Use a separate table when…
The data is deeply nested (arrays of objects with sub-arrays) The data has its own identity worth referencing (an FK target)
It belongs entirely to one parent (no cross-references) Multiple parents share the same children
You don't need to filter SQL queries by its contents You want to WHERE on it ("find all classes with the Combat tag")
The shape changes over time and rigid columns would churn the schema The shape is stable
Atomicity is "save the whole thing or none of it" already satisfied by writing one row Atomicity needs batchQueryD1 + a transaction

Concrete examples from the current schema:

  • advancements — JSON. A class's advancement timeline is an array of typed rules (HitPoints / ItemChoice / ASI / Subclass / ...). Deeply nested; belongs entirely to the class; rarely queried by its inner contents. JSON is the right call. The classes.md schema doc has a direct quote on this: "JSON for fields like advancements because they are deeply nested arrays of rules that change frequently. This avoids creating dozens of junction tables for a single entity."
  • activities + effects — JSON. Same reasoning, repeated on every Foundry-automatable entity. Each one is a Foundry document shape; the editor composes them as JSON and the export emits them as JSON. No cross-entity reference happens through these fields.
  • proficiencies — JSON. Class proficiencies are a nested object (saves, skills, tools, weapons, armor, languages, each with choices and fixed lists). The proficiency catalog tables (skills, tools, etc.) are the queryable side; the class's composition of those proficiencies is JSON.
  • tags on each entity — JSON array today, junction table tomorrow. See below. This is the canonical case of "we'd rather have a junction but JSON was faster to ship."
  • Lore meta + junctions — separate tables. Lore articles have multiple referenceable children (characters, locations, organizations, deities); each has its own table. Lore articles also relate to multiple eras + multiple campaigns + multiple tags; each of those is a junction table. Junctions are the right call because the relationships are many-to-many and queryable.

Tags — current state and migration plan

Current. Tags live in two tables: tags (the leaves) and tag_groups (the parents, supporting the "subtag implies its parent" invariant the TagPicker enforces). Entities reference tags via JSON arrays on their own row: spells.tags, feats.tags, items.tags, classes.tag_ids. The arrays carry tag IDs; the picker resolves them to display names + parent groups at render time.

Why JSON. The original migration shipped this way because it was the smallest change from the Firestore-era shape (Firestore had tags as embedded arrays). It also has a practical benefit: filtering an entity by its tags is a client-side operation that already has the array on hand — no extra query.

The cost. SQL queries that need to find "all classes with this tag" can't use an index. They're stuck doing LIKE '%' || tag_id || '%' on the JSON column — works for occasional admin queries, doesn't scale to runtime filtering.

The migration plan. The patterns doc flags this as Priority 6: "Migrate JSON-array tag references (classes.tag_ids, feats.tags, items.tags, spells.tags) into proper junction tables." When that lands, filtering by tag becomes an indexed query, and the JSON arrays go away.

In the meantime, the editor side treats tags as a JSON array of IDs and the public browsers do client-side filtering through the same arrays. The TagPicker enforces the "subtag-implies-parent" invariant at toggle time so the array stays consistent.

Source-scoped uniqueness

Identifiers (the URL-safe slug used in routes and references) are not globally unique — they're unique per source. The reasoning: it's legitimate for the same name to appear in two sourcebooks. Blade of Disaster exists in FRHF and TCE with mechanical differences; both deserve a row.

The migrations that landed this:

Migration What it added
20260526-2300 UNIQUE(source_id, identifier) on feats and items
20260527-1400 UNIQUE(source_id, identifier) on spells, classes, subclasses
20260527-1410 UNIQUE(parent_type, parent_id, identifier) on features

The features constraint is the polymorphic-parent variant: a feature's identifier is unique within its owning class or subclass, not globally. That's the right granularity — two classes can both have an "ability-score-improvement" feature without colliding.

Editor behavior on conflict. When the constraint fires (someone tries to save an entity with a duplicate identifier in the same source), the upsert throws and the editor's error handler surfaces it via toast.error. The identifier auto-generation in normalizeCompendiumData uses slugify(name), which means the conflict surface is "two entities with the exact same name from the same source" — rare enough to be a real authoring signal.

The source-scoped identifier convention already exists in routing. Hash-links use the <identifier>_<sourceAbbrev> form — e.g. the spell Firebolt is reached at #firebolt_phb. This is built by useCompendiumHashLink for the Spell / Feat / Item / Facilities browsers, and the class routes adopted the same form (cleric_phb) via buildClassSlug / useClassRouteId. The separator is an underscore and the suffix is the source's abbreviation (lowercased), not a slug.

You confirmed it's worth adopting in general — the source-scoped form is the right model wherever an identifier needs to disambiguate the same name across sourcebooks. The stored identifier column today is still the bare slug (firebolt); the source suffix is composed at link-build time. Standardising on the _<sourceAbbrev> form more broadly is the direction.

Foreign keys + cascade behavior

D1 runs with PRAGMA foreign_keys = ON by default — FK constraints are enforced. Three behaviors apply, depending on the FK:

Behavior When it's used Concrete example
ON DELETE SET NULL The child can survive losing its parent. Deleting the parent leaves the child intact with the FK cleared. items.source_id — if a sourcebook is removed, the items it published stay in the catalog with no source.
ON DELETE CASCADE The child cannot exist without its parent. Deleting the parent deletes the child too. unique_option_items.group_id — deleting an option group deletes its items. Also why AGENTS rule #2 bans INSERT OR REPLACE: it triggers a cascade because OR REPLACE internally deletes and reinserts.
No cascade (application-managed) The relationship is polymorphic, so SQL can't enforce it. The editor or a future cascade-delete handler is responsible. features.parent_id + scaling_columns.parent_id — deleting a class doesn't auto-delete its features today. See summary A4 for the future cascade-delete rules.

The future class cascade-delete rules (from A4, captured for cross-reference):

  • Deleting a class cascades to: its subclasses, its class features, its advancements (the advancement rows).
  • Deleting a class does not cascade to anything those advancements grant — e.g., an advancement granting Warlock Invocations leaves the Invocation option items intact (they're independent content possibly referenced elsewhere).
  • Implementation will batch the delete via batchQueryD1 — one transaction, all-or-nothing.

Pattern D — atomic multi-table writes

When one logical save needs to touch multiple tables, the editor uses batchQueryD1 to wrap all the SQL in a single D1 transaction. Either every query succeeds or none of them do. No half-written state.

The canonical consumer today is src/lib/lore.ts's upsertLoreArticle: one batch writes the parent lore_articles row, the relevant lore_meta_<type> row, plus replaces all rows in the four junction tables (lore_article_eras, lore_article_campaigns, lore_article_tags, lore_links) for this article. The "replace" pattern is DELETE WHERE article_id = ? followed by INSERT rows for the new associations — both inside the same transaction so the deletes don't surface to readers.

Future Pattern D consumers per the survey + answers:

  • Class cascade-delete (per A4) — class + subclasses + features + advancements.
  • Character builder finalization — character row + selections + inventory + spell list — all atomic.
  • Any new entity that owns multiple child tables — the principle of "start with separate tables" (per A1) tends to land more Pattern D over time.

Future: the content_hash column

As of the Phase 1 foundation patches landed on main on 2026-05-27, the following tables grew a content_hash TEXT column: spells, feats, items, classes, subclasses, features, unique_option_groups.

Today the column is NULL on every row. Population is Phase 1.5 work owned by the system-applications branch: every editor save will compute a SHA-256 of a canonical JSON serialization of the row's content fields (allowlist of fields that constitute "meaningful content"), and store it.

What it's for: enables the Foundry-side live-content bridge to detect "this Fireball on an actor was authored against a stale version of the source." Foundry can compare the hash on the actor's embedded copy against the catalog's current hash; mismatch → offer to refresh.

Operational note for editor-side work: when the hash hook lands, every editor's save path picks it up transparently — the hook lives inside upsertDocument / per-entity helpers. Editors don't need to do anything to participate. The only thing that matters from the editor's perspective is: the allowlist of "content fields". Fields that the hash hook considers "content" are the ones whose changes count as a meaningful edit; fields like updated_at or caller-specific state don't count. The system-applications branch will land the canonical list. Editors don't tune it.

New tables go in camelCase, in PERSISTENT_TABLES

Two conventions decided 2026-05-27 (per A1 + A3 in section 02) that apply to any new tables landing during the editor cleanup:

Convention Why
New tables use camelCase column names (e.g. imageUrl, not image_url). Aligns with Foundry's convention (camelCase end-to-end) and with React form state. Eliminates the editor ↔ D1 dance for the new tables from day one. Existing tables keep their snake_case until a separate migration pass.
New compendium tables are added to PERSISTENT_TABLES in src/lib/d1.ts. Compendium content is read-heavy + changes rarely — the same reason items / feats / spells / sources are already in the list. The 1-hour sessionStorage cache pays for itself within the first navigation.

Direct application: when Backgrounds and Races get promoted to their own tables in roadmap step 1, those tables follow both conventions out of the gate.

Findings — what stood out

F1 — The schema has three different polymorphic patterns, each for a different reason

parent_id+parent_type (features / scaling), base_*_id chain (items), and feat_type discriminator (races/backgrounds placeholder). Worth knowing them apart: the first two are designed-in and SQL-enforced where possible; the third is a placeholder that gets unwound when its target tables land. Conflating them would lead someone to expect the wrong cascade behavior.

F2 — The JSON-array tag pattern is the largest "ship faster, fix later" debt

Four entities (spells, feats, items, classes) reference tags via JSON arrays. The migration plan (Priority 6) is well-known but unscheduled. When step 3 of the roadmap (shared widgets) reaches the TagPicker, this would be a natural moment to revisit — the picker's behavior is unaffected, but the underlying storage shift would unlock SQL-side filtering and shed the LIKE-on-JSON workaround.

F3 — features.parent_id is unenforced by SQL on purpose

Polymorphic FKs can't be CHECK-constrained in SQLite, so the application layer is responsible. Practical impact: a class deletion today leaves features with stale parent_id values pointing at nothing. This is the orphan-features problem documented historically; the cascade-delete handler (A4) fixes it.

F4 — Items' polymorphic base_*_id chain is more elegant than it looks

Three FK columns (base_weapon_id / base_armor_id / base_tool_id), exactly one non-null, dispatched by item_type. The chain lets each individual FK be SQL-enforced (unlike parent_id+parent_type). The sibling base_item slug column preserves Foundry's value for re-resolution if the FK fails to wire at import time. Nice design.

F5 — The lore family is the model for what catalog tables would become if we kept normalizing

One parent table, four metadata variants, four junction tables. Pattern D enforced atomicity via batchQueryD1. If the catalog ever fully migrates off JSON-array tags AND adopts cascade-delete for classes, the catalog cluster ends up shaped more like the lore family. Not a destination on any roadmap; a useful "what would full normalization look like?" reference.
