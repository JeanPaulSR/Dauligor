The save and the read — at a glance

Every save round-trips through a five-stage pipeline. Every read does the reverse. The whole thing is held together by a three-tier cache, a global "something changed" timestamp, and one shared file (src/lib/compendium.ts) that handles the translation between the form's preferred shape and the database's preferred shape.

The save leg, step by step

What actually happens when you click Save in a compendium editor:

  1. Editor builds a payload. Every editor maintains its working state in a formData object (camelCase keys). On Save, it builds the payload — usually a shallow merge of formData with any computed fields (identifier auto-gen from name, Foundry-side bookkeeping, etc.).
  2. normalizeCompendiumData swaps the shape. Walks the payload and does eight separate things:
    • Renames camelCase fields to snake_case (imageUrl → image_url, sourceId → source_id, etc. — one mapping table, ~50 entries).
    • Unwraps the automation wrapper ({ activities, effects }) into flat activities + effects fields.
    • Decomposes the uses object ({ max, spent, period, recovery }) into the flat columns the schema has — except for items, which keep uses as a single JSON column. The discriminator is the presence of item_type.
    • Decomposes the components object for spells (V / S / M each into its own column).
    • Decomposes the prerequisites object for features.
    • Coerces numeric fields (level, quantity, armor_value, etc.).
    • Coerces booleans to integers (SQLite has no native boolean — true becomes 1). The coercion only fires on actual booleans, leaving string-enum columns like attunement: 'required' alone.
    • Strips a hardcoded list of UI-only / legacy fields (the "forbidden" list — id, automation, effectsStr, status, usage, the legacy Firestore source object, etc.) so the upsert doesn't try to write them.
  3. The per-entity helper does the upsert. Most editors call a helper from compendium.ts — upsertSpell / upsertFeat / upsertItem / upsertFeature. These layer on top of upsertDocument in d1.ts, which compiles to INSERT ... ON CONFLICT(id) DO UPDATE SET ... — explicitly not INSERT OR REPLACE (that would cascade-delete children; see the four rules).
  4. D1 stores the row. Snake-case columns; JSON columns are written as JSON strings; integers and floats as numbers; booleans as 0/1.
  5. queryD1 invalidates the cache. It looks at the SQL for an INTO / UPDATE / DELETE FROM / REPLACE INTO clause, extracts the table name, and runs clearCache(tableName) — wiping both the in-memory cache and any sessionStorage entries for that table.
  6. If the table is "persistent," it bumps the foundation timestamp. Any write to a persistent table (sources, spells, classes, feats, items, skills, weapons, etc. — the full list lives in PERSISTENT_TABLES in d1.ts) triggers bumpFoundationUpdate, which updates a single row in system_metadata keyed last_foundation_update. Every signed-in client polls that timestamp every 30 seconds. When it changes, everyone wipes their caches and refreshes.

The read leg, step by step

Going the other way:

  1. Editor calls fetchCollection or fetchDocument. Or sometimes a per-entity helper (fetchSpell, fetchFeat). All of them route through queryD1.
  2. queryD1 checks three caches before hitting the network. First the in-memory cache (5-minute TTL). Then sessionStorage (1-hour TTL, but only for tables in the PERSISTENT_TABLES allowlist). Then the in-flight request map (so two components asking for the same SQL at the same time share one network call). Only if all three miss does it fetch.
  3. The Pages Function returns the raw rows. Snake-case columns, JSON columns still serialised as JSON strings.
  4. queryD1 auto-parses an allowlisted set of JSON columns. A hardcoded list of ~50 column names gets JSON.parse'd on the way back, in a try/catch — if the value is already an object (e.g. because it came from another consumer that already parsed it), the catch swallows the error and leaves it as-is. The list includes both snake_case names and a few camelCase aliases (tag_ids AND tagIds) so consumers don't have to care which arrived.
  5. If the editor uses denormalizeCompendiumData, the shape gets translated. Reverse of normalize: snake_case columns become camelCase fields; flat uses_* columns get re-wrapped into a uses object; flat components_* columns become a components object; flat prerequisites_* columns become a prerequisites object; activities + effects get parsed (with already-parsed passthrough) and re-wrapped into an automation wrapper. Plus a small _id → id mapping on activities/effects so editor-side code can use one consistent key.
  6. The editor populates formData. User starts editing.

The camel/snake dance — and why it's there

Form state is camelCase by convention; D1 columns are snake_case by SQL convention. The mapping lives in two functions in src/lib/compendium.ts — normalizeCompendiumData for write, denormalizeCompendiumData for read — and the two are mirror images of each other.

Why this exists: it's a Firestore-era convention. The original Firestore documents used camelCase; D1 wanted snake_case for SQL convention reasons. Rather than rip out the camelCase form code, the migration added a translation layer at the boundary. The patterns doc flags this as a future cleanup — "pick one and stop dancing" — under Priority 4. Until that lands, every save and every read crosses the boundary twice.

Three things make this trickier than a flat key rename:

Concept What it does on write What it does on read
The automation wrapper Strips it; promotes automation.activities → activities and automation.effects → effects Rebuilds it from flat activities + effects for editor convenience
The uses object Decomposes into uses_max / uses_spent / uses_period / uses_recovery — except for items (where uses stays as a single JSON column on the row) Reconstructs the object from those flat columns when they're present
The components object (spells only) Decomposes into components_vocal / _somatic / _material / _material_text / _consumed / _cost Reconstructs from those flat columns

The discriminator on uses-decomposition is worth flagging: items keep uses as a whole JSON column (per a 2026-05-26 migration that promoted it to a structured sub-object), so the decompose step skips when item_type is present. Without that skip, item saves would silently strand the data.

There's also a "forbidden" list inside normalize — a hardcoded set of fields the payload might carry that should never reach the SQL: id (handled separately by upsert), automation (already unwrapped), effectsStr (UI-only intermediate state), the legacy Firestore source object (only source_id survived the migration), and a handful of others that callers might pass for legacy reasons. Adding a new always-rejected field is the way the forbidden list grows.

The cache layers

Three caches, sitting in front of /api/d1/query:

1. In-memory QUERY_CACHE

Keyed by SQL + params. Holds the parsed result and a timestamp. 5-minute TTL. Lives for the lifetime of the tab; cleared on cache-invalidation events. The fast path for any repeat read.

2. sessionStorage (persistent tables only)

A subset of tables (sources, spells, feats, items, tags, tag_groups, weapons, armor, languages, damage_types, the lore tables, etc.) get an extra layer in sessionStorage. 1-hour TTL. Survives page reloads. Tables in this list are read-heavy and change rarely; caching them across reloads avoids refetching the catalogue on every navigation.

There's a ~3.5MB cap per entry and a "drop other entries, then retry" eviction path when the quota gets hit — spells can grow large enough to trip it on its own as the catalogue expands. If even that fails, the write is silently dropped and only the in-memory cache holds it.

3. INFLIGHT_REQUESTS deduplication

Per-SQL map of in-flight promises. If two components ask for the same SQL at the same time, only one network call goes out and both await the same promise. Cleaned up after the promise resolves.

Invalidation is the other half. On any non-SELECT call, queryD1 extracts the table name from the SQL (matching INTO / UPDATE / DELETE FROM / REPLACE INTO) and calls clearCache(tableName) — which wipes both the in-memory entries and the sessionStorage entries that match. If the SQL is something weird the matcher can't classify, it falls back to wiping all caches (the "nuclear option" comment in the source).

batchQueryD1 applies the same logic per query in the batch — one invalidation per affected table, plus one foundation bump if any of them were persistent.

The foundation update bump

This is the cross-client invalidation primitive. One row in system_metadata keyed last_foundation_update holds a timestamp. Any write to a persistent table calls bumpFoundationUpdate, which updates that row to CURRENT_TIMESTAMP.

Every signed-in client polls the timestamp via GET /api/me/foundation-update every 30 seconds. When the value differs from the last seen one, the client calls clearCache() on its in-memory cache, drops the sessionStorage entries for our prefix, and refetches the user profile (in case the change touched it).

Why a poll? It's the simplest cross-tab + cross-client invalidation surface that doesn't require a websocket. The endpoint sends Cache-Control: no-store so the polling read always hits live data. The cost is one tiny HTTP round-trip every 30 seconds per tab.

system_metadata itself is excluded from the bump list (writes to it don't bump itself) so the bump doesn't recursively trigger another bump.

Multi-table writes — the atomic-write idiom

Most editor saves are single-table (write one row, done). But some entities have child rows that must stay consistent with the parent — lore articles are the canonical example: one parent row in lore_articles, four metadata variants (characters / locations / organizations / deities), four era-and-campaign junction tables (lore_article_eras, lore_article_campaigns, etc.), one tag-junction table, and so on.

For those, the editor uses batchQueryD1 — which sends every query as one batch to /api/d1/query. The handler runs them in a single D1 transaction. Either every query succeeds or none of them do; no half-written state.

This is the pattern that the future class cascade-delete (per summary A4) will use when it lands: one batch deletes the class, its subclasses, the features under both, and the advancements on each — all in one transaction, so a partial failure can't leave orphans.

Proposal mode — the save handler swap

Proposal mode is a remarkably small change to the editor data flow. The proposal-edit URLs wrap the same editor component inside a ProposalEditorWrapper. The wrapper provides a React context with two things: the entity-type identifier, and a save handler that queues writes into a local accumulator instead of calling D1.

The editor reads the context with useProposalContextOptional on render:

  • Outside the wrapper (admin direct route): no proposal context — the editor's normal Save handler runs (the upsertX path described above).
  • Inside the wrapper (proposal-edit route): the proposal context's applyProposalWrite is invoked instead. It queues the normalized payload into the accumulator. The user keeps editing; the queue grows.

When the user clicks Submit Changes, the accumulator flushes its queue as a proposal bundle to the proposals API. Admin reviews accept / reject / cascade.

Almost every editor file imports a handful of proposal-aware hooks (useProposalAccumulator, useProposalEntityDrafts, useDraftedEntityIds, useEditBaseUnlocks, useCascadeDependent, ...) and a few proposal-aware components (CascadeDependentBanner, TagReplacementPicker). Most of them are conditional pass-throughs — they noop when no proposal context is present. The pattern is clean: the editor file looks busy at the top of the imports, but the save handler logic in the body is short.

Per summary A5, the proposal-mode editor surface is not considered done. Some editors are more wired than others. None of this affects the underlying data flow above — only the destination of the save changes.

Post-save UI — the "preserve selection" idiom

In any Pattern E editor (the master-detail manager), there's a real risk: a user clicks Save, the network goes, the user impatiently clicks a different row in the list, and when the save resolves the editor would have repointed to the just-saved row — wiping whatever the user typed into the new selection.

The SpellsEditor pattern (now codified in CompendiumEditorShell consumers) handles this by capturing editingIdAtStart and savedId before the await, and using a ref (editingIdRef) to track the user's current selection. After the save resolves:

  • Refresh the list so the left column reflects the new name / state.
  • Refresh the saved row's cache so re-selecting it shows current data.
  • Only adopt savedId as the new editingId if the user is still on the row they were saving (i.e. editingIdRef.current === editingIdAtStart). If they've moved, leave their current selection alone.

This is the kind of subtle thing that's invisible when it works and obviously broken when it doesn't. It's worth knowing about when touching any Pattern E editor.

The four non-negotiable rules

From AGENTS.md, expressed at the data-flow level:

Rule 2 — Never INSERT OR REPLACE

D1 has PRAGMA foreign_keys = ON by default. INSERT OR REPLACE resolves a PK conflict by deleting the existing row and reinserting — which fires ON DELETE CASCADE on every FK child. There's a documented data-loss incident from before this rule was added. Always use INSERT ... ON CONFLICT(<pk>) DO UPDATE SET ... for single-PK upserts, or INSERT OR IGNORE for junction tables.

Rule 4 — JSON column passthrough on remap

queryD1 auto-parses ~50 JSON columns on the way out. If your editor does its own remap on a JSON column (read-time), the check must be: 

typeof X === 'string' ? JSON.parse(X) : (X ?? <default>)

Forgetting the passthrough means: when the value arrives already parsed (the common case), the typeof X === 'string' branch is skipped, the else branch returns the default, and the field silently empties. The auto-parse list in queryD1 is the source of truth for which columns get this treatment.

Rule 5 — Drift-managed sibling files

A handful of libraries live in both src/lib/X.ts (client) and api/_lib/_X.ts (server). classExport, classProgression, referenceSyntax, requirements. Touch one, touch both. The drift contract is documented in headers on the server-side files.

Rule 7 — Local-first D1 migrations, remote only with explicit permission

Migrations land on local D1 first (npx wrangler d1 execute dauligor-db --local --file=...), and only get applied to remote D1 with explicit per-migration permission. Treat --remote as a one-way door. Affects this page only insofar as: when an editor depends on a new column, that column may exist locally but not on remote until the migration is approved.

Findings — what stood out

F1 — The cache invalidation is granular but powerful

A single mutation wipes that table's cached SELECTs from both in-memory and sessionStorage, AND if the table is "persistent" bumps a global timestamp that every other client picks up within 30 seconds. Three layers (in-memory, sessionStorage, foundation poll), one consistent invalidation surface. I expected this to be hand-managed per editor; it's centralised.

F2 — The forbidden list is the migration's exhaust pipe

Almost every entry in the normalize forbidden list (automation, effectsStr, the legacy Firestore source object, uniqueOptionGroupIds, etc.) is something a caller might send for legacy reasons that the schema can't accept. The list isn't pretty; it's an honest record of the migration's rough edges. Adding to it is the right call when a "no such column" error surfaces in the wild.

F3 — The items-payload skip on uses decomposition is a real footgun

A migration shifted items' uses from flat columns to a JSON object. The normalize function has to detect items (via item_type presence) and skip the decomposition, else item saves silently lose the field. This is exactly the kind of cross-cutting concern the "single naming convention" cleanup (Priority 4 in the patterns doc) would simplify — if everything is JSON-on-D1, the decompose step disappears.

F4 — The automation wrapper exists only for editor convenience

On disk, activities and effects are separate JSON columns. The wrapper object ({ automation: { activities, effects } }) is rebuilt on read for editor consumption and stripped on write. It's not load-bearing — an editor could be refactored to read flat activities + effects directly — but the wrapper makes the activity / effect editor mounting points cleaner. I'd file this under "easy to keep until someone has a reason to remove it."

F6 — Foundry uses camelCase — the cleanup direction is now decidable

Confirmed from the spell-import contract and the spell-export library: Foundry's payloads are camelCase end-to-end. That means a future camel/snake cleanup can target either side (camelCase-everywhere or snake_case-everywhere) and align at least one boundary in the process. Captured for posterity; see A1 for the trade-off.

F5 — Proposal mode wires through the existing flow, not around it

The save handler swap is small: the editor calls a context-provided save function instead of the per-entity upsert helper. normalizeCompendiumData still runs, the same payload still gets built, only the destination changes. That keeps the proposal-mode bundle compatible with admin-side direct writes — admin accepts a proposal by routing its bundle through the same per-entity upsert helpers the direct flow uses. No two save paths to keep in sync; one path, one extra branch.

Your clarifications

Your answers to the four questions, captured for posterity. A1's Foundry-convention investigation is folded in.

A1 — Camel/snake dance: cleanup pending, decision now informed by Foundry convention

You said: the dance should be cleaned up. Whether it's worth it depends on what Foundry uses — if D1 and Foundry can use the same convention, the future is a mono decision.

Foundry's convention, confirmed: camelCase end-to-end. From the spell import contract: flags.dauligor-pairing.sourceId, system.activations, system.activation.type, system.materials, dbId — all camelCase. Foundry's dnd5e system schema is camelCase throughout (system.<area>.<subarea>), and the existing export libraries already emit that shape.

What that means for the cleanup direction: we have two plausible end states.

  • camelCase end-to-end on the app side. Form state stays camelCase; D1 columns become camelCase too. The editor ↔ D1 dance disappears. The Foundry export becomes mostly a pass-through (still has to wrap things in the system envelope, but no key renaming). Cost: a big D1 column-rename migration, plus every SQL string in the codebase has to learn the new column names.
  • snake_case end-to-end on the app side. Form state moves to snake_case; D1 columns stay snake_case. The editor ↔ D1 dance disappears. The Foundry export does the snake→camel translation it already does today (one boundary, not two). Cost: every editor's form-state shape changes; UI code that reads formData.imageUrl becomes formData.image_url.

Both options are sizeable but bounded. The first option is nicer for new developers (camelCase is the React idiom) and aligns the entire pipeline. The second option is closer to SQL convention and leaves more files touched. You decide which side to pick.

Patterns doc Priority 4 estimate is 1–2 days; that probably assumes the snake-everywhere option since the migration footprint is smaller. The camelCase-everywhere option is bigger but cleaner; closer to a week.

A2 — Foundation poll behavior matches your original design

You said: the original design was to query D1 every 30 seconds to check if anything changed. You suspected the implementation may have drifted to "manual wipe every 30 seconds."

Verified against the code: the implementation matches your original design. Every 30 seconds, the client fetches GET /api/me/foundation-update, which returns the last_foundation_update timestamp. The client compares it to the last seen value. If — and only if — the timestamp differs, the cache wipe fires (clearCache() on in-memory + sessionStorage, plus a profile refresh). When nothing has changed, the comparison short-circuits and nothing happens.

Net effect: 30-second polling cost (one tiny HTTP round-trip per tab) + a cache wipe only when a write actually landed somewhere. Cheap when idle, correct when busy.

Code references: App.tsx:128–147 drives the polling loop and the conditional wipe; d1.ts:158–171 is the fetch helper; d1.ts:143–145 is the bumpFoundationUpdate writer.

A3 — What "persistent table" actually means

You said: not sure what the distinction is. Fair — the term reads more important than it is. Plain explanation:

"Persistent" here just means "this table's reads are cached in sessionStorage in addition to the in-memory cache, so they survive a page reload."

Concretely:

Table type In-memory cache sessionStorage cache Survives reload?
Not in PERSISTENT_TABLES Yes, 5-min TTL No No — refetched on every reload
In PERSISTENT_TABLES Yes, 5-min TTL Yes, 1-hour TTL Yes — first visit on a fresh load reads from sessionStorage instead of the network

The criterion the list seems to use: read-heavy + changes rarely. Sources, weapons, armor, skills, damage types, languages — the reference vocabulary that every page touches but admins almost never change — are in the list. Items, feats, spells, tag groups, campaigns, the lore tables are also in it. Tables NOT in the list: classes, subclasses, features, characters, system_metadata, and anything proposal-related.

Recommendation for new tables: when Backgrounds and Races get promoted to their own tables (per roadmap step 1), they belong in PERSISTENT_TABLES for the same reason items / feats / spells are — read every time a character is built, changed only by admin authoring.

A4 — "Preserve selection" idiom is a Pattern E requirement

You said: it should be in every Pattern E editor.

Treating that as a design requirement going forward: when adding a new Pattern E editor, the post-save logic must capture editingIdAtStart + editingIdRef before the await and only adopt savedId as the new selection if the user is still on the row they were saving. The CompendiumEditorShell already bakes this in — consumers get it for free as long as they don't override the save path with a custom handler that skips the ref check.

Bespoke (Pattern B) editors should adopt the same pattern if their layout permits a "click-another-row-mid-save" interaction. ClassEditor in particular opens features through a modal that has the same shape; worth checking when its cleanup pass comes around (roadmap step 2).
