Compendium Editors — Process Reference

How the compendium editors actually work today · branch compendium-editors · 2026-05-27
data flow table relations shared components app-side endpoints foundry-side OUT of scope
Purpose. This reference set documents the current behavior of the compendium editors — not the plan, not the decision tree, not the migration history. The user uses these pages to verify Claude's understanding and correct course where the documented model drifts from the code. Edits welcome; treat any contradiction with the source as a bug in this doc, not in the code.

How to read this set

The reference is split into six sections. Read them in order if you want a top-down briefing; jump straight to a section if you're trying to spot-check a specific concern.

Contents

  1. Survey — the human-focused summary of my mental model of the system. Start here for review. Has a sibling reference page with the file-by-file detail.
  2. Data flow — form state → normalize → upsert → D1 → cache. The round-trip of a save and what each layer does to the payload.
  3. Tables & relations — how FKs, junction tables, JSON columns, and the parent_id + parent_type pattern fit together.
  4. Shared components — AdvancementManager, ActivityEditor, ActiveEffectEditor, RequirementsEditor, TagPicker, FilterBar, EntityListSection, the two shells. What each does, who uses it, what data it expects.
  5. Endpoints — functions/api/** (Pages Functions) and the api/_lib/_*Export.ts server-side composition libraries that back them. Foundry-module-side consumption is out of scope.
  6. Per-editor walkthroughs — deep dives on individual editors (ClassEditor, SpellsEditor, FeatsEditor, etc.). Populated iteratively after the user signs off on the survey.

Scope & non-goals

In scope

  • Every editor page under src/pages/compendium/** plus the admin-side editors under src/pages/admin/** that author compendium-shaped data (proficiencies, statuses, weapons, armor, etc.).
  • The shared editor components under src/components/compendium/** — what they render, who they're mounted by, and what shape of data they expect.
  • The D1 tables those editors write to, the FK and junction relations between them, and the JSON columns auto-parsed by src/lib/d1.ts.
  • The Pages Functions and the api/_lib/_*Export.ts server-side libraries that compose payloads exposed at /api/module/* for the Foundry pairing module.

Out of scope

  • Foundry-module-side consumption. What happens inside module/dauligor-pairing/ when the module fetches a payload is owned by another agent; this reference stops at the wire (what the server emits) and does not document the importer / viewer / actor-bundle ingestion logic on the Foundry side.
  • Character builder + character sheet. Those consume compendium data but aren't editors; documented separately under docs/features/character-*.md.
  • Proposal-mode workflow internals. The proposal-aware behavior is layered on top of the editors via ProposalEditorWrapper + accumulator hooks. We describe how editors react to proposal mode but don't recap the proposal-cycle internals — see docs/features/content-proposals.md.
  • Wiki / lore editors. They follow Pattern D (relational helpers in src/lib/lore.ts) which is owned by the system-applications branch. We reference it as one of the five patterns but don't deep-dive.

Conventions used in this reference

  • File paths are relative to repo root: src/pages/compendium/ClassEditor.tsx.
  • Line citations appear when a claim depends on a specific block, formatted src/lib/compendium.ts:153.
  • Drift callouts (red left border) mark places where an existing doc (the patterns doc, a feature doc) says something different from what the code currently does. Each drift item names the doc, what it says, and what the code does.
  • Stub cards (dashed border) mark sections that are intentionally not yet written — placeholder for upcoming content, not gaps in the survey itself.
  • Badges: topic tag, drafted, stub, verified, to verify.

How to correct me

If a claim on a page is wrong, the fastest correction loop is:

  1. Tell me which page and which section heading (e.g. "01-survey, section 'Pattern E editors'").
  2. Tell me what's wrong — either the correct claim, or a pointer to the source file that contradicts mine.
  3. I'll re-read the source, fix the page, and update any related docs in docs/architecture/ or docs/features/ that propagate the same mistake.

I expect to be wrong about specific things — especially component prop contracts and edge-case save behavior — and reading the source against the doc is the whole point of this artifact.
