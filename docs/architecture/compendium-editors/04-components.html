04 — Shared components

A collaborative walkthrough: document each shared widget, flag UI-side issues, fix what we can as we go.
walkthrough component-by-component
This page works differently from 01–03. Those were read-then-review. This one is a walkthrough: we take one shared component at a time, I document what it does and how it's wired, you (and I) toss UI-side issues onto its list, and we try to handle the ones we can right then. The global UI-issues log tracks everything found across all components so nothing gets lost between sessions.

How the walkthrough works

  1. Pick a component (from the coverage order, or you name one out of order).
  2. I document it against its source — purpose, props, internal state, data shape, who mounts it, gotchas. Each component gets its own sub-page under components/.
  3. We flag UI issues — you call out what bugs you about it; I add what I notice. Each lands as a row in the component's local issue list and in the global log.
  4. We handle what we can — quick UI fixes get made during the walkthrough (separate from the docs branch if they're real code changes; this branch is documentation-first, so code fixes either go on a sibling branch or get flagged for one). Bigger ones get logged with enough detail to action later.
  5. Move to the next component.
Note on code changes from this branch. compendium-editors is a documentation branch (its manifest claims only docs/**). If a UI fix we find is a real source-code change, it should land on a separate branch — otherwise this branch's scope creeps into src/** and collides with the editor-cleanup work itself. The walkthrough's job is to find and specify the issues precisely; small/safe fixes can be split off, larger ones become roadmap step-1/step-3 work items. (If you'd rather this branch also carry the fixes, say so and I'll widen the manifest.)

The per-component template

Each component sub-page follows this shape:

SectionContents
SourceFile + the key types/exports
PurposeWhat it does and the user-facing affordance, in plain language
PropsAnnotated table — type, default, behavior; mandatory props in bold
Internal state vs consumer stateWhat the component owns vs what the mounting editor manages via callbacks
Data shapeThe value it reads / emits (the Advancement array, the Requirement tree, the activities/effects JSON, etc.)
ConsumersEvery editor that mounts it + the context props each passes
UI issuesThe running list for this component — status, description, proposed fix
GotchasContext-dependent behavior; things that look like bugs but are intentional

Coverage order

Step-1-relevant components first (the ones the Backgrounds / Races / Items editors mount), since that's where the roadmap goes next. You can override the order any time by naming a component.

#Component(s)Why this slot
1 CompendiumEditorShell The Pattern E backbone. Backgrounds / Races / Items all live inside it. Understand the shell before its tenants.
2 TagPicker (+ inline SpellTagPicker) Mounted by feats + items; the tags → junction migration (A1) may pair here.
3 AdvancementManager Heaviest shared widget; class/subclass/feat/item/race/background all mount it. Changes ripple widest.
4 ActivityEditor Per-type automation forms; mounted nearly everywhere.
5 ActiveEffectEditor Companion to ActivityEditor; same broad consumer set.
6 RequirementsEditor Feats + option items; structured prereq tree.
7 FilterBar + SectionFilterPanel + SpellFilterShell The filter trio shared by browsers + managers.
8 ScalingColumnsPanel Class/subclass/feat/item scaling tables; polymorphic owner (A3 confirmed uniform).
9 Detail panels: FeatDetailPanel, ItemDetailPanel, SpellDetailPanel The right-pane previews reused from the public browsers.
10 Mechanic field sets: ItemUsesField, WeaponMechanicsFields, ArmorMechanicsFields, ToolMechanicsFields, WeaponPropertiesPicker, WeaponTypeSelect, ProficienciesEditor Items + classes mechanic sub-forms.
11 List/hero accessories: EntityListSection, EntityEditModal, FeaturesView, FeatureModalHero, ModularChoiceView, RuleMembershipPanel Supporting widgets for the bespoke + class editors.
12 Workbenches: FeatImportWorkbench, ItemImportWorkbench, SpellImportWorkbench Foundry-import mode tabs (consume the module export contract; light-touch here).
13 Auxiliaries: ActiveEffectKeyInput, BakeNowButton, ClassImageEditor, SpellArtPreview, the two shells' read-only cousins Small single-purpose helpers.
