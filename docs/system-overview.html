What The Archive Is

The Archive is a campaign-management web app for Dungeon Masters running D&D 5e. It is two things at once: a technical dashboard for the dense reference work that a long campaign accrues — lore, NPCs, custom rules, character sheets, spell scaling tables — and an immersive workspace that dresses all of that utility in a parchment-and-gold "Sword & Sorcery" aesthetic so the DM stays in the world while they prepare for it.

The Archive does not try to replace the table. It tries to be the room behind the screen — the binder, the index card box, the wall of notes — while a virtual tabletop like FoundryVTT handles play. To that end, a companion Foundry module pulls Dauligor's canonical content into the table in real time.

The system is live in production at www.dauligor.com.

The Five Modules

Each module is a separate slice of the DM's workflow. They share auth, theme, and the underlying data plane, but feature work in one rarely needs to touch another.

World Engine Live

The lore wiki. Where the campaign world is written down.

Hierarchical articles — regions contain cities contain factions contain people. Each article supports inline DM-only secrets ([secret]…[/secret] blocks) that only staff can see, plus per-article notes that never reach players. Articles can be drafts or published, scoped to specific campaigns or eras, and cross-linked via reference syntax so a person's article auto-mentions every place they appear.

Surface: /wiki. Backing data: lore_articles + half a dozen junction tables for secrets, metadata, and cross-links.

Rules Foundry Live

The mechanical content library — classes, spells, feats, items, the works.

A full mechanical compendium with first-class editors for every content type a DM might want to author or modify: classes (with subclasses, features, and advancements), spells, feats, items, facilities, plus the supporting machinery — scaling columns (cantrip scaling, spell-point progression, spells-known curves), unique option groups, tags and tag groups for filtering, and a requirements tree for prerequisites.

The compendium is also where the activity editor and active effect editor live — shared components used by every entity that can "do something" or "apply a status."

Surface: /compendium/*. Backing data: roughly twenty SQL tables, each documented under docs/database/structure/.

Character Workspace Live

The character builder + sheet, with Foundry-compatible export.

A full character builder that walks the player through class selection, level-up advancements, ability scores, equipment, and spell selection — using the same progression engine that drives the compendium. The companion character sheet handles play: HP bars, mana, hit dice, ability plaques, skill proficiency dots, saving-throw trackers, portrait integration.

A character can be exported to FoundryVTT as a dauligor.actor-bundle.v1 payload — advancements baked into Foundry's native format, ItemBumpUses bumps folded into system.uses.max, audit-trail flags so the module side knows where each piece came from.

Surface: /characters. Backing data: characters + ~8 character_* child tables linked by FK cascade.

Administrative Hub Live

User management, campaign control, the image library, system catalogs.

The DM's control room. RBAC user CRUD with password reset and bootstrapping; campaign + era management (with members and roles); a centrally managed image library (R2-backed, with WebP auto-conversion and reference scanning); system catalogs for skills, tools, weapons, armor, languages, damage types, and conditions; and the content proposal queue where Librarians and trusted users can submit changes for staff review without direct write access.

Surface: /admin/*. Backing data: users, campaigns, campaign_members, eras, image_metadata, user_permissions, pending_revisions.

Foundry Pairing Module Live In Flight

The companion FoundryVTT module that brings Dauligor into the table.

A companion module for FoundryVTT v13 + dnd5e 5.x. It currently imports Dauligor content into Foundry as native compendium documents — classes, spells, feats, items, characters — consuming the app's /api/module/* endpoints. It also adds a Dauligor-flavoured alternate character sheet, a Prepare Spells manager, and per-class spell catalogs that mirror the compendium.

In flight: the module is being upgraded from a static importer into a live viewer. After this work lands, Foundry windows will fetch canonical content from /api/module/* on demand and cache per-world — eliminating the "stale Fireball" problem that comes with baking content into a compendium pack. See the pending-tasks roadmap for status.

Source: module/dauligor-pairing/.

Runtime Topology

Four layers, one direction. Every request originates in the browser, traverses an authenticated proxy, and ends at the data plane via a stateless Worker. 

Browser  (React SPA, Firebase Auth SDK for JWT issuance)
   │
   │  Authorization: Bearer <Firebase ID token>
   ▼
Pages Functions (production)  |  Express server.ts (local dev)
   │  • Verifies JWT via jose+JWKS  (no Firebase Admin SDK)
   │  • Loads role from D1
   │  • Gates request by table + action
   │
   │  Authorization: Bearer <API_SECRET>
   ▼
Cloudflare Worker  (worker/index.js — stateless, two bindings)
   │
   │  DB binding → D1     BUCKET binding → R2
   ▼
Cloudflare D1 (SQL)  +  Cloudflare R2 (binary)

Local development uses the same code paths via an Express server (server.ts) that mounts Vite for the SPA and provides Pages-equivalent route handlers. The only divergence is /api/module/*, which is Pages-only — for local module testing run npx wrangler pages dev.

The Worker is the trust boundary. It accepts only requests bearing the shared API_SECRET and never sees a user JWT. User-level authentication and RBAC enforcement live one layer up, in Pages Functions / Express. The Worker must never be exposed to the public internet without the API_SECRET wall.

One Worker, two bindings

Both D1 and R2 traffic go through the same Worker by deliberate choice: a single auth surface, lower cold-start cost, and a single wrangler dev process for local emulation of both bindings.

Data & Storage

The Archive runs on Cloudflare D1 for relational data and R2 for binary objects. Firebase Authentication sits alongside as the JWT issuer for user accounts — tokens only, no data storage. Every read and write to application data goes through D1.

LayerTechnologyWhat lives there
Relational data Cloudflare D1 (dauligor-db) Every entity: users, campaigns, lore, compendium, characters, proposals, system catalogs.
Binary objects Cloudflare R2 (dauligor-storage, public at images.dauligor.com) Article images, portraits, tokens, map tiles — auto-converted to WebP; metadata mirrored in image_metadata.
Auth tokens Firebase Authentication JWT issuance + admin user-management via Identity Toolkit REST.

Schema changes go through timestamped migration files in worker/migrations/ (e.g. 20260527-1420_content_hash_columns.sql). Migrations always apply to local D1 first; remote D1 requires explicit per-migration permission — this is a one-way door.

API Surface

The application exposes its data through two patterns:

1. Per-route endpoints

The preferred shape for anything sensitive. Each route is a Pages Function with its own SQL, its own gate, and its own contract.

RoutePurpose
/api/meResolve the signed-in user's profile, auto-create on first sign-in, auto-promote bootstrap admins.
/api/characters/[id]Read / write / delete a character with all child tables in one batch.
/api/lore/articles/[id]Read / write articles. dm_notes stripped for non-staff readers.
/api/campaigns/...Campaigns, members, era assignment.
/api/admin/*Admin-only operations: user CRUD, character admin view, era management.
/api/module/[[path]]The Foundry module's data surface — sources catalog, class bundles, spell payloads, tag catalog. (Pages-only.)
/api/r2/[action]Single dispatcher for all seven R2 operations (upload, list, delete, rename, move-folder, scan-references, rewrite-references).

2. Generic D1 query proxy

/api/d1/query is a catch-all that still handles most compendium reads and writes. Its gate is table-aware: writes to sensitive tables (users, characters, lore_*, campaigns, etc.) are refused with a pointer at the proper per-route endpoint. New endpoints should prefer pattern 1; the proxy is defense-in-depth, not the primary boundary.

Permissions & Roles

Every user has exactly one of five base roles. Some capabilities don't fit neatly on that ladder — those are additive permissions with optional scope.

RoleUI LabelWhat they can do
adminGMEverything. CRUD on all tables, user + campaign management, admin endpoints.
co-dmCo-DMSame as admin for lore + assigned campaigns. Not admin endpoints.
lore-writerLibrarianCRUD on lore (drafts + published). Not characters, not campaigns.
trusted-playerPlayer+Same as user, with extended visibility on certain DM-tagged content.
userAdventurerRead-only on published lore + revealed secrets. Own characters.

Additive permissions live in the user_permissions table. The first one is content-creator: holders can submit proposals for tags, spell rules, and spell lists into a review queue (pending_revisions), without write access to the underlying tables. Scope is JSON-narrowed by world, campaign, or era.

Enforcement happens at three layers

  1. Server-side route guards — the only layer that actually keeps anyone out. Helpers like requireStaffAccess, requireImageManagerAccess, requireAdminAccess, requireCharacterAccess verify the JWT, load the role from D1, and refuse the request before any SQL runs. requireCharacterAccess returns 404 rather than 403 when access is denied, so probes can't enumerate character IDs.
  2. Client-side effectiveProfile — a UX layer. Pages use it to hide Edit/Delete buttons, filter results to what a viewer is allowed to see, and honor preview mode (admins viewing the app "as a player"). Always prop-drilled, never contextual.
  3. D1 CHECK constraints — an integrity layer. Catches stray writes of bogus values (unknown roles, unknown permission keys). Not a security boundary.
Preview mode is UX only. The user's JWT still carries their real role, so server endpoints don't change behaviour. It exists so admins can sanity-check what a player sees, not as a security primitive.

Bootstrap admins

Three hardcoded staff principals are forced to role: admin on every /api/me load: luapnaej101@gmail.com, admin@archive.internal, gm@archive.internal. The intent is to survive a wiped users table — signing in as one of these bootstraps the system back into administrability.

Rich Content & References

The Archive uses a three-layer rich-text pipeline:

  • BBCode is the storage format. Articles, descriptions, and notes are persisted as BBCode strings.
  • TipTap is the visual editor. The editor converts BBCode ↔ rich text for the author.
  • BBCodeRenderer is the display component. Reads BBCode and renders the visual representation.

Inside BBCode, reference syntax lets authors link mechanical content symbolically rather than by ID. References are resolved at render time and at Foundry export time:

ReferenceResolves to
@profThe character's proficiency bonus.
@levelThe character's class level.
@scale.<class>.<id>A scaling table value for the current class at the current level.
@ability.<key>An ability modifier (str, dex, con, int, wis, cha).
@article[<slug>]A live link to a wiki article. Phase 2 will resolve this in Foundry too.
@condition[<key>] · @rule[<key>]Planned — system-page glossary entries. Phase 2 in flight.

Foundry Pairing

The Foundry module is a companion, not a standalone tool. It runs on FoundryVTT v13 + dnd5e 5.x and is installed via an NTFS junction pointing at the repo's module/dauligor-pairing/ directory — so module changes ship by editing the repo source, no copy step required.

Today: static importer

The module currently consumes baked endpoint payloads (per-class bundles, per-spell documents, the sources catalog) and creates real Foundry compendium documents from them. This works but means a content edit on the app side doesn't propagate to existing Foundry copies — the "stale Fireball" problem.

Tomorrow: live viewer

The live-content bridge — the work this branch is doing — flips the architecture. After Phase 2 lands, Foundry text containing @article[…] or @condition[…] will be rewritten by a custom enricher into clickable links, and clicking one opens a Foundry window that fetches the current HTML from /api/module/<source>/articles/<slug>. Per-world caching keeps the experience snappy and lets the table play offline. Drag-and- drop for mechanical content still creates real Foundry documents, but the payload is fetched live and tagged with a sourceHash so the module can detect when its copy went stale.

Architectural rule for enrichers: register via CONFIG.TextEditor.enrichers.push(…). Never override TextEditor.enrichHTML or any pipeline stage — that breaks dnd5e and every other module installed in the same world.

Full module contracts live in module/dauligor-pairing/docs/; the master index is import-contract-index.md.

How Work Gets Coordinated

The repo is worked by multiple agents and the DM in parallel. To avoid two branches silently overwriting each other on shared files (the registry of mapping tables in compendium.ts, the JSON-parse list in d1.ts, the route table in App.tsx), the project uses a coordination folder:

  • handoffs/BRANCH_REGISTRY.md — live registry of every in-progress branch and the files it claims. Read before editing any shared file.
  • handoffs/<branch>/manifest.md — per-branch declaration of primary (exclusive) and shared (append-only) files, plus open requests to other branches.
  • handoffs/<branch>/YYYY-MM-DD-<topic>.md — accumulating handoff documents that survive context resets and agent hand-overs.

Migration files are timestamped (YYYYMMDD-HHMM_*.sql) post-0018 so two parallel agents can't collide on the same sequential number.
