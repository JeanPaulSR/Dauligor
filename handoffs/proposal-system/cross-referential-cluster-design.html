Proposal System · Design Explainer

Letting content-creators build whole classes through proposals

A class isn't one thing — it's a cluster of pieces that point at each other (scaling columns, subclasses, unique option groups). The proposal system was built for independent entities, so authoring a full class as a content-creator breaks in two specific ways. This explains why, and the plan to fix it.

What the user hit

Bug 1

Draft pieces are invisible to each other

You create a scaling column / option group / class as a draft in your block — then go to use it somewhere else, and it's not in the dropdown. A created column can't be picked in the class. An option group can't be found. A class can't be selected inside the option group.

Bug 2

Some pieces can't be saved at all

Saving scaling columns (and other nested pieces) inside the class / subclass editor fails with a permission error — even though saving the class itself works fine.

Why it happens

The proposal system has one mental model: one independent entity, one whole-row change, queued for review. That's perfect for a spell or a tag. But a class is a web of cross-references, and two assumptions break:

reference by id (the part that breaks) ownership / parent link

Broken assumption #1 — "every piece is savable on its own"

The proposal "writer" only catches saves for the one entity type the editor is bound to (the class). Nested pieces — scaling columns — are written with a different, direct path that skips the proposal queue and hits the database gate, which refuses non-staff users. So a content-creator gets a permission error. Scaling columns aren't even on the list of things the proposal system knows how to handle.

Broken assumption #2 — "references point at things that already exist"

Drafts live in a staging table, not the real tables. Every dropdown ("pick a column", "pick a class") reads only the real tables. So anything you just created in your block — which is still a draft — simply isn't there to pick.

The insight that makes this easy. Every id in this cluster is generated on the client as a permanent UUID, and the proposal system keeps ids unchanged through approval. So a reference you author against a draft id (e.g. "this advancement uses that draft column") is already correct — it lights up the moment both sides go live. We don't need to rewrite ids or resolve placeholders. We only need three things: let nested pieces be saved, let pickers see the block's drafts, and make the cluster approve as one unit.

The fix, in four parts

Part Aproposal-system (me)

Teach the system about scaling columns

Add scaling_column as a thing the proposal system can carry: a config entry, a type entry, and a small DB migration so the staging table accepts it. This alone removes the permission error's root cause.

Part Bcompendium-editors

Route column saves through the queue

Change the column editor panel so that, in proposal mode, saving a column goes into the block instead of straight to the database.

Part Ccompendium-editors

Show block drafts in the pickers

Make the dropdowns merge in the current block's drafts (clearly marked "in this block") so a just-created column / group / class is selectable. Uses a small helper I provide.

Part Dproposal-system (me)

Approve the whole block at once

A new admin action applies every piece of the block in dependency order (the class before the things that point at it, etc.) so references never go live pointing at a rejected draft.

How "approve whole block" works (Part D)

1
Admin clicks "Approve block." The server gathers every pending revision in that block.
2
It orders them by dependency. Things that are referenced (columns, the class) get applied before the things that reference them (advancements, option items).
3
It applies them in sequence. Because the ids never changed, every cross-reference resolves as it lands.
4
If something fails mid-way, it reports exactly what landed and what didn't — the block stays open so the admin can fix and retry. (The database can't wrap all of this in a single transaction, so we're honest about partial application rather than pretending it's atomic.)
